# OPS — Chiffrement applicatif des données cliniques (story 11.4)

> Livrable **ops** (non exécuté par la boucle BMAD — même garde-fou que
> `prisma migrate deploy` et le provisioning du bucket Storage 9.2).
> Réf. : `docs/architecture/6-chiffrement-applicatif-decision.md` (ADR §6),
> `docs/stories/11.4.chiffrement-applicatif.story.md`.

## Ce que livre 11.4

Chiffrement applicatif **au repos** (AES-256-GCM, enveloppe versionnée
`enc:v1:<iv>:<tag>:<ciphertext>`) des contenus cliniques libres :

- `consultation_notes.content` (notes de consultation, 9.1) ;
- `medical_history_entries.content` (antécédents médicaux, 9.3).

Chiffrement/déchiffrement à la frontière des Server Actions
(`lib/security/crypto.ts`). **Aucun changement de schéma** : le texte chiffré tient
dans la colonne `content` existante.

## 1. Générer et poser la clé (OBLIGATOIRE EN PRODUCTION)

```bash
openssl rand -base64 32
```

Poser la valeur produite dans la variable d'environnement **`DATA_ENCRYPTION_KEY`**
de l'environnement de production (Vercel / hébergeur), puis redéployer.

- **Sans clé** (`DATA_ENCRYPTION_KEY` absente/vide/longueur ≠ 32 octets) → le
  chiffrement est **désactivé** (passe-plat). Acceptable en dev/preview/CI,
  **interdit en production**.
- ⚠️ **Conserver la clé hors base** (gestionnaire de secrets). **Sa perte rend les
  données chiffrées illisibles** (aucune récupération possible). Sauvegarder de
  façon sûre avant toute mise en production avec des données réelles.

## 2. Comportement en lecture (rétro-compatibilité)

- Les lignes **déjà en clair** (créées avant 11.4) restent **lisibles** : la lecture
  renvoie la valeur telle quelle si elle n'est pas au format `enc:v1:`.
- Les lignes en clair sont **ré-écrites chiffrées** à leur prochaine mise à jour,
  ou via le backfill ci-dessous.
- Une ligne `enc:v1:…` **sans clé** ou avec une **mauvaise clé** → erreur de lecture
  (jamais de contenu corrompu renvoyé silencieusement).

## 3. Backfill optionnel des lignes en clair (étape ops manuelle)

Pour chiffrer immédiatement l'historique existant (au lieu d'attendre une mise à
jour), exécuter un script ponctuel **avec `DATA_ENCRYPTION_KEY` posée** :

```ts
// scripts/backfill-encrypt-clinical.ts  (À EXÉCUTER MANUELLEMENT, hors boucle)
import { prisma } from "@/lib/prisma";
import { encryptField, isEncrypted, isEncryptionEnabled } from "@/lib/security/crypto";

async function main() {
  if (!isEncryptionEnabled()) {
    throw new Error("DATA_ENCRYPTION_KEY absente : backfill annulé.");
  }
  for (const note of await prisma.consultationNote.findMany()) {
    if (!isEncrypted(note.content)) {
      await prisma.consultationNote.update({
        where: { id: note.id },
        data: { content: encryptField(note.content) },
      });
    }
  }
  for (const entry of await prisma.medicalHistoryEntry.findMany()) {
    if (!isEncrypted(entry.content)) {
      await prisma.medicalHistoryEntry.update({
        where: { id: entry.id },
        data: { content: encryptField(entry.content) },
      });
    }
  }
}

main().finally(() => process.exit(0));
```

> Le backfill est **idempotent** (`isEncrypted` saute les lignes déjà chiffrées).
> Sauvegarder la base avant exécution. Ce script est fourni à titre indicatif et
> **n'est pas exécuté par la boucle**.

## 4. Hors périmètre (résiduels assumés)

- Rotation de clé (`v1` → `v2`) et HSM/KMS : durcissement futur (le format versionné
  le permet).
- Champs non chiffrés : `Patient.notes`, `Appointment.notes/motif/modalite/lieu`,
  **binaire** des documents médicaux (chiffrement au repos Supabase Storage).
- Audit best-effort/non transactionnel (cf. story 11.3, `AUDIT-BESTEFFORT-TX`).
