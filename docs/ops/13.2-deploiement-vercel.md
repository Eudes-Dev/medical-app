# Runbook — Déploiement de production Vercel (story 13.2)

> **Périmètre.** Procédure reproductible de mise en ligne sur Vercel. Les étapes marquées
> 🧑‍🔧 **(ops/humaine)** sont **irréversibles ou touchent la production** et ne sont **jamais**
> exécutées par la boucle BMAD (garde-fou **OPS-002**). La story 13.2 livre la *configuration*
> et l'*outillage* ; ce document décrit comment un humain s'en sert pour déployer.

## 0. Pré-requis
- Compte Vercel + droits sur le projet, CLI installée : `npm i -g vercel`.
- Projet Supabase de production provisionné (DB + Auth + Storage).
- Accès aux secrets (gestionnaire de mots de passe).

## 1. Lier le dépôt au projet Vercel — 🧑‍🔧 (ops/humaine)
```bash
vercel link
```
La configuration de build est **déclarative** dans `vercel.json` :
- `"framework": "nextjs"` — préréglage explicite.
- `"regions": ["cdg1"]` — compute en **zone UE (Paris)**, cohérent RGPD (épopée 11).
- `"crons"` — rappels automatiques `/api/cron/reminders` (story 6.2), toutes les heures.

Le client Prisma est régénéré automatiquement au build par le script `postinstall`
(`prisma generate`) ajouté en 13.2 : `lib/generated/prisma` étant gitignoré, c'est ce qui
garantit un `next build` Vercel reproductible. **Ne pas** ajouter de `buildCommand` custom.

## 2. Renseigner les variables d'environnement de production — 🧑‍🔧 (ops/humaine)
Référence exhaustive : [`.env.example`](../../.env.example). Vérifier l'inventaire en local :
```bash
npm run check:prod-env
```
Le script (pur, sans accès base ni réseau) liste les variables **obligatoires manquantes ou
vides** et sort en code 1 le cas échéant. Variables **obligatoires en production** :

| Variable | Rôle |
|---|---|
| `DATABASE_URL` | Connexion applicative (pooler PgBouncer) |
| `DIRECT_URL` | Connexion directe (migrations DDL) |
| `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase client |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase serveur |
| `JWT_SECRET` | Signature cookie `booking_token` (tunnel public 4.2) |
| `DATA_ENCRYPTION_KEY` | Chiffrement clinique AES-256-GCM au repos (11.4) |
| `NEXT_PUBLIC_APP_URL` | URL publique (liens emails 6.1) |
| `RESEND_API_KEY` / `EMAIL_FROM` | Emails transactionnels (6.1) |
| `CRON_SECRET` | Auth Vercel Cron ↔ route rappels (6.2) |

Variables **optionnelles** (repli code ou module désactivé) : `SUPABASE_MEDICAL_DOCS_BUCKET`,
`NEXT_PUBLIC_CABINET_SLUG`, `PRACTITIONER_NOTIFICATION_EMAIL`, et le bloc SMS Twilio
(`SMS_ENABLED=false` par défaut → no-op ; renseigner `TWILIO_*` seulement si activé).

Poser chaque variable dans **Vercel → Project → Settings → Environment Variables** (scope
`Production`, et `Preview` au besoin). Générer les secrets aléatoires :
```bash
openssl rand -base64 32   # pour JWT_SECRET, DATA_ENCRYPTION_KEY, CRON_SECRET
```
⚠️ `DATA_ENCRYPTION_KEY` est **obligatoire en prod** : sans clé valide, le chiffrement est en
passe-plat (cf. [`docs/ops/11.4-chiffrement-applicatif.md`](./11.4-chiffrement-applicatif.md)).

## 3. Appliquer les migrations en base de production — 🧑‍🔧 (ops/humaine, OPS-002)
**Jamais automatisé** (ni en CI 13.1, ni au build Vercel). Depuis une machine de confiance,
avec `DIRECT_URL` pointant vers la **base prod** :
```bash
npx prisma migrate deploy
```
Vérifier ensuite l'état : `npx prisma migrate status`.

## 4. Pré-requis Supabase Storage — 🧑‍🔧 (ops/humaine)
Documents médicaux (story 9.2) : créer le bucket privé + RLS via
[`docs/ops/9.2-medical-documents-storage.sql`](./9.2-medical-documents-storage.sql).

## 5. Déployer — 🧑‍🔧 (ops/humaine)
```bash
vercel deploy --prod   # ou via l'intégration Git (push sur main)
```

## 6. Smoke-test post-déploiement — 🧑‍🔧 (ops/humaine)
- [ ] Page publique de réservation `/book` répond (200) et liste des créneaux.
- [ ] Connexion praticien au dashboard `/dashboard` (Supabase Auth) OK.
- [ ] Un email transactionnel de test part (Resend) ; lien d'annulation valide.
- [ ] Cron : `/api/cron/reminders` renvoie 401 sans `Authorization: Bearer $CRON_SECRET`
      (preuve que la route est protégée) ; Vercel l'appelle avec le header injecté.
- [ ] Données cliniques écrites puis relues correctement (chiffrement transparent 11.4).
