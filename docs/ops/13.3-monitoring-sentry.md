# Runbook — Monitoring des erreurs Sentry (story 13.3)

> **Sentry est désactivé par défaut.** Sans DSN, aucun client n'est initialisé (no-op total) :
> dev, preview Vercel, CI et boucle BMAD ne sont pas impactés. L'activation est une action
> humaine (création de projet + pose du DSN). 🧑‍🔧 = étape ops/humaine.

## 1. Coût
- **Plan gratuit (Developer)** suffisant pour un cabinet single-tenant : ~5 000 erreurs/mois,
  1 utilisateur, rétention courte. Voir sentry.io/pricing pour les seuils à jour.
- Alternatives 0 € : **self-hosted** Sentry (Docker) ou **GlitchTip** (compatible SDK Sentry).

## 2. Créer le projet Sentry — 🧑‍🔧
1. Compte gratuit sur https://sentry.io.
2. **Create Project** → plateforme **Next.js** → récupérer le **DSN**.

## 3. Activer en production (Vercel) — 🧑‍🔧
Poser dans **Vercel → Settings → Environment Variables** (scope Production, et Preview si voulu) :

| Variable | Rôle | Obligatoire |
|---|---|---|
| `NEXT_PUBLIC_SENTRY_DSN` | DSN (client + serveur). **Active** Sentry. | pour activer |
| `SENTRY_DSN` | DSN serveur (repli sur `NEXT_PUBLIC_SENTRY_DSN`). | non |
| `SENTRY_ENVIRONMENT` | Ex. `production` (défaut `NODE_ENV`). | non |
| `SENTRY_TRACES_SAMPLE_RATE` | Traces perf ∈ [0,1] (défaut `0` = erreurs seules). | non |
| `SENTRY_AUTH_TOKEN` / `SENTRY_ORG` / `SENTRY_PROJECT` | Upload des source maps au build. Sans token → upload désactivé, build vert. | non |

Vérifier l'inventaire : `npm run check:prod-env` (les variables Sentry sont **optionnelles**,
non bloquantes).

## 4. Source maps lisibles (optionnel) — 🧑‍🔧
Pour des stack traces dé-minifiées : créer un **Auth Token** (sentry.io → Settings → Auth
Tokens, scope `project:releases`), puis poser `SENTRY_AUTH_TOKEN`, `SENTRY_ORG`, `SENTRY_PROJECT`.
Le build uploadera alors les source maps (`next.config.ts` : `sourcemaps.disable` est levé dès
que le token est présent).

## 5. Garanties RGPD (intégrées au code)
L'application traite des **données de santé**. La configuration (`lib/observability/sentry.ts`)
impose, **avant tout envoi** :
- `sendDefaultPii: false` (pas d'IP/identité par défaut) ;
- `beforeSend = scrubSentryEvent` : suppression de `user`, `cookies`, **corps de requête**,
  en-têtes `cookie`/`authorization`, et **redaction des tokens d'URL** (`token`, `optOutToken`…) ;
- **Session Replay désactivé** (`replays*SampleRate: 0`).

⚠️ Ne pas réactiver le Replay ni `sendDefaultPii` sans revue RGPD. Toute évolution de
`scrubSentryEvent` doit conserver les tests `tests/unit/observability/sentry.test.ts` verts.

## 6. Vérifier l'activation — 🧑‍🔧
Après déploiement avec DSN : déclencher une erreur de test (ex. route de debug temporaire),
vérifier sa réception dans Sentry, puis confirmer que l'événement **ne contient aucune donnée
patient** (user/cookies/corps absents, tokens redactés).
