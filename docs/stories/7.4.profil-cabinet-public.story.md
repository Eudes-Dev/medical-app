## Story
**As a** praticien,
**I want** configurer les informations publiques de mon cabinet (nom, présentation, adresse, téléphone, e-mail de contact, accès) depuis le dashboard,
**so that** la page d'accueil publique (`/[cabinet-slug]`) affiche des informations exactes et à jour sans intervention d'un développeur.

## Contexte métier
Aujourd'hui, l'identité publique du cabinet est **codée en dur** :
- `lib/cabinet/config.ts` exporte un objet constant `CABINET_INFO` (au minimum `name`, `address`, `phone`, `openingHoursLabel`), consommé par [components/public/CabinetHeader.tsx](../../components/public/CabinetHeader.tsx) sur la landing publique.
- Pire, [components/public/HeroSection.tsx:73](../../components/public/HeroSection.tsx#L73) code en dur la chaîne **« Cabinet Rive Gauche »** directement dans le JSX (incohérence : si `CABINET_INFO.name` change, le hero ne suit pas).
- L'entrée « Profil cabinet public » des paramètres ([app/dashboard/settings/page.tsx:13-17](../../app/dashboard/settings/page.tsx#L13)) porte `storyId: "7.4"` mais **n'a pas de `href`** (placeholder non cliquable, comme 7.3 avant implémentation).

Le praticien ne peut donc **ni modifier le nom, l'adresse, le téléphone, ni rédiger une présentation** de son cabinet. Cette story introduit un **profil cabinet éditable** (single-tenant) qui remplace `CABINET_INFO`, expose un CRUD dashboard, et alimente la landing publique. C'est la 4ᵉ et dernière brique de l'épopée 7 (Paramétrage cabinet).

Elle traite aussi le **hand-off documenté de la story 7.1** ([ROADMAP](./ROADMAP.md), épopée 7) : *« dériver le label horaires public depuis `WorkingHours` »* — l'actuel `CABINET_INFO.openingHoursLabel` (texte figé) doit être remplacé par un libellé **dérivé des horaires réels** (modèle `WorkingHours`, story 7.1).

## Acceptance Criteria

1. **CRUD (profil unique)** — Une page `/dashboard/settings/profile` (espace praticien authentifié) permet d'**afficher et modifier** le profil public du cabinet. L'entrée « Profil cabinet public » de [app/dashboard/settings/page.tsx](../../app/dashboard/settings/page.tsx) (storyId 7.4, actuellement **sans `href`**) devient cliquable (`href: "/dashboard/settings/profile"`), comme 7.1/7.2/7.3. Le profil est un **singleton** (un seul cabinet — single-tenant) : sauvegarde par `upsert`, pas de liste ni de suppression.
2. **Champs du profil** — Le profil porte : `name` (nom du cabinet, requis, 2–80 car.), `tagline` (sous-titre/accroche public optionnel, ≤ 120 car.), `description` (présentation publique optionnelle, ≤ 1000 car.), `address` (adresse postale requise, ≤ 200 car.), `phone` (téléphone de contact requis, format FR validé), `email` (e-mail de contact public optionnel, format e-mail), `accessInfo` (infos d'accès/transports optionnelles, ≤ 500 car.). *(Les `photos` évoquées dans la description du placeholder sont **hors périmètre de cette story** — voir Dev Notes « Périmètre photos ».)*
3. **Exposition sur les surfaces publiques** — La page d'accueil `/[cabinet-slug]` ([app/(public)/[cabinet-slug]/page.tsx](<../../app/(public)/[cabinet-slug]/page.tsx>), y compris son `metadata`), le **layout public** ([app/(public)/layout.tsx](<../../app/(public)/layout.tsx>) — header nom + bouton téléphone + footer nom/adresse) et les composants ([CabinetHeader.tsx](../../components/public/CabinetHeader.tsx), [HeroSection.tsx](../../components/public/HeroSection.tsx)) consomment le profil **persisté** au lieu de `CABINET_INFO` / des chaînes en dur. La chaîne **« Cabinet Rive Gauche »** codée en dur dans `HeroSection` ([HeroSection.tsx:73](../../components/public/HeroSection.tsx#L73)) est remplacée par `profile.name`. Aucune valeur d'identité du cabinet ne reste codée en dur dans les surfaces publiques.
4. **Libellé horaires dérivé (hand-off 7.1)** — Le libellé d'horaires affiché publiquement (actuel `CABINET_INFO.openingHoursLabel`) est **calculé à partir des `WorkingHours`** (story 7.1) — p. ex. « Lun–Ven : 9h–18h » — via un helper pur, et non saisi manuellement. S'il n'y a aucun horaire configuré, un libellé de repli neutre est affiché (cf. Dev Notes « Dérivation du libellé horaires »).
5. **Cohérence des e-mails transactionnels** — Les 5 templates d'e-mail (`ConfirmationEmail`, `CancellationEmail`, `ReminderD1Email`, `ReminderH2Email`, `PractitionerNotifyEmail`) qui affichent aujourd'hui nom/adresse/téléphone du cabinet **depuis `CABINET_INFO`** doivent refléter le **profil persisté** (sinon : incohérence — la landing affiche le nouveau nom, les e-mails l'ancien). Le profil est injecté dans les templates (props) par la couche d'envoi qui lit le profil. *(cf. Dev Notes « Migration des e-mails » — ce volet peut être tracé/différé sur décision PO, mais l'incohérence doit être explicitement assumée si différé.)*
6. **`CABINET_INFO` réduit à un repli (pas supprimé)** — `lib/cabinet/config.ts` **n'est PAS supprimé** : il continue d'exporter `CABINET_TIMEZONE` (utilisé par 6.2/5.3) et `CABINET_DEFAULT_SLUG` (utilisé par le layout public). Seul le **rôle de source de vérité d'identité** de `CABINET_INFO` (`name`/`address`/`phone`/`openingHoursLabel`) est retiré : l'objet est conservé comme **valeurs de repli** (defaults d'amorçage) lorsque la table profil est vide (premier déploiement), et n'est plus importé directement par les surfaces publiques (vérifier par recherche globale `CABINET_INFO`).
7. **Validation** — Double validation Zod **client + serveur** (convention projet) : bornes de longueur des champs (AC 2), `phone` validé via l'utilitaire FR existant (`libphonenumber-js`, cf. `lib/sms/phone`), `email` au format e-mail. Toute écriture passe par une revalidation serveur avant persistance.
8. **Permissions & défense en profondeur** — Les Server Actions de mutation (`updateCabinetProfile`) appellent `requireUser()` (single-tenant, derrière le middleware `/dashboard`). La lecture publique du profil (`getPublicCabinetProfile`) est **publique, en lecture seule**, avec un `select` limité aux champs destinés à l'affichage (aucun champ interne sensible).
9. **États UI, a11y & responsive** — La page gère : **loading** (skeleton + bouton `isPending`/disabled), **success** (toast `sonner`), **error** (toast + conservation de la saisie). Formulaire navigable au clavier, labels associés, zones tactiles ≥ 44px, utilisable mobile (≥ 375px), WCAG AA — cohérent avec 7.1/7.2/7.3 et l'[ADR UI Shadcn](../architecture/4-ui-design-system-decision.md).
10. **Non-régression** — La migration `CabinetProfile` n'impacte ni `WorkingHours`, ni `TimeOff`, ni `ServiceType`, ni `Appointment`. Le tunnel de réservation `/[cabinet-slug]/book` et la résolution du `cabinet-slug` (route dynamique non validée — toute valeur résout vers l'unique cabinet, cf. `page.tsx`) **restent fonctionnels**. Les tests existants restent verts (mocks à étendre, pas à réécrire).

### Modèle de données — `CabinetProfile`
```prisma
/// Profil public du cabinet (story 7.4). Singleton single-tenant : une seule ligne.
/// Remplace l'ancien objet codé en dur `CABINET_INFO` (lib/cabinet/config.ts).
model CabinetProfile {
  id          String   @id @default(uuid())
  /// Nom public du cabinet (2–80 car.).
  name        String
  /// Accroche / sous-titre public optionnel (≤ 120 car.).
  tagline     String?
  /// Présentation publique optionnelle (≤ 1000 car.).
  description String?
  /// Adresse postale (≤ 200 car.).
  address     String
  /// Téléphone de contact (format FR validé côté app via libphonenumber-js).
  phone       String
  /// E-mail de contact public optionnel.
  email       String?
  /// Infos d'accès / transports optionnelles (≤ 500 car.).
  accessInfo  String?  @map("access_info")
  createdAt   DateTime @default(now()) @map("created_at")
  updatedAt   DateTime @updatedAt @map("updated_at")

  @@map("cabinet_profile")
}
```
**Choix de conception** :
- **Singleton** : pas de relation, pas de suppression. Lecture via `findFirst`, écriture via `upsert` (création de la 1ʳᵉ ligne ou mise à jour). Convention de nommage snake_case via `@map`/`@@map` (cohérent avec les modèles existants).
- **`openingHoursLabel` n'est PAS stocké** : il est **dérivé** des `WorkingHours` (AC 4) à la lecture publique — source unique de vérité = la story 7.1.

### Dérivation du libellé horaires (AC 4)
Le hand-off 7.1 demande de dériver le libellé public depuis `WorkingHours`. Helper pur `formatOpeningHoursLabel(workingHours)` :
- Entrée : les lignes `WorkingHours` **actives** (0=Dimanche … 6=Samedi, `startTime`/`endTime` « HH:mm », plusieurs plages possibles par jour).
- Sortie : libellé lisible avec regroupement des jours consécutifs identiques (p. ex. « Lun–Ven : 9h–18h · Sam : 9h–12h »).
- Repli : « Sur rendez-vous » si aucune plage active.
[Source: lib/validations/working-hours.ts (forme des plages), app/dashboard/settings/schedule/actions.ts (`getWorkingHours` renvoie les 7 jours ordonnés avec `ranges`)]

Décision recommandée : **seed d'amorçage** (Task 1) pour conserver l'affichage actuel sans rupture, puis le praticien édite. Documenter le choix retenu dans le Dev Agent Record.
