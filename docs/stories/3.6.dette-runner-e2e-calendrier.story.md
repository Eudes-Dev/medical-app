# Story 3.6: Dette technique — Runner E2E Calendrier

## Status
Ready for Review

## Story
**As a** équipe qualité / praticien,
**I want** disposer d'un runner E2E opérationnel pour le module calendrier,
**so that** les scénarios E2E définis pour les stories 3.1, 3.2 et 3.3 puissent être exécutés automatiquement et garantir la non-régression du parcours agenda.

## Contexte
Cette story est créée pour formaliser la **dette technique** identifiée lors des revues QA des stories 3.1 et 3.2 :

- **Story 3.1** : test E2E `3.1-E2E-001` (persistance Zustand sur `viewMode` et `showCancelled` après rechargement de `/dashboard/calendar`) — recommandé mais non exécuté.
- **Story 3.2** : 4 scénarios E2E définis dans [docs/qa/assessments/3.2-test-design-20260129.md](../qa/assessments/3.2-test-design-20260129.md) mais non exécutés. Le QA Gate PASS note explicitement « runner E2E calendrier à mettre en place ; recommandation future dans la gate ».
- **Story 3.3** : scénarios E2E gestion des créneaux (création / modification / annulation depuis l'agenda) à vérifier également.

Tant que cette dette n'est pas adressée, l'AC8 de la 3.2 (responsive) et la persistance des préférences ne sont validés qu'en revue manuelle.

## Acceptance Criteria
1. **Playwright** est installé et configuré pour la page `/dashboard/calendar` via `playwright.config.ts` (baseURL `http://localhost:3000`, projects `desktop-chromium` + `mobile-chromium` viewport iPhone 13). *(Décision tranchée : aucun runner E2E existant dans le projet à date — `tests/e2e/` actuel contient des tests composant Testing Library, voir AC9.)*
2. Le runner peut s'authentifier en tant que praticien via un helper `loginAsPractitioner()` qui produit un `storageState.json` réutilisable par tous les tests E2E.
3. Une base de données de test **isolée** est utilisée :
   - Postgres : schéma dédié `medical_app_test` (variable `DATABASE_URL_TEST`) avec migrations Prisma exécutées au démarrage de la suite.
   - Reset entre suites via `prisma migrate reset --force --skip-seed` puis seed dédié couvrant les statuts CONFIRMED, PENDING, CANCELLED.
   - Garde-fou : refus de démarrage si `DATABASE_URL_TEST` pointe vers une base contenant `prod` ou `production` dans son nom.
4. Le scénario **3.1-E2E-001** (persistance `viewMode` + `showCancelled` après reload de `/dashboard/calendar`) est implémenté et passe.
5. Les 4 scénarios E2E de la 3.2 (test design 2026-01-29) sont implémentés et passent : affichage des RDV, navigation prev/next/today, switch vue jour/semaine, responsive mobile (drawer).
6. Les scénarios E2E de la 3.3 (test design 2026-01-30) sont implémentés et passent : **3.3-E2E-001** (clic créneau vide → modal création), **3.3-E2E-002** (recherche patient existant), **3.3-E2E-003** (création nouveau patient depuis modal RDV), **3.3-E2E-004** (conflit horaire → toast erreur), **3.3-E2E-005** (création réussie → toast + refresh agenda), **3.3-E2E-006** (clic RDV existant → détails + actions).
7. Intégration CI — **mode différé** (décision PO 2026-05-16, Story 13.1 encore en Draft, pas de `.github/workflows/`) :
   - Le script `npm run test:e2e` doit être exécutable en local sur la machine de tout développeur sans configuration manuelle au-delà de `.env.test` (cf. AC8).
   - Un ticket de suivi `[FOLLOW-UP] Intégrer e2e-calendar à la CI` est créé dans `docs/backlog/` (ou la zone backlog applicable) référençant la Story 13.1 comme bloquante.
   - Le README E2E (AC8) doit mentionner explicitement le statut "exécution manuelle requise jusqu'à livraison 13.1".
   - Lors de la livraison de 13.1, le job `e2e-calendar` (Postgres service + cache Playwright browsers) sera ajouté à `.github/workflows/ci.yml` — non couvert par cette story.
8. Un README `tests/e2e/README.md` documente : prérequis (Node, Postgres local), configuration `.env.test`, commande de lancement, conventions de nommage des fichiers de test, lien vers les test designs 3.1/3.2/3.3.
9. **Renommage de l'arborescence existante** : le répertoire actuel `tests/e2e/` (qui contient en réalité des tests composant Testing Library : `patient-data-table.test.tsx`, `dashboard-stats.test.tsx`, etc.) est renommé en **`tests/component/`** (décision PO 2026-05-16) pour libérer `tests/e2e/` au profit des vrais tests E2E Playwright. La configuration Vitest et tout import éventuel sont mis à jour en conséquence ; les commandes existantes `npm run test:unit` et `npm run test:integration` continuent de passer.
