## Story
**As a** praticien-développeur (mainteneur solo du dépôt),
**I want** un pipeline d'intégration continue GitHub Actions qui exécute automatiquement, à chaque `push` sur `main` et à chaque pull request, les portes dures du projet (suite Vitest + build de production) ainsi que les suites E2E Playwright en opt-in,
**so that** les régressions sont détectées automatiquement avant fusion et que les suites E2E des stories 3.6 / 5.2 — jamais exécutées faute de runner — disposent enfin d'un environnement d'exécution reproductible.

> Première story de l'épopée 13 « DevOps / Sécu infra » (post-MVP). Le ROADMAP signalait
> explicitement ce besoin comme prioritaire : « la suite E2E des stories 3.6/5.2 n'est pas
> exécutée en CI tant que 13.1 n'existe pas ».
> [Source: docs/stories/ROADMAP.md — note de cadrage épopée 13 + dette TEST-BASELINE-001]
>
> Cette story livre **un seul artefact** : le workflow `.github/workflows/ci.yml`, qui
> orchestre deux jobs (`test` puis `e2e`) en s'appuyant sur l'infrastructure de test
> **déjà existante** (Vitest config, Playwright config, scripts de reset/seed E2E,
> `.env.test.example`) livrée par les stories 3.6 / 5.2. Elle **n'introduit aucun code
> applicatif** ni changement de schéma : c'est un artefact d'infrastructure d'intégration.
>
> Le volet **CD** (déploiement Vercel) reste **hors périmètre** de cette story (épopée 13.2) :
> le pipeline ne fait **pas** de `prisma migrate deploy` en production ni de déploiement —
> conformément au garde-fou OPS-002 (application en base prod = action ops humaine).

## Acceptance Criteria
1. Un workflow GitHub Actions `.github/workflows/ci.yml` (nom : `CI`) se déclenche sur `push` vers la branche `main` **et** sur tout `pull_request`. Un groupe de `concurrency` (`ci-${{ github.ref }}`, `cancel-in-progress: true`) annule les exécutions obsolètes sur une même référence.
2. Un job `test` s'exécute sur `ubuntu-latest` avec un service **Postgres jetable** (`postgres:16`, base `medical_app_ci`, health-check `pg_isready`). Les variables d'environnement `DATABASE_URL` / `DIRECT_URL` pointent vers cette base éphémère du runner ; `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY` proviennent de secrets de dépôt **avec valeurs de repli factices** (`https://dummy.supabase.co` / `dummy-anon-key`) pour que le job passe sans secrets configurés.
3. Le job `test` enchaîne : `actions/checkout@v4` → `actions/setup-node@v4` (Node 22, cache npm) → `npm ci` → `npx prisma generate`. Aucune étape n'exécute d'action contre une base de production (la seule base touchée est le service éphémère du runner).
4. Le job `test` applique **deux portes informatives** (`continue-on-error: true`, n'échouent jamais la CI) : `npm run lint` (eslint) et `npx tsc --noEmit` (type-check). Ce choix est **documenté** dans l'en-tête du workflow comme dette baseline connue (apostrophes JSX, `as any` des mocks de test, narrowing de filtres Prisma — cf. TEST-BASELINE-001).
5. Le job `test` applique **deux portes dures** (échec ⇒ CI rouge) : (a) `npm run test:run` — suite Vitest complète (unit + integration + component), indépendante de la base (Prisma/Supabase mockés) ; (b) `npm run build` — build de production Next.js, précédé de `npx prisma migrate deploy` contre la **base CI jetable** (car `next build` prérend des pages qui interrogent Prisma et exige donc une base réelle joignable, comme le build Vercel).
6. Un job `e2e` (`needs: test`, donc déclenché seulement si `test` passe) exécute les suites Playwright, en **opt-in** via la variable de dépôt `vars.E2E_ENABLED == 'true'`. Tant que cette variable n'est pas définie, le job est **sauté** — évitant des échecs sur les forks/PR sans secrets Supabase de test.
7. Le job `e2e` fournit, depuis des **secrets de dépôt** (`SUPABASE_URL_TEST`, `SUPABASE_ANON_KEY_TEST`, `SUPABASE_SERVICE_ROLE_KEY_TEST`), les variables attendues par l'infra E2E existante (`NEXT_PUBLIC_SUPABASE_URL[_TEST]`, `NEXT_PUBLIC_SUPABASE_ANON_KEY[_TEST]`, `SUPABASE_SERVICE_ROLE_KEY_TEST`, identifiants praticien E2E) ainsi qu'une base Postgres jetable dédiée (`medical_app_test`). La réinitialisation de base passe par le script existant `npm run test:e2e:db:reset` (via `global-setup.ts`), protégé par son garde-fou anti-prod.
8. Le job `e2e` installe les navigateurs Playwright (`npx playwright install --with-deps chromium`), exécute `npm run test:e2e`, et **téléverse le rapport** (`actions/upload-artifact@v4`, `playwright-report/`, rétention 7 jours, `if: !cancelled()`).
9. Le workflow **ne livre aucun code applicatif** : aucun changement de `prisma/schema.prisma`, aucune migration, aucune nouvelle dépendance npm. Il s'appuie intégralement sur les scripts npm et l'infra de test préexistants (3.6 / 5.2). Le périmètre **CD** (déploiement Vercel, story 13.2) en est explicitement exclu, de même que `prisma migrate deploy` en production (garde-fou OPS-002).
10. Les portes dures du job `test` sont **reproductibles localement** : `npm run test:run` est vert (91 fichiers / 800 tests) et `npm run build` est validé sur l'arbre courant (HEAD = story 10.2, dont la boucle a rapporté un build vert ; la CI a été introduite à un commit antérieur et fait partie de cet arbre).

## Dev Notes

### Nature de la story (infra, pas de code applicatif)
- Cette story livre **uniquement** un workflow CI (`.github/workflows/ci.yml`). Elle ne touche ni `prisma/schema.prisma`, ni `app/`, ni `components/`, ni `lib/`, ni les dépendances. Elle **consomme** les scripts npm (`test:run`, `build`, `test:e2e`, `test:e2e:db:reset`) et l'infra de test déjà livrés par 3.6 (runner E2E) et 5.2 (sécurisation/tests).
[Source: .github/workflows/ci.yml, package.json#scripts]

### Portes dures vs informatives (choix assumé)
- **Dures** (échec ⇒ CI rouge) : `npm run test:run` (Vitest) et `npm run build`. Ce sont les garanties de non-régression fonctionnelle et de buildabilité prod.
- **Informatives** (`continue-on-error: true`) : `npm run lint` et `npx tsc --noEmit`. Le dépôt porte une **dette baseline connue** (apostrophes JSX, `as any` des mocks de test, narrowing de filtres Prisma) — cf. `TEST-BASELINE-001` du ROADMAP. Les rendre bloquantes ferait échouer la CI sur de la dette pré-existante non liée à la PR courante. Choix cohérent avec les notes des stories 9.4/10.x (« dette baseline tolérée — cf. CI 13.1 »).
[Source: docs/stories/ROADMAP.md#dette-transverse — TEST-BASELINE-001]

### Pourquoi un Postgres jetable dans le job `test`
- Vitest **mocke** Prisma/Supabase → la suite n'a pas besoin de base. Mais `next build` **prérend** des pages qui interrogent Prisma (ex. `/unsubscribe`, dashboard) et exige donc une base **réelle joignable**, exactement comme le build Vercel en prod. D'où le service `postgres:16` éphémère + `prisma migrate deploy` sur cette base jetable (jamais une base prod). Base vide → `findFirst`/`findMany` renvoient `null`/`[]`, gérés par les pages.
[Source: .github/workflows/ci.yml (commentaires job `test`)]

### E2E en opt-in (garde-fou forks/secrets)
- Le job `e2e` est **sauté** tant que `vars.E2E_ENABLED != 'true'`. Cela évite des échecs systématiques sur les PR/forks ne disposant pas des secrets Supabase de test, tout en gardant le pipeline prêt à exécuter 3.6/5.2 dès que les secrets sont posés. La réinitialisation de base réutilise `reset-e2e-db.ts` / `global-setup.ts` **avec leur garde-fou anti-prod** (refus si l'URL contient `prod`/`production`).
[Source: tests/e2e/global-setup.ts, scripts/reset-e2e-db.ts, .env.test.example]

### Hors périmètre (CD / OPS)
- **Pas** de déploiement Vercel (story 13.2), **pas** de `prisma migrate deploy` en production (OPS-002 — action ops humaine), **pas** de monitoring Sentry (13.3) ni de backups (13.4). Le pipeline s'arrête à la validation (CI), pas à la livraison (CD).
[Source: docs/stories/ROADMAP.md#épopée-13, docs/stories/5.3.consolidation-pre-prod.story.md — OPS-002]

### Secrets / variables de dépôt requis (config GitHub humaine)
- Variable : `E2E_ENABLED=true` (Settings → Variables) pour activer le job e2e.
- Secrets (Settings → Secrets → Actions) : `SUPABASE_URL_TEST`, `SUPABASE_ANON_KEY_TEST`, `SUPABASE_SERVICE_ROLE_KEY_TEST` (cf. `.env.test.example`). Le job `test` fonctionne **sans** secret (replis factices).
[Source: .github/workflows/ci.yml (en-tête), .env.test.example]
