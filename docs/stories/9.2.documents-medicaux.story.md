# Story 9.2: Documents médicaux

## Status
Done

## Story
**As a** praticien,
**I want** rattacher des documents (ordonnances, comptes rendus, imagerie, analyses…) au dossier d'un patient, les lister, les télécharger et les supprimer,
**so that** je centralise les pièces du dossier patient de façon sécurisée et consultable depuis sa fiche.

> Deuxième story de l'épopée 9 « Dossier patient ». Elle introduit le **stockage
> de fichiers** du projet, conformément à l'ADR
> [docs/architecture/5-stockage-fichiers-decision.md](../architecture/5-stockage-fichiers-decision.md) :
> binaire dans un **bucket Supabase Storage privé**, **métadonnées** en Postgres,
> accès par **URL signées courtes** + `requireUser()`, aucune URL publique.

> ⚠️ **Réserve RGPD (mise en production conditionnée).** Les documents médicaux sont
> des données de santé (art. 9 RGPD). Cette story livre le **socle minimal** sécurisé
> (bucket privé, URL signées, allowlist MIME, plafond de taille) et est **développable
> et testable en dev/preview**. Sa **mise en production** doit être conditionnée à un
> arbitrage PO/RGPD (épopée 11 : chiffrement 11.4, audit 11.3, consentement/oubli
> 11.1/11.2 — non livrés). [Source: docs/architecture/5-stockage-fichiers-decision.md#cadrage-rgpd--sécurité-important]

## Acceptance Criteria
1. Un nouveau modèle Prisma `MedicalDocument` est ajouté : `id`, `patientId` (FK `Patient`, `onDelete: Cascade`), `storagePath`, `fileName`, `mimeType`, `sizeBytes` (Int), `category` (enum `MedicalDocumentCategory`), `createdAt`, `updatedAt`, avec `@@index([patientId])` et la relation inverse `medicalDocuments` sur `Patient`. Un enum `MedicalDocumentCategory { PRESCRIPTION, REPORT, IMAGING, ANALYSIS, OTHER }` est ajouté. Une migration SQL **non destructive** (aucun DROP/ALTER de table existante) accompagne le changement.
2. La validation des métadonnées passe par un schéma Zod isomorphe (`medicalDocumentSchema`) : `fileName` trimé non vide ≤ 255 car. ; `mimeType` ∈ **allowlist** (`application/pdf`, `image/jpeg`, `image/png`) ; `sizeBytes` entier > 0 et ≤ **10 Mo** ; `category` ∈ valeurs de l'enum. Le binaire n'est **jamais** stocké en base.
3. Le chemin de stockage est construit côté serveur via `buildMedicalDocumentPath(patientId, mimeType)` au format `patients/{patientId}/{uuidV4}.{ext}` ; **l'extension dérive du `mimeType` de l'allowlist** (pas du nom de fichier utilisateur). Le `fileName` d'origine n'est conservé qu'en métadonnée et **jamais** utilisé comme chemin (anti-traversal). [Source: docs/architecture/5-stockage-fichiers-decision.md#2-chemin-de-stockage-anti-collision--anti-traversal]
4. Une abstraction `lib/storage/medical-documents.ts` encapsule le client Supabase Storage (bucket privé `SUPABASE_MEDICAL_DOCS_BUCKET`, défaut `medical-documents`) : `createUploadUrl(path)` (URL d'upload signée), `createDownloadUrl(path, expiresIn)` (URL de lecture signée, TTL ≤ 60 s), `removeObject(path)`. Aucune URL publique n'est générée.
5. Quatre Server Actions, **toutes** protégées par `requireUser()` et `assertValidUuid()` sur tout `id`, sont disponibles dans `app/dashboard/patients/medical-document-actions.ts` :
   - `getMedicalDocuments(patientId)` → `MedicalDocumentData[]` triés du plus récent au plus ancien (sans exposer `storagePath`) ;
   - `createMedicalDocument(patientId, meta)` → valide `meta`, construit le chemin, crée l'URL d'upload signée, persiste la ligne de métadonnées, revalide la fiche ; retourne `{ success, document, upload: { bucket, path, token, signedUrl } }` ;
   - `getMedicalDocumentDownloadUrl(documentId)` → URL de lecture signée courte ;
   - `deleteMedicalDocument(documentId)` → supprime l'objet Storage (best-effort) **puis** la ligne, et revalide la fiche.
6. La fiche patient (`/dashboard/patients/[id]`) affiche une section « Documents médicaux » listant les documents (récent → ancien) avec, pour chacun : `fileName`, badge de `category`, taille lisible (Ko/Mo), date FR ; et des actions **Télécharger** et **Supprimer** (avec confirmation).
7. Le praticien peut **ajouter** un document : choix d'un fichier + d'une catégorie, validation **client et serveur** (type hors allowlist ou taille > 10 Mo refusés sans appel Prisma côté serveur), upload du binaire vers l'URL signée via le client navigateur Supabase ; à la réussite, toast de succès + liste rafraîchie. Si l'upload binaire échoue après création de la ligne, la ligne est **annulée** (rollback via `deleteMedicalDocument`) et un toast d'erreur s'affiche.
8. La suppression d'un patient purge ses documents en base (cascade FK). La purge des **objets Storage** correspondants n'étant pas garantie par la cascade SQL, ce point est tracé comme limite connue (nettoyage applicatif à l'échelle patient hors périmètre de cette story). [Source: docs/architecture/5-stockage-fichiers-decision.md#5-cohérence-suppression]

## Dev Notes

### Décision d'architecture (socle de cette story)
- ADR §5 « Stockage de fichiers » : Supabase Storage **bucket privé** + métadonnées Postgres ; binaire jamais en base ni en URL publique ; accès par URL signées courtes + `requireUser()` ; chemin serveur `patients/{patientId}/{uuid}.{ext}` ; abstraction `lib/storage/` mockable ; bucket/RLS = étape ops hors boucle.
[Source: docs/architecture/5-stockage-fichiers-decision.md]

### Logique de Données
- Relations : Patient (1) <---> (N) MedicalDocument (onDelete Cascade côté MedicalDocument).
- Accès : Server Actions uniquement (pas d'API route). Single-tenant : pas de `userId`/`tenantId`.
[Source: docs/architecture/2-logique-de-donnes.md, prisma/schema.prisma]

### Sécurité (héritée 5.2 / 5.3, identique à 9.1)
- Toutes les Server Actions commencent par `await requireUser()` (`@/lib/server/auth`).
- Tout `id` est validé via `assertValidUuid()` (`@/lib/validations/uuid`) avant tout appel Prisma/Storage.
- Inputs validés par Zod isomorphe ; Prisma paramétré (aucune interpolation SQL brute).
- Allowlist MIME stricte + plafond de taille ; chemin de stockage **dérivé du mime**, jamais du nom de fichier (anti-traversal).
[Source: app/dashboard/patients/consultation-note-actions.ts, lib/validations/uuid.ts, docs/architecture/5-stockage-fichiers-decision.md#4-validation-zod-isomorphe]

### Convention Server Actions (forme de retour)
- Mutations : `{ success: true; … } | { success: false; error: string }` (comme `createConsultationNote`).
- `getMedicalDocuments` renvoie directement `MedicalDocumentData[]` (ou lève `UnauthorizedError`), UUID invalide → `[]`.
[Source: app/dashboard/patients/consultation-note-actions.ts]

### Upload — flux à 2 temps
1. Server Action `createMedicalDocument` : valide la **métadonnée**, crée l'URL d'upload signée (`createSignedUploadUrl`), persiste la ligne, retourne `{ bucket, path, token, signedUrl }`.
2. Client : `createClient()` (`@/lib/supabase/client`) → `storage.from(bucket).uploadToSignedUrl(path, token, file)`.
3. En cas d'échec d'upload, rollback de la ligne via `deleteMedicalDocument`.
> Limite connue assumée (dev/preview) : une ligne peut être orpheline si le rollback échoue lui-même ; acceptable pour le socle, à durcir avec l'épopée 11 (audit/cohérence).
[Source: docs/architecture/5-stockage-fichiers-decision.md#3-accès--bucket-privé--url-signées-courtes]

### Migration (format attendu)

en-tête de commentaire (story + « migration non destructive »), `CREATE TYPE` (enum), `CREATE TABLE "medical_documents"`,
`CREATE INDEX`, puis `ADD CONSTRAINT … FOREIGN KEY … ON DELETE CASCADE` (patient). Colonnes snake_case, `created_at` `DEFAULT CURRENT_TIMESTAMP`, `size_bytes` `INTEGER`.
[Source: prisma/migrations/20260625000010_add_waitlist_entries/migration.sql]
