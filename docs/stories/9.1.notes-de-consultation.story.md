# Story 9.1: Notes de consultation

## Status
Done

## Story
**As a** praticien,
**I want** consigner des notes de consultation horodatées dans le dossier d'un patient (et, si pertinent, les rattacher à un rendez-vous précis),
**so that** je conserve un historique clinique structuré et chronologique de chaque patient, consultable et modifiable depuis sa fiche.

> Première story de l'épopée 9 « Dossier patient » (post-MVP). Elle introduit un
> historique de notes **structuré** (liste datée d'entrées), distinct du champ
> libre unique `Patient.notes` (note administrative one-shot, conservé tel quel).

## Acceptance Criteria
1. Un nouveau modèle Prisma `ConsultationNote` est ajouté : `id`, `patientId` (FK `Patient`, `onDelete: Cascade`), `appointmentId` optionnel (FK `Appointment`, `onDelete: SetNull`), `content`, `createdAt`, `updatedAt`. Une migration SQL **non destructive** (aucun DROP/ALTER de table existante) accompagne le changement de schéma.
2. Le contenu d'une note est validé par un schéma Zod isomorphe (`consultationNoteSchema`) : `content` trimé, non vide, ≤ 5000 caractères ; `appointmentId` optionnel et, s'il est fourni, UUID valide.
3. Quatre Server Actions sont disponibles et **toutes** protégées par `requireUser()` : `getConsultationNotes(patientId)`, `createConsultationNote(patientId, data)`, `updateConsultationNote(noteId, data)`, `deleteConsultationNote(noteId)`. Tout `id` reçu est validé via `assertValidUuid()` avant l'appel Prisma.
4. La fiche patient (`/dashboard/patients/[id]`) affiche une section « Notes de consultation » listant les notes du patient, **triées de la plus récente à la plus ancienne**, chacune avec sa date/heure de création (format FR) et, le cas échéant, l'indication du RDV rattaché.
5. Le praticien peut **ajouter** une note via un formulaire (textarea) ; à la réussite, un toast de succès s'affiche, le champ est vidé et la liste est revalidée/rafraîchie.
6. Le praticien peut **modifier** une note existante (édition inline) et **supprimer** une note (avec confirmation), avec toasts de succès/erreur dans les deux cas.
7. La validation échoue proprement (note vide ou > 5000 caractères) côté client **et** serveur, sans appel Prisma côté serveur, et affiche un message d'erreur lisible.
8. La suppression d'un patient purge ses notes (cascade), et la suppression d'un rendez-vous détache la note sans la perdre (`appointmentId` → null).

## Dev Notes

### Logique de Données
- Relations : Patient (1) <---> (N) ConsultationNote ; Appointment (1) <---> (0..N) ConsultationNote (lien optionnel).
- Accès : couche de services via Next.js Server Actions uniquement (pas d'API route).
- Single-tenant : pas de `userId`/`tenantId` (cohérent avec tous les modèles existants).
[Source: docs/architecture/2-logique-de-donnes.md, prisma/schema.prisma]

### Sécurité (héritée des stories 5.2 / 5.3)
- **Toutes** les Server Actions commencent par `await requireUser()` (`@/lib/server/auth`).
- Tout `id` utilisateur est validé via `assertValidUuid()` (`@/lib/validations/uuid`) **avant** tout appel Prisma.
- Les inputs ne sont consommés que via Prisma paramétré (aucune interpolation SQL brute) et validés par Zod.
[Source: app/dashboard/patients/actions.ts, lib/validations/uuid.ts]

### Distinction avec `Patient.notes`
- `Patient.notes` (champ `String?` existant) reste une **note administrative libre unique** — NE PAS le supprimer ni le réutiliser pour l'historique.
- `ConsultationNote` est un **historique daté** de plusieurs entrées. Les deux coexistent.
[Source: prisma/schema.prisma — model Patient]

### Convention Server Actions (forme de retour)
Mêmes signatures que `createPatient`/`updatePatient`/`deletePatient` :
```typescript
| { success: true; note: ConsultationNoteData }
| { success: false; error: string }
```
`getConsultationNotes` renvoie directement `ConsultationNoteData[]` (ou lève `UnauthorizedError`), comme `getPatients`/`getPatientById`.
[Source: app/dashboard/patients/actions.ts]

### Migration (format attendu)
Reprendre EXACTEMENT le style de `prisma/migrations/20260625000010_add_waitlist_entries/migration.sql` :
en-tête de commentaire (story + « migration non destructive »), `CREATE TABLE "consultation_notes"`,
`CREATE INDEX`, puis `ADD CONSTRAINT … FOREIGN KEY … ON DELETE CASCADE` (patient) et `ON DELETE SET NULL` (appointment).
Colonnes en snake_case (`patient_id`, `appointment_id`, `created_at`, `updated_at`), `created_at` `DEFAULT CURRENT_TIMESTAMP`.
> ⚠️ Garde-fou boucle : **ne PAS** lancer `prisma migrate deploy` / `migrate dev` contre une base distante. Le fichier de migration est créé à la main + `prisma generate` pour les types. La suite Vitest mocke Prisma (aucune base requise).
[Source: prisma/migrations/20260625000010_add_waitlist_entries/migration.sql]
