# Story 9.3: Antécédents médicaux

## Status
Done

## Story
**As a** praticien,
**I want** consigner et tenir à jour les antécédents médicaux d'un patient — allergies, traitements en cours, antécédents chirurgicaux et familiaux — sous forme d'entrées catégorisées,
**so that** je dispose d'une vue clinique de fond, structurée et lisible, distincte de l'historique daté des consultations, directement depuis la fiche du patient.

> Troisième story de l'épopée 9 « Dossier patient » (post-MVP), après 9.1 (notes
> de consultation, **Done**) et 9.2 (documents médicaux, **Done**). Elle complète
> le sous-onglet « Antécédents médicaux » prévu pour la fiche patient.
> [Source: docs/specifications-fonctionnelles/espace-medecin-sidebar.md#3-patients----dashboardpatients]
>
> CRUD de **données structurées** (texte catégorisé), **sans stockage de fichier** :
> le patron de référence est la story 9.1 (notes de consultation), enrichi d'un
> **enum de catégorie** (comme la catégorie de la story 9.2). Aucune dépendance à
> Supabase Storage.

## Acceptance Criteria
1. Un nouveau modèle Prisma `MedicalHistoryEntry` est ajouté : `id`, `patientId` (FK `Patient`, `onDelete: Cascade`), `category` (enum `MedicalHistoryCategory`), `content`, `createdAt`, `updatedAt`, avec `@@index([patientId])` et la relation inverse `medicalHistoryEntries` sur `Patient`. Un enum `MedicalHistoryCategory { ALLERGY, CURRENT_TREATMENT, SURGICAL_HISTORY, FAMILY_HISTORY, OTHER }` est ajouté. Une migration SQL **non destructive** (aucun DROP/ALTER de table ou colonne existante) accompagne le changement de schéma.
2. Le contenu d'une entrée est validé par un schéma Zod isomorphe (`medicalHistoryEntrySchema`) : `content` trimé, non vide, ≤ 2000 caractères ; `category` ∈ valeurs de l'enum `MedicalHistoryCategory`. La validation échoue proprement (contenu vide / > 2000 caractères / catégorie hors enum) côté client **et** serveur, sans appel Prisma côté serveur, avec un message d'erreur lisible.
3. Quatre Server Actions sont disponibles dans `app/dashboard/patients/medical-history-actions.ts` (`"use server"`, **pas** une route), **toutes** protégées par `requireUser()`, et tout `id` reçu est validé via `assertValidUuid()` **avant** tout appel Prisma :
   - `getMedicalHistoryEntries(patientId)` → `MedicalHistoryEntryData[]` ;
   - `createMedicalHistoryEntry(patientId, data)` → `{ success: true; entry } | { success: false; error }` ;
   - `updateMedicalHistoryEntry(entryId, data)` → même forme de retour ;
   - `deleteMedicalHistoryEntry(entryId)` → `{ success: true } | { success: false; error }`.
   Chaque mutation appelle `revalidatePath('/dashboard/patients/{patientId}')` (le `patientId` de mise à jour/suppression est dérivé de la ligne retournée par Prisma).
4. La fiche patient (`/dashboard/patients/[id]`) affiche une section « Antécédents médicaux » regroupant les entrées **par catégorie** (Allergies, Traitements en cours, Antécédents chirurgicaux, Antécédents familiaux, Autres) ; à l'intérieur de chaque groupe, les entrées sont triées de la **plus récente à la plus ancienne** (`createdAt desc`). Un libellé FR lisible est affiché pour chaque catégorie. Les catégories sans entrée ne créent pas de bruit visuel inutile (un état vide global clair est affiché si le patient n'a aucun antécédent).
5. Le praticien peut **ajouter** une entrée via un formulaire : sélecteur de catégorie + zone de texte (`Textarea`) + bouton « Ajouter ». À la réussite : toast de succès, champ vidé, liste rafraîchie (`router.refresh()`).
6. Le praticien peut **modifier** une entrée existante (édition inline : contenu **et** catégorie) et **supprimer** une entrée (avec `window.confirm`), avec toasts de succès/erreur dans les deux cas.
7. Les libellés et messages utilisateur passent par les catalogues centralisés : une clé `medicalHistory` est ajoutée à `lib/ui/toast-messages.ts` (`created`, `updated`, `deleted`) ; les libellés FR des catégories sont exposés de façon réutilisable (constante partagée dans `lib/validations/medical-history.ts`).
8. La suppression d'un patient purge ses antécédents (cascade FK). Le champ administratif libre existant `Patient.notes` reste **inchangé** et n'est ni réutilisé ni supprimé (les antécédents structurés coexistent avec lui, comme les notes 9.1).

## Dev Notes

### Distinction avec les autres surfaces du dossier patient
- `Patient.notes` (`String?`) : note administrative libre **unique** — laissée intacte (ne pas réutiliser ni supprimer).
- `ConsultationNote` (9.1) : **historique daté** d'entrées de consultation (chronologie clinique).
- `MedicalDocument` (9.2) : **pièces jointes** (fichiers) du dossier.
- `MedicalHistoryEntry` (cette story) : **fond clinique structuré et catégorisé** (allergies, traitements, ATCD), sans dimension chronologique de consultation et sans fichier. Les quatre coexistent.
[Source: prisma/schema.prisma — model Patient, ConsultationNote, MedicalDocument]

### Logique de Données
- Relations : Patient (1) <---> (N) MedicalHistoryEntry (`onDelete: Cascade` côté MedicalHistoryEntry).
- Accès : Server Actions uniquement (pas d'API route). Single-tenant : pas de `userId`/`tenantId` (cohérent avec tous les modèles existants).
- Pas de FK vers `Appointment` (contrairement à 9.1) : un antécédent n'est pas rattaché à un RDV.
[Source: docs/architecture/2-logique-de-donnes.md, prisma/schema.prisma]

### Sécurité (héritée 5.2 / 5.3, identique à 9.1 / 9.2)
- **Toutes** les Server Actions commencent par `await requireUser()` (`@/lib/server/auth`).
- Tout `id` est validé via `assertValidUuid()` (`@/lib/validations/uuid`) **avant** tout appel Prisma.
- Inputs validés par Zod isomorphe ; Prisma paramétré (aucune interpolation SQL brute).
[Source: app/dashboard/patients/consultation-note-actions.ts, lib/validations/uuid.ts]

### Convention Server Actions (forme de retour)
Mêmes signatures que `createConsultationNote`/`updateConsultationNote`/`deleteConsultationNote` :
```typescript
| { success: true; entry: MedicalHistoryEntryData }
| { success: false; error: string }
```
`getMedicalHistoryEntries` renvoie directement `MedicalHistoryEntryData[]` (ou lève `UnauthorizedError`), UUID invalide → `[]`.
[Source: app/dashboard/patients/consultation-note-actions.ts]

### Migration (format attendu)

en-tête de commentaire (story + « migration non destructive »), `CREATE TYPE "MedicalHistoryCategory" AS ENUM (...)`, `CREATE TABLE "medical_history_entries"` (colonnes snake_case, `created_at` `DEFAULT CURRENT_TIMESTAMP`, `category "MedicalHistoryCategory" NOT NULL`), `CREATE INDEX`, puis `ADD CONSTRAINT … FOREIGN KEY ("patient_id") REFERENCES "patients" ("id") ON DELETE CASCADE ON UPDATE CASCADE`.
