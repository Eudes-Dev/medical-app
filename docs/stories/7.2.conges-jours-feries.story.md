# Story 7.2: Congés et jours fériés

## Status
Done

## Story
**As a** praticien,
**I want** bloquer des journées entières ou des plages horaires spécifiques (congés, formation, jours fériés),
**so that** aucun rendez-vous ne puisse être pris sur ces périodes via le tunnel public, et que ces absences soient visibles dans mon agenda.

## Contexte métier
La story **7.1** a remplacé les horaires figés par un modèle persistant `WorkingHours` et a refactoré le générateur de créneaux pour qu'il lise la base. Elle a **explicitement laissé un point d'extension** pour la 7.2 : la fonction pure `filterAvailableSlots(date, appointments, ranges)` ([lib/cabinet/slots.ts:84](../../lib/cabinet/slots.ts#L84)) où *« les exceptions `TimeOff` (congés/fériés) viendront retrancher des créneaux par-dessus la sortie de `generateDaySlots` »*.

Aujourd'hui un praticien ne peut donc proposer des créneaux que sur ses plages hebdomadaires récurrentes : il **ne peut pas** poser une semaine de congés, bloquer une après-midi de formation, ni neutraliser les jours fériés. Cette story ajoute une couche d'**exceptions** (`TimeOff`) qui se superpose aux horaires : congés (une ou plusieurs journées), absences partielles (plage intra-journée) et jours fériés français pré-calculés et désactivables. C'est la 2ᵉ brique de l'épopée 7 (Paramétrage cabinet) et un **prérequis de la story 8.1** (annulation/reprogrammation).
[Source: docs/stories/ROADMAP.md — dépendance `7.1 → 7.2 → 8.1` ; docs/stories/7.1.configuration-horaires.story.md « Extension future (story 7.2) »]

## Acceptance Criteria

1. **Page de gestion** — Une page `/dashboard/settings/timeoff` (espace praticien authentifié) permet de **créer, lister et supprimer** des exceptions de deux natures : (a) **journée(s) entière(s)** définies par une plage de dates `startDate → endDate` (inclusives, `endDate ≥ startDate`, `allDay = true`) ; (b) **plage intra-journée** sur un **jour unique** (`startDate = endDate`, `allDay = false`, avec `startTime`/`endTime` "HH:mm"). Un motif (`reason`) optionnel peut être saisi.
2. **Jours fériés français** — Les jours fériés FR métropole de l'année courante (et de l'année suivante) sont **pré-remplis** (calcul local, voir Dev Notes « Jours fériés ») et présentés dans une section dédiée. Chacun est **désactivable sans suppression** (`active = false`, même sémantique que les plages 7.1) : un férié désactivé **ne bloque plus** les créneaux mais reste listé et réactivable. Un sélecteur d'année permet de consulter/ajuster les fériés d'une année donnée.
3. **Masquage des créneaux (enforcement)** — Toute exception **active** masque les créneaux concernés : (a) **dans le tunnel public `/book`** — une journée entièrement bloquée renvoie `{ slots: [] }` ; une plage intra-journée retranche les créneaux qui la chevauchent (enforcement **dur**, source de vérité) ; (b) **dans le calendrier praticien** `/dashboard/calendar` — les périodes bloquées sont **visualisées** (overlay « Congé / Férié ») pour information.
4. **Avertissement RDV impactés** — À la création d'une exception, le système calcule **de façon synchrone** les `Appointment` non-`CANCELLED` tombant dans la période bloquée et affiche un **avertissement** listant ces RDV. Le praticien peut alors choisir explicitement de **prévenir les patients** : les RDV impactés passent en `CANCELLED` et une **notification d'annulation** est envoyée (réutilisation de l'email d'annulation 6.1 ; SMS hors périmètre, voir Dev Notes). Sans action explicite, l'exception est créée **sans** toucher aux RDV existants (le praticien reste maître).
5. **Validation** — Double validation Zod **client + serveur** : `endDate ≥ startDate` ; si `allDay = false` alors `startDate = endDate` **et** `endTime > startTime` (format "HH:mm") ; `reason` borné (≤ 200 car.) ; dates cohérentes (pas de borne dans un passé lointain arbitraire — voir cas limites). Toute écriture passe par une revalidation serveur avant persistance.
6. **Permissions & défense en profondeur** — Toutes les Server Actions de `/dashboard/settings/timeoff` appellent `requireUser()` (modèle single-tenant, derrière le middleware `/dashboard`). En complément, le tunnel public **`createGuestBooking` rejette** une réservation tombant sur une exception active (un client obsolète ne doit pas pouvoir réserver un créneau bloqué) → `SLOT_TAKEN`.
7. **États UI** — La page gère explicitement : **loading** (skeleton initial + boutons `isPending`/disabled), **empty** (aucune exception → message + CTA « Ajouter un congé »), **success** (toast `sonner`), **error** (toast + conservation de la saisie dans le formulaire). La confirmation d'annulation des RDV impactés est une étape distincte et explicite (pas de destruction silencieuse).
8. **Cas limites** — Exception couvrant un jour déjà fermé (aucune plage 7.1) ⇒ aucun effet visible, sans erreur. Chevauchement de deux exceptions ⇒ toléré (union des blocages, idempotent côté générateur). Plage intra-journée ne couvrant aucun créneau ⇒ no-op. Suppression d'une exception ⇒ les créneaux redeviennent disponibles immédiatement (tunnel relit la base à chaque appel). Désactivation d'un férié ⇒ ses créneaux réapparaissent.
9. **Responsive & accessibilité** — Interface utilisable mobile (≥ 375px) et desktop ; zones tactiles ≥ 44px. Sélecteur de dates (DateRangePicker) **accessible** : navigation clavier, `aria-label` des jours, focus visible, libellés associés. Switch des fériés avec label accessible (réutilise le `Switch` a11y de la 7.1). A11y WCAG AA.
10. **Non-régression 7.1 / tunnel** — Le générateur reste compatible : `filterAvailableSlots` reçoit un **4ᵉ paramètre optionnel** `timeOffs` (défaut `[]`) ⇒ les tests et appels existants de la 7.1 restent verts sans modification. Une **migration** crée `TimeOff` sans impacter `WorkingHours`. Le tunnel `/book` (4.1/4.2) et la génération 7.1 continuent de fonctionner à l'identique en l'absence d'exception.

### Modèle de données — `TimeOff`
```prisma
/// Source d'une exception d'agenda.
enum TimeOffSource {
  /// Saisie manuelle du praticien (congé, formation…).
  MANUAL
  /// Jour férié français matérialisé automatiquement (désactivable).
  HOLIDAY
}

/// Exceptions d'agenda du cabinet (story 7.2) : congés, absences, jours fériés.
/// Se superposent aux `WorkingHours` (7.1) pour retrancher des créneaux.
/// Single-tenant : pas de `userId`/`tenantId`.
model TimeOff {
  id        String        @id @default(uuid())
  /// Premier jour bloqué (inclus). Type `Date` (pas de composante horaire) pour
  /// éviter toute ambiguïté de fuseau au stockage — comparaison calendaire.
  startDate DateTime      @db.Date @map("start_date")
  /// Dernier jour bloqué (inclus). `endDate >= startDate`. Pour une plage
  /// intra-journée, `endDate == startDate`.
  endDate   DateTime      @db.Date @map("end_date")
  /// true = journée(s) entière(s) bloquée(s) ; false = plage intra-journée.
  allDay    Boolean       @default(true) @map("all_day")
  /// Heure de début "HH:mm" si plage intra-journée (sinon null).
  startTime String?       @map("start_time")
  /// Heure de fin "HH:mm" (exclusive) si plage intra-journée (sinon null).
  endTime   String?       @map("end_time")
  /// Motif optionnel (≤ 200 car.) : "Congés", "Formation", libellé du férié…
  reason    String?
  /// Origine de l'exception.
  source    TimeOffSource @default(MANUAL)
  /// Exception active (false = désactivée sans suppression — fériés notamment).
  active    Boolean       @default(true)
  createdAt DateTime      @default(now()) @map("created_at")
  updatedAt DateTime      @updatedAt @map("updated_at")

  @@index([startDate, endDate])
  @@map("time_off")
}
```
**Choix de conception** :
- `@db.Date` pour `startDate`/`endDate` : ce sont des **jours calendaires**, pas des instants. On compare jour-à-jour, on évite la dérive TZ au stockage (même esprit que les "HH:mm" `String` de la 7.1).
- `allDay` + (`startTime`/`endTime` nullables) : une exception est soit **journée(s) entière(s)**, soit **une plage sur un jour unique** (`startDate == endDate`). On **n'autorise pas** une plage horaire récurrente sur plusieurs jours (ambiguïté évitée — voir AC 1/5). C'est une décision PO de simplification ; le besoin « récurrent » relèvera d'une story future si nécessaire.
- `active` reprend exactement la sémantique 7.1 (désactiver sans supprimer) ⇒ cohérence et réutilisation directe pour les fériés « désactivables » (AC 2).

### Jours fériés (AC 2) — décision : calcul local, pas d'API runtime
**Décision PO** : calculer les fériés **en local** (module pur `lib/cabinet/holidays.ts`) plutôt que d'appeler une API externe au runtime. Justification : pas de dépendance réseau ni de point de défaillance/latence sur un chemin public, testabilité totale, aucun secret/clé. L'API gouvernementale Etalab (`calendrier.api.gouv.fr`) sert de **référence de correction** pour les tests, pas de source runtime.

Fériés **France métropole** (11) : Jour de l'an (01/01), Lundi de Pâques (Pâques+1), Fête du Travail (01/05), Victoire 1945 (08/05), Ascension (Pâques+39), Lundi de Pentecôte (Pâques+50), Fête nationale (14/07), Assomption (15/08), Toussaint (01/11), Armistice (11/11), Noël (25/12). *(Vendredi Saint et 26/12 = Alsace-Moselle uniquement ⇒ exclus, à prévoir en option future si besoin régional.)*

Calcul de Pâques (algorithme grégorien anonyme / Meeus-Butcher) :
```typescript
// lib/cabinet/holidays.ts (module pur, sans Prisma)
function easterSunday(year: number): Date {
  const a = year % 19, b = Math.floor(year / 100), c = year % 100;
  const d = Math.floor(b / 4), e = b % 4, f = Math.floor((b + 8) / 25);
  const g = Math.floor((b - f + 1) / 3), h = (19 * a + b - d - g + 15) % 30;
  const i = Math.floor(c / 4), k = c % 4;
  const l = (32 + 2 * e + 2 * i - h - k) % 7;
  const m = Math.floor((a + 11 * h + 22 * l) / 451);
  const month = Math.floor((h + l - 7 * m + 114) / 31);   // 3=mars, 4=avril
  const day = ((h + l - 7 * m + 114) % 31) + 1;
  return new Date(Date.UTC(year, month - 1, day));
}
```

**Matérialisation (AC 2 « pré-remplis annuellement »)** : pas de cron. À l'ouverture de la page (`getTimeOffs(year)`), `ensureHolidaysMaterialized(year)` insère les fériés de l'année en lignes `TimeOff` (`source=HOLIDAY`, `allDay=true`, `reason=label`, `active=true`) **si absents**. « Désactivable » = `toggleHoliday(id, false)`. Matérialiser (vs calculer à la volée dans le générateur) permet de **persister l'état désactivé** sans table dédiée.

### Algorithme — composition `TimeOff × créneaux` (module pur)
```typescript
// lib/cabinet/time-off.ts (pur)
export interface TimeOffInterval {
  startDate: Date; endDate: Date; allDay: boolean;
  startTime: string | null; endTime: string | null;
}
// Comparaison CALENDAIRE (year/month/date), insensible à l'heure et donc robuste
// au point TZ tant que `date` et les bornes désignent le même jour local.
const dayKey = (d: Date) => d.getFullYear() * 10000 + d.getMonth() * 100 + d.getDate();
export const isDayFullyBlocked = (date: Date, offs: TimeOffInterval[]): boolean =>
  offs.some((o) => o.allDay && dayKey(o.startDate) <= dayKey(date) && dayKey(date) <= dayKey(o.endDate));
export const slotInPartialTimeOff = (
  slot: Date, slotMinutes: number, offs: TimeOffInterval[],
): boolean => {
  const slotStart = slot.getHours() * 60 + slot.getMinutes();
  const slotEnd = slotStart + slotMinutes;
  return offs.some((o) =>
    !o.allDay && dayKey(o.startDate) === dayKey(slot) && o.startTime && o.endTime &&
    slotStart < toMinutes(o.endTime) && slotEnd > toMinutes(o.startTime));
};
```
Refactor minimal et **non-régressif** du générateur (4ᵉ paramètre optionnel) :

```typescript
  date: Date,
  appointments: AppointmentInterval[],
```

### Flux AC 4 — avertissement & annulation (synchrone, praticien dans la boucle)
Reformulation de la tâche initiale « job d'avertissement » : **pas de job asynchrone**. Le flux est **synchrone et explicite** :
1. À la soumission du formulaire, `previewTimeOffImpact(input)` renvoie les `Appointment` non-`CANCELLED` impactés (allDay ⇒ tous les RDV des jours couverts ; partiel ⇒ ceux chevauchant la plage).
2. Si la liste est non vide, l'UI affiche `timeoff-impact-warning` (liste patient/horaire) + case **« Prévenir les patients (annule les RDV) »** décochée par défaut.
3. `createTimeOff(input, { notifyCancellations })` : crée l'exception ; si `notifyCancellations`, passe les RDV en `CANCELLED` (dans une transaction) puis envoie les emails d'annulation **en réutilisant** `sendCancellationEmail` (fire-and-forget, comme `updateAppointmentStatus`). Sinon, l'exception est créée **sans** toucher aux RDV (ils restent visibles dans l'agenda, simplement « hors disponibilités »).

Ce choix « in-the-loop » est volontaire : annuler des RDV patients est **irréversible et sensible** ⇒ jamais silencieux (AC 7).

### Improvements Checklist

- [ ] **REL-002 (low)** — Dans `createTimeOff`, le `prisma.appointment.findMany` qui calcule les RDV impactés s'exécute *hors* de la transaction (lignes 306-314), puis la création + l'annulation `updateMany` sont dans `$transaction`.
