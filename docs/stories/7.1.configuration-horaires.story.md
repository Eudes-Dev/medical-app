# Story 7.1: Configuration des horaires d'ouverture

## Status
Ready for Review

## Story
**As a** praticien,
**I want** définir mes horaires d'ouverture par jour de la semaine (une ou plusieurs plages, avec une durée de créneau par plage),
**so that** les patients ne peuvent réserver, via le tunnel public `/book`, que sur mes plages réellement travaillées.

## Contexte métier
Aujourd'hui les créneaux proposés dans le tunnel public sont générés à partir d'une **constante figée** (`CABINET_INFO.openingHours = { start: 8, end: 18, slotMinutes: 30 }` dans `lib/cabinet/config.ts`). Le praticien ne peut donc ni adapter ses horaires (mi-temps, pause déjeuner, jour de fermeture), ni changer la granularité des rendez-vous. Cette story remplace cette constante par un **modèle persistant `WorkingHours`** éditable depuis le dashboard, et refactore le générateur de créneaux pour qu'il lise la base. C'est la première brique de l'épopée 7 (Paramétrage cabinet) et un **prérequis de la story 7.2** (congés/jours fériés, qui viendra superposer des exceptions par-dessus ces horaires) et de la 8.1.

## Acceptance Criteria

1. **Page de configuration** — Une page `/dashboard/settings/schedule` (espace praticien authentifié) permet de définir, pour chacun des 7 jours de la semaine, **zéro, une ou plusieurs plages horaires** (heure de début / heure de fin au format 24h).
2. **Persistance & application immédiate** — Les modifications sont persistées en base (`WorkingHours`) et appliquées **immédiatement** aux créneaux proposés dans le tunnel public `/book` (la Server Action `getAvailableSlots` lit la base à chaque appel ; pas de cache à invalider côté tunnel). La page de réglages est revalidée après sauvegarde (`revalidatePath`).
3. **Durée de créneau configurable par plage** — Chaque plage porte une durée de créneau choisie parmi **15 / 30 / 45 / 60 min** (`slotDuration`, défaut 30). C'est cette durée qui détermine le pas de génération des créneaux de la plage.
4. **Désactivation sans suppression** — Une plage peut être **désactivée** (`active = false`) sans être supprimée : elle reste visible/éditable dans l'UI mais ne génère aucun créneau dans le tunnel.
5. **Validation anti-chevauchement** — Les plages **d'un même jour ne peuvent pas se chevaucher** (validation Zod côté serveur **et** feedback inline côté client). De plus : `endTime` doit être strictement postérieure à `startTime`, et l'amplitude de la plage doit contenir au moins un créneau (`endTime − startTime ≥ slotDuration`).
6. **Permissions** — Seul le praticien authentifié peut lire/modifier les horaires. Toutes les Server Actions appellent `requireUser()` (défense en profondeur derrière le middleware). Modèle **single-tenant** : un seul jeu d'horaires pour l'unique cabinet (pas de `userId`/`tenantId` sur le modèle, cohérent avec l'architecture).
7. **États UI** — La page gère explicitement les états : **loading** (skeleton au chargement initial + bouton de sauvegarde en `isPending`/disabled), **empty** (jour sans plage → affichage « Fermé » + bouton « Ajouter une plage »), **success** (toast `sonner` de confirmation), **error** (toast d'erreur + conservation des saisies). Le bouton « Enregistrer » est désactivé tant qu'aucune modification n'est en attente ou qu'une validation échoue.
8. **Cas limites** — Un jour entièrement vide (aucune plage active) ⇒ le tunnel ne propose **aucun** créneau ce jour-là (jour fermé), sans erreur. Ajout / suppression de plages dynamiques. Saisie d'heures invalides rejetée avec message clair. Concurrence : la dernière sauvegarde fait foi (remplacement complet du jeu d'horaires, cf. Dev Notes « Stratégie d'écriture »).
9. **Responsive & accessibilité** — Interface utilisable en mobile (≥ 375px) et desktop : zones tactiles ≥ 44px, layout en liste verticale de 7 jours (empilée en mobile). A11y WCAG AA : navigation clavier complète, `<label>` associés à chaque champ, `aria-invalid` + message d'erreur lié via `aria-describedby`, focus visible, `Switch` avec label accessible, ordre de tabulation logique.
10. **Non-régression du tunnel** — Après refactor, le tunnel `/book` (stories 4.1/4.2) continue de fonctionner : génération correcte des créneaux, exclusion des `Appointment` non-`CANCELLED`, calcul de `endTime` à la réservation invité aligné sur la `slotDuration` de la plage du créneau choisi. Une **migration + un seed de valeurs par défaut** (Lun→Ven 08:00–18:00, créneaux 30 min ; Sam/Dim fermés) garantissent qu'aucun environnement ne se retrouve sans horaires après déploiement.

### Modèle de données — `WorkingHours`
```prisma
/// Horaires d'ouverture hebdomadaires du cabinet (story 7.1).
/// Single-tenant : un seul jeu d'horaires. Plusieurs lignes par jour = plusieurs plages.
model WorkingHours {
  id           String   @id @default(uuid())
  /// Jour de la semaine : 0=Dimanche … 6=Samedi (convention JS Date.getDay()).
  dayOfWeek    Int      @map("day_of_week")
  /// Heure de début "HH:mm" (24h).
  startTime    String   @map("start_time")
  /// Heure de fin "HH:mm" (24h, exclusive).
  endTime      String   @map("end_time")
  /// Durée d'un créneau en minutes : 15 | 30 | 45 | 60.
  slotDuration Int      @default(30) @map("slot_duration")
  /// Plage active (false = désactivée sans suppression).
  active       Boolean  @default(true)
  createdAt    DateTime @default(now()) @map("created_at")
  updatedAt    DateTime @updatedAt @map("updated_at")

  @@index([dayOfWeek])
  @@map("working_hours")
}
```
**Choix de conception** :
- `dayOfWeek` aligné sur `Date.getDay()` (0=Dimanche) pour que le générateur fasse `date.getDay()` sans conversion (l'UI réordonne Lun→Dim à l'affichage via une table de libellés).
- `startTime`/`endTime` en `String` "HH:mm" (pas de `DateTime`) car ce sont des **heures du jour**, indépendantes d'une date — évite toute ambiguïté de fuseau au stockage. La conversion en `Date` (TZ) se fait au moment de la génération, voir « Fuseau horaire » ci-dessous.
- `slotDuration` **par plage** : l'AC 3 (« durée par défaut configurable ») est satisfaite en pré-remplissant 30 à la création d'une plage, tout en autorisant une granularité différente par plage.

### Algorithme — génération des créneaux (module pur)
```typescript
// lib/cabinet/working-hours.ts
export interface WorkingHourRange { startTime: string; endTime: string; slotDuration: number }
export const toMinutes = (hhmm: string): number => {
  const [h, m] = hhmm.split(":").map(Number);
  return h * 60 + m;
};
export const fromMinutes = (min: number): string =>
  `${String(Math.floor(min / 60)).padStart(2, "0")}:${String(min % 60).padStart(2, "0")}`;
export const rangesOverlap = (a: WorkingHourRange, b: WorkingHourRange): boolean =>
  toMinutes(a.startTime) < toMinutes(b.endTime) && toMinutes(b.startTime) < toMinutes(a.endTime);

// lib/cabinet/slots.ts (refactor)
export function generateDaySlots(
  date: Date,
  ranges: WorkingHourRange[],
): { start: Date; slotMinutes: number }[] {
  const slots: { start: Date; slotMinutes: number }[] = [];
  for (const r of ranges) {
    const startM = toMinutes(r.startTime);
    const endM = toMinutes(r.endTime);
    for (let m = startM; m + r.slotDuration <= endM; m += r.slotDuration) {
      const start = new Date(date);
      start.setHours(Math.floor(m / 60), m % 60, 0, 0); // cf. note Fuseau horaire
      slots.push({ start, slotMinutes: r.slotDuration });
    }
  }
  return slots.sort((a, b) => a.start.getTime() - b.start.getTime());
}

export function filterAvailableSlots(
  date: Date,
  appointments: AppointmentInterval[],
  ranges: WorkingHourRange[],
): { start: Date; slotMinutes: number }[] {
  return generateDaySlots(date, ranges).filter(
    (s) => !appointments.some((apt) => isOverlapping(s.start, s.slotMinutes, apt)),
  );
}
```
La condition `m + r.slotDuration <= endM` garantit qu'aucun créneau ne dépasse la fin de plage (ex. une plage 08:00–18:00/30 produit exactement 20 créneaux, le dernier 17:30→18:00). `isOverlapping` est inchangé : `slot < apt.endTime && slotEnd > apt.startTime`, en excluant les RDV `CANCELLED` côté requête Prisma.

### Stratégie d'écriture (saveWorkingHours)
Pour la simplicité et la cohérence (un seul cabinet, faible fréquence d'édition), `saveWorkingHours` effectue un **remplacement complet** dans une transaction :
```typescript
await prisma.$transaction([
  prisma.workingHours.deleteMany({}),
  prisma.workingHours.createMany({ data: rowsFromValidatedInput }),
]);
```
Cela évite toute logique de diff/merge et règle naturellement la concurrence (« dernière sauvegarde gagne »). Les plages désactivées (`active=false`) sont **conservées** en base (insérées telles quelles) afin de respecter l'AC 4 (désactiver sans supprimer).

### Fuseau horaire (carry-over QA 4.1 / 4.2 — REL-001)
`setHours` s'appuie sur le **fuseau local du serveur**. Sur Vercel (UTC), un créneau « 10:00 » serait interprété en UTC. Le projet documente déjà ce risque (gates 4.1/4.2) et expose `CABINET_TIMEZONE = "Europe/Paris"` dans `lib/cabinet/config.ts`. **Décision de périmètre** : cette story **ne change pas** la stratégie TZ existante (conserver `setHours` pour ne pas régresser ni diverger) mais hérite du même TODO de correction globale (forcer `Europe/Paris` via `date-fns-tz`) à traiter avant mise en production, de façon transverse aux stories 4.1/4.2/7.1. Ne pas introduire une gestion TZ locale isolée ici.
[Source: docs/qa/gates/4.1-page-reservation-publique.yml, docs/stories/4.2.flux-reservation-invite.story.md — QA Results REL-001]

### Validation (stratégie Zod) — client + serveur
Double validation conforme à la convention projet : feedback inline côté client (état du `schedule-editor`) **et** revalidation serveur dans `saveWorkingHours` avant toute écriture.
```typescript
// lib/validations/working-hours.ts
const HHMM = /^([01]\d|2[0-3]):[0-5]\d$/;
export const timeRangeSchema = z
  .object({
    startTime: z.string().regex(HHMM, "Heure de début invalide (HH:mm)"),
    endTime: z.string().regex(HHMM, "Heure de fin invalide (HH:mm)"),
    slotDuration: z.union([z.literal(15), z.literal(30), z.literal(45), z.literal(60)]),
    active: z.boolean(),
  })
  .refine((r) => toMinutes(r.endTime) > toMinutes(r.startTime), {
    message: "L'heure de fin doit être après l'heure de début.",
    path: ["endTime"],
  })
  .refine((r) => toMinutes(r.endTime) - toMinutes(r.startTime) >= r.slotDuration, {
    message: "La plage est trop courte pour contenir un créneau.",
    path: ["endTime"],
  });
export const dayScheduleSchema = z
  .object({ dayOfWeek: z.number().int().min(0).max(6), ranges: z.array(timeRangeSchema) })
  .refine(
    (d) => d.ranges.every((a, i) => d.ranges.every((b, j) => i >= j || !rangesOverlap(a, b))),
    { message: "Les plages d'un même jour ne doivent pas se chevaucher.", path: ["ranges"] },
  );
export const weekScheduleSchema = z.array(dayScheduleSchema);
export type WeekScheduleValues = z.infer<typeof weekScheduleSchema>;
```
