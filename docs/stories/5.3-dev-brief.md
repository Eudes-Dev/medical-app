## 0. Pourquoi cette story existe
Le tunnel public (épopées 4/6/7) **ne peut pas être mis en production** en l'état : créneaux décalés (TZ), aucune protection anti-abus, lien d'annulation en 404, migrations non appliquées, baseline de tests rouge. Ces points étaient « tracés » vers des stories supprimées (13.5, 8.1) → désormais regroupés ici. Détail : [`docs/dette/2026-05-30-bloquants-pre-prod.md`](../dette/2026-05-30-bloquants-pre-prod.md).

## 1. Ordre d'attaque recommandé
Les 5 chantiers sont indépendants. Ordre conseillé (du plus risqué/structurant au plus simple) :

1. **Task 5 — baseline tests verte d'abord** (sinon impossible de détecter les régressions des autres tasks).
2. **Task 1 — TZ `Europe/Paris`** (touche le cœur de la génération, à valider avec toute la suite).
3. **Task 2 — rate-limiting** (isolé, faible risque de régression).
4. **Task 3 — route d'annulation** (nouvelle surface, réutilise l'existant 6.1).
5. **Task 4 — procédure migrations** (doc, déjà ébauchée — à compléter/vérifier).
6. **Task 6 — exécution E2E + consignation** (en dernier, une fois tout vert).

## 2. Chantier par chantier — points d'ancrage réels

### Task 1 — Fuseau horaire (REL-001) ⚠️ le plus sensible
- **Coupable :** `lib/cabinet/slots.ts` → `generateDaySlots` utilise `setHours()` (TZ du process). Sur Vercel UTC, « 10:00 » = 12:00 Paris (été).
- **Déjà dispo :** `CABINET_TIMEZONE = "Europe/Paris"` dans `lib/cabinet/config.ts` (ajouté en 6.2, **inutilisé**). Bon exemple à copier : `lib/booking/format.ts` (4.2) formate déjà en Paris via `Intl`.
- **À faire :** `npm i date-fns-tz` ; remplacer `setHours` par `fromZonedTime(<date+"HH:mm">, CABINET_TIMEZONE)`. Auditer aussi `startOfDay`/`endOfDay` dans `app/(public)/[cabinet-slug]/book/actions.ts` et `lib/cabinet/time-off.ts` (`dayKey`).
- **Preuve attendue :** un test avec `process.env.TZ="UTC"` qui prouve qu'un créneau « 10:00 » Paris reste correct, **en CET et en CEST** (tester une date d'hiver ET une date d'été).
- ⚠️ Ne pas casser la comparaison calendaire de `time-off.ts` (déjà robuste si `date` et bornes désignent le même jour local).

### Task 2 — Rate-limiting (SEC-001)
- **Cibles :** `getAvailableSlots` (lecture, ~30/min/IP) et `createGuestBooking` (création, ~5/10 min/IP) dans `app/(public)/[cabinet-slug]/book/actions.ts`.
- **Implémentation MVP :** limiteur **en mémoire** (Map + timestamps) dans `lib/server/rate-limit.ts` — pas de Redis obligatoire (documenter la limite single-instance + point d'extension Upstash).
- **Câblage UI :** ajouter le code `RATE_LIMITED` dans `lib/ui/error-mapper.ts` + `lib/ui/toast-messages.ts` (suivre le pattern de `BAD_REQUEST` ajouté en 5.2), mapper dans `BookingCalendar.tsx` / `GuestForm.tsx`.
- **Important :** retirer le `TODO(story 13.5)` du fichier `actions.ts` (la dette est close ici, pas reportée).

### Task 3 — Route d'annulation (CANCEL-ROUTE-001)
- **Vérifié :** `app/(public)/[cabinet-slug]/book/cancel` **n'existe pas** → créer `cancel/page.tsx`.
- **Réutiliser l'existant :** `Appointment.cancellationToken` (UUID `@unique`, posé en 6.1) ; `sendCancellationEmail(...)` (6.1, fire-and-forget) ; pattern `revalidatePath("/dashboard/calendar")`.
- **Action :** `cancelByToken(token)` — `findUnique({ where: { cancellationToken } })`, passage `CANCELLED` **idempotent** (déjà annulé = succès), email fire-and-forget, **rate-limité** (Task 2). Publique mais protégée par le token + message neutre si token inconnu (pas d'énumération).
- **Boucler :** vérifier que les liens de `lib/email/templates/ConfirmationEmail.tsx` et `lib/sms/send-confirmation-sms.ts` résolvent (test sur l'URL générée == route existante).

### Task 4 — Migrations (OPS-001)
- **Doc déjà ébauchée :** [`docs/dette/2026-05-30-procedure-migration-prod.md`](../dette/2026-05-30-procedure-migration-prod.md) (9 migrations listées, procédure, vigilances). La compléter/valider.
- **À faire (dev, pas ops) :** `npx prisma migrate status` → consigner ; confirmer que les migrations posées via `db push` sont bien matérialisées en fichiers SQL. **NE PAS** lancer `migrate deploy` sur une base live (action ops au déploiement).

### Task 5 — Baseline tests (TEST-BASELINE-001)
- `tests/unit/hooks/use-debounce.test.tsx` ×4 (fake timers vitest 4 / jsdom).
- `tests/component/dashboard/dashboard-stats.test.tsx` ×1 (mock Prisma / clé API).
- `tests/integration/calendar/appointment-actions.test.ts` `3.3-INT-008/009` → **régression réelle** : l'assertion n'attend pas l'`include: { patient }` ajouté en 6.x. **Corriger l'assertion, pas la prod.**
- **Objectif :** `npx vitest run` → 0 échec, ou `skip` documentés dans `docs/dette/2026-05-16-tests-residuels-suite-vitest.md`.

### Task 6 — E2E
- Lancer `npm run test:e2e` en local (prérequis : `.env.test`, Postgres `medical_app_test`, `npx playwright install chromium` — voir `tests/e2e/README.md`).

## 3. Contraintes & garde-fous
- **App médicale** : ne jamais perdre/modifier de RDV existant lors des refactors. `cancelByToken` doit être idempotent.
- **Pas de fuite** : toutes les erreurs publiques restent neutres (mapping centralisé `error-mapper.ts`).
- **Conventions projet** : `requireUser()` **ne s'applique pas** aux actions publiques (protection = token + rate-limiter) ; double validation Zod ; catch générique + `console.error` serveur.
- **UI** : Shadcn uniquement (voir [ADR](../architecture/4-ui-design-system-decision.md)) — pas de nouveau composant animate-ui.
