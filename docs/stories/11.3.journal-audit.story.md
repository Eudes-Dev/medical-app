# Story 11.3: Journal d'audit RGPD (piste d'audit append-only)

## Status

Done

## Story

**As a** praticien (responsable de traitement),
**I want** qu'une **piste d'audit immuable** consigne automatiquement **qui** a fait **quoi** et **quand** sur les données sensibles du patient — au minimum les **exports** (art. 20), les **effacements / droit à l'oubli** (art. 17) et les **changements de consentement** (11.1) — et que je puisse la consulter en lecture seule,
**so that** je dispose d'une **preuve de traçabilité** des accès et traitements sur les données de santé, troisième brique de conformité qui — avec 11.1 (consentement) et 11.2 (droits du patient) — fait progresser la levée de la réserve de mise en production du dossier patient (épopées 9.x).

> Troisième story de l'épopée 11 « RGPD / Sécurité » (post-MVP). Elle livre le
> **journal d'audit append-only** explicitement reporté par 11.1 et 11.2
> (« Le journal d'audit immuable… relève de l'épopée 11.3 »). Le patron de
> référence reste les stories 9.x / 11.1 / 11.2 : Server Actions protégées
> (`requireUser` + `assertValidUuid`), logique **pure et testable** isolée d'un
> côté, couche d'accès **fine et mockable** de l'autre, messages centralisés,
> tests unit + intégration avec Prisma/Supabase **mockés** (aucune base requise).
>
> **Décision de conception structurante** : le journal d'audit **doit survivre à
> l'effacement du patient** (art. 17). Une entrée « effacement du patient X »
> qui disparaîtrait avec le patient n'aurait aucune valeur probante. Le modèle
> `AuditLog` stocke donc `patientId` **dénormalisé** (`String?`, **aucune FK,
> aucune cascade**) et un **snapshot du libellé patient** (`patientLabel`) figé
> au moment de l'action, afin de rester lisible après suppression. Le journal est
> **append-only** : seules les opérations *create* et *read* existent — aucune
> Server Action de mise à jour ni de suppression d'entrée n'est exposée.
>
> **Politique best-effort** : l'écriture d'audit est **non bloquante** pour
> l'action métier qu'elle trace. Si l'enregistrement de l'audit échoue, l'action
> principale (export, effacement, consentement) **réussit quand même** et l'échec
> est journalisé en console — même philosophie que la purge Storage best-effort
> de 11.2 (ne jamais rendre une opération impossible à cause d'une dépendance
> annexe). Le durcissement (audit transactionnel / inviolabilité cryptographique)
> reste hors périmètre.

## Acceptance Criteria

1. Un nouvel enum Prisma `AuditAction { PATIENT_EXPORT, PATIENT_ERASURE, CONSENT_GRANTED, CONSENT_REVOKED, CONSENT_RESET }` et un nouveau modèle **append-only** `AuditLog` sont ajoutés : `id` (`String @id @default(uuid())`), `action` (`AuditAction`), `patientId` (`String? @map("patient_id")` — **dénormalisé, AUCUNE relation/FK/cascade**, afin de survivre à l'effacement du patient), `patientLabel` (`String? @map("patient_label")` — snapshot du nom au moment de l'action), `actorId` (`String @map("actor_id")` — `user.id` issu de `requireUser()`), `actorEmail` (`String? @map("actor_email")`), `summary` (`String?` — détail FR lisible), `createdAt` (`DateTime @default(now()) @map("created_at")`). Index `@@index([patientId])` et `@@index([createdAt])`, `@@map("audit_logs")`, conventions snake_case identiques aux modèles existants. **Aucune relation inverse n'est ajoutée sur `Patient`** (le lien est volontairement dénormalisé). Une migration SQL **non destructive** (aucun DROP/ALTER de table/colonne existante) accompagne le changement de schéma.

2. Un module **pur** `lib/rgpd/audit.ts` (aucun import Prisma/Supabase, testable en unit) expose : le type `AuditActionType` (union dérivée de l'enum), `AUDIT_ACTION_LABELS` (`Record<AuditActionType, string>`, libellés FR courts : Export des données / Effacement définitif / Consentement accordé / Consentement retiré / Consentement réinitialisé), `AUDIT_LABELS` (libellés UI : titre de la page, en-têtes de colonnes, état vide), et un helper `formatPatientLabel({ firstName, lastName })` produisant un libellé sûr et compact (trim, `"Patient"` en repli si vide). Le module reste **déterministe** et sans effet de bord.

3. Une couche d'accès **fine et mockable** `lib/server/audit.ts` (`"use server"` **non requis** — module serveur importé par les actions) expose `recordAuditEvent(input: AuditEventInput): Promise<void>` qui insère une entrée via `prisma.auditLog.create`. `AuditEventInput` = `{ action; actorId; actorEmail?; patientId?; patientLabel?; summary? }`. La fonction est **best-effort et ne lève jamais** : toute erreur Prisma est capturée et journalisée (`console.error`) sans se propager, afin de **ne jamais faire échouer** l'action métier tracée. Aucune logique métier dans cette couche (juste le `create`) — elle est conçue pour être mockée en test exactement comme `@/lib/storage/medical-documents`.

4. Le module `app/dashboard/patients/data-rights-actions.ts` (11.2) est instrumenté **sans changer ses signatures ni sa sémantique de retour** : après un `exportPatientData` réussi, une entrée `PATIENT_EXPORT` est enregistrée (`patientId`, `patientLabel` dérivé du patient chargé, `actorId`/`actorEmail` de `requireUser()`, `summary` FR) ; pour `erasePatientData`, le **nom du patient est capturé AVANT la suppression** (lecture `firstName`/`lastName`) puis, après le `prisma.patient.delete` réussi, une entrée `PATIENT_ERASURE` est enregistrée (avec `patientLabel` snapshot et `summary` mentionnant le nombre de documents purgés). Les appels à `recordAuditEvent` sont **best-effort** (ne bloquent jamais le retour de succès de l'action). L'`actorId` est obtenu en conservant la valeur de retour de `requireUser()`.

5. Le module `app/dashboard/patients/consent-actions.ts` (11.1) est instrumenté de la même façon : après un `setConsentStatus` réussi, une entrée `CONSENT_GRANTED` (si `granted === true`) ou `CONSENT_REVOKED` (si `granted === false`) est enregistrée (`patientId`, `summary` incluant la finalité concernée via `CONSENT_TYPE_LABELS`) ; après un `deleteConsentRecord` réussi, une entrée `CONSENT_RESET` est enregistrée (`patientId` dérivé de la ligne supprimée). Best-effort, signatures inchangées.

6. Une Server Action `getAuditLogEntries(options?: { patientId?: string; limit?: number })` dans un nouveau module `app/dashboard/audit/actions.ts` (`"use server"`, **pas** une route), protégée par `requireUser()`, retourne `AuditLogEntryData[]` triées par `createdAt` décroissant (les plus récentes d'abord), limitées (`limit` borné, défaut 200, max 500). Si `patientId` est fourni, il est validé via `assertValidUuid()` **avant** tout appel Prisma et filtre les entrées de ce patient ; un `patientId` invalide → `[]` (même politique que `getConsentRecords`/`getMedicalHistoryEntries`). `AuditLogEntryData` expose `id`, `action`, `patientId`, `patientLabel`, `actorEmail`, `summary`, `createdAt`. **Aucune** action d'écriture/suppression d'entrée n'est exposée (append-only).

7. Une page **lecture seule** `/dashboard/audit` (`app/dashboard/audit/page.tsx`, server component) affiche le journal d'audit dans la coquille dashboard standard (`SidebarProvider` + `AppSidebar` + `SidebarInset` + `Breadcrumb`, identique à la fiche patient) : un titre clair, un texte d'introduction (finalité : preuve de traçabilité des accès/traitements RGPD), et la liste des entrées (`getAuditLogEntries()`), chacune montrant l'**action** (libellé `AUDIT_ACTION_LABELS` + pastille de couleur cohérente avec le langage visuel : effacement « danger », export/consentement neutres), le **patient** (`patientLabel` — affiché tel quel même si le patient a depuis été supprimé), l'**acteur** (`actorEmail`), le **détail** (`summary`) et la **date/heure** formatée FR. Un état vide explicite (« Aucun événement d'audit pour le moment. ») est rendu si la liste est vide. Une entrée de navigation « Journal d'audit » est ajoutée à la barre latérale (`components/app-sidebar.tsx`) de façon cohérente avec les entrées existantes.

8. La **réserve RGPD de mise en production** (9.x) est mise à jour dans la story et le `ROADMAP.md` : 11.3 livre la **piste d'audit append-only** (export / effacement / consentement), mais ne lève pas seule la réserve — le **chiffrement applicatif (11.4)** reste non livré, et l'audit best-effort/non transactionnel est une limite assumée (durcissement = story ultérieure). Les notes de réserve cohérentes 9.x/11.1/11.2 sont actualisées en ce sens.

## Dev Notes

### Périmètre & décisions structurantes

- **Append-only** : le modèle `AuditLog` n'est jamais modifié ni supprimé par l'application. Seules les opérations `create` (via `recordAuditEvent`) et `read` (via `getAuditLogEntries`) existent. Aucune Server Action d'édition/suppression d'entrée n'est exposée.
- **Survie à l'effacement (art. 17)** : `patientId` est **dénormalisé** (`String?`, **aucune FK ni cascade**) et `patientLabel` est un **snapshot** du nom figé au moment de l'action. Ainsi, l'entrée « effacement du patient X » **persiste** après la suppression du patient — c'est précisément l'objet d'un journal d'effacement. C'est la **différence clé** avec `ConsentRecord`/`MedicalDocument`/etc. qui, eux, partent en cascade FK.
  [Source: app/dashboard/patients/data-rights-actions.ts (erasePatientData → prisma.patient.delete cascade), prisma/schema.prisma (cascades existantes)]
- **Best-effort** : `recordAuditEvent` ne lève jamais. Un échec d'audit ne doit pas empêcher un export ou rendre un patient « non effaçable ». Même politique que la purge Storage best-effort de 11.2 et `deleteMedicalDocument`. La trace inviolable/transactionnelle est un durcissement futur (hors périmètre).
  [Source: app/dashboard/patients/data-rights-actions.ts (removeObjects best-effort)]

### Identité de l'acteur

- `requireUser()` (`@/lib/server/auth`) retourne `{ id: string; email?: string }`. Les actions instrumentées **conservent** cette valeur (`const user = await requireUser()`) pour renseigner `actorId`/`actorEmail`. Single-tenant : un seul praticien attendu, mais l'`actorId` est tout de même tracé (défense en profondeur + futur multi-praticien 18.x).
  [Source: lib/server/auth.ts (AuthUser)]

### Sécurité (héritée 5.2 / 5.3, identique à 9.x / 11.1 / 11.2)

- **Toutes** les Server Actions commencent par `await requireUser()`.
- Tout `id` reçu est validé via `assertValidUuid()` **avant** tout appel Prisma (ici : le `patientId` optionnel de `getAuditLogEntries`).
- Prisma paramétré ; aucune surface publique (page et action réservées au dashboard authentifié). Aucune donnée sensible de contenu n'est dupliquée dans l'audit : `summary` reste un libellé d'action (pas le contenu d'une note/document).
  [Source: app/dashboard/patients/consent-actions.ts, lib/validations/uuid.ts, lib/server/auth.ts]

### Forme du modèle (append-only, dénormalisé)

```prisma
enum AuditAction {
  PATIENT_EXPORT
  PATIENT_ERASURE
  CONSENT_GRANTED
  CONSENT_REVOKED
  CONSENT_RESET
}

/// Journal d'audit RGPD append-only (story 11.3). Lien patient DÉNORMALISÉ
/// (pas de FK/cascade) → l'entrée survit à l'effacement du patient (art. 17).
model AuditLog {
  id           String      @id @default(uuid())
  action       AuditAction
  patientId    String?     @map("patient_id")     // dénormalisé : aucune FK
  patientLabel String?     @map("patient_label")  // snapshot du nom
  actorId      String      @map("actor_id")
  actorEmail   String?     @map("actor_email")
  summary      String?
  createdAt    DateTime    @default(now()) @map("created_at")

  @@index([patientId])
  @@index([createdAt])
  @@map("audit_logs")
}
```

> **Aucun** changement des modèles existants. Migration `CREATE TYPE` + `CREATE TABLE` + 2 `CREATE INDEX`, **sans** `ADD CONSTRAINT FOREIGN KEY`.
