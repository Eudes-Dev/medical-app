# Story 6.2: Rappels automatiques de rendez-vous

## Status

Done

## Story

**As a** praticien,
**I want** que mes patients reçoivent un rappel automatique avant leur RDV,
**so that** je réduis le taux de no-show.

## Acceptance Criteria

1. Un rappel email est envoyé J-1 à 18h00 (heure Paris, `Europe/Paris`) pour tout RDV `CONFIRMED` du lendemain — le cron tourne à 17h00 UTC (= 18h00 Paris en heure d'hiver, heure standard CET).
2. Un rappel "H-2" est envoyé 2h avant le RDV — le cron horaire sélectionne les RDV dont `startTime` est dans la fenêtre [now+1h50, now+2h10] (fenêtre glissante de ±10 min).
3. Le système est idempotent : `reminderD1SentAt IS NULL` / `reminderH2SentAt IS NULL` garantit qu'un même RDV ne reçoit jamais deux rappels identiques, même si le cron est rejoué.
4. Le patient peut désactiver les rappels via un lien dans l'email pointant vers `/unsubscribe?token=<optOutToken>` — token UUID opaque stocké sur le modèle `Patient` (`optOutToken`). La page publique positionne `patient.reminderOptOut = true`.
5. Les rappels ne sont pas envoyés pour les RDV `CANCELLED` ou `COMPLETED` (seuls les RDV `CONFIRMED` sont sélectionnés par la requête).

## Dev Notes

### Contexte et prérequis

- **Prérequis obligatoire** : Story 6.1 doit être Done. Cette story réutilise `lib/email/client.ts`, `sendEmail`, les templates `format.ts`, et le modèle `EmailLog`.
- **Patterns à respecter** : identiques à 6.1 — fire-and-forget, `sendEmail` ne throw jamais, EmailLog tracé.

### Décisions d'architecture

| Décision | Choix retenu | Raison |
|---|---|---|
| `reminderOptOut` sur Appointment ou Patient | **Patient** | L'opt-out est par patient, pas par RDV. Un patient qui se désinscrit ne doit plus recevoir de rappels pour aucun de ses futurs RDV. |
| `optOutToken` | UUID opaque sur `Patient` (même pattern que `cancellationToken` de 6.1) | Consistant avec le pattern établi. Pas de JWT — non décodable côté client. |
| Timezone "heure cabinet" | `Europe/Paris` hardcodé dans `lib/cabinet/config.ts` (nouvelle constante `CABINET_TIMEZONE`) | Aucune config timezone n'existe aujourd'hui. Pour le MVP single-tenant, hardcoder est acceptable. |
| Fenêtre H-2 | `[now + 1h50, now + 2h10]` (window de 20 min centrée sur 2h avant) | Absorbe les dérives de planification du cron (±quelques minutes). Idempotence garantie par `reminderH2SentAt IS NULL`. |
| Fenêtre D-1 | Guard `if (heure UTC !== 17)` + requête sur `startTime` demain Paris | Limite l'exécution de la logique D-1 à une seule plage horaire par jour. |

### Algorithme fenêtre D-1 (cron 17h UTC)

```typescript
import { CABINET_TIMEZONE } from "@/lib/cabinet/config";

// Guard : exécuter la logique D-1 uniquement si heure UTC ∈ [17, 18[
const nowUtc = new Date();
if (nowUtc.getUTCHours() !== 17) {
  // skip D-1 this run
}

// "demain" en heure Paris
const tomorrowStart = new Date(nowUtc);
tomorrowStart.setUTCHours(0, 0, 0, 0);
// Calculer minuit → fin demain en Paris via Intl ou simple +24h/+48h UTC
// Pattern simple (MVP) : startTime entre [now+6h, now+30h] capture "demain Paris"
// depuis 17h UTC : +7h = minuit Paris, +31h = minuit+1 Paris
const windowStart = new Date(nowUtc.getTime() + 7 * 3600_000);   // 00:00 Paris
const windowEnd   = new Date(nowUtc.getTime() + 31 * 3600_000);  // 00:00 Paris+1j

const appointments = await prisma.appointment.findMany({
  where: {
    status: "CONFIRMED",
    startTime: { gte: windowStart, lt: windowEnd },
    reminderD1SentAt: null,
    patient: { reminderOptOut: false },
  },
  include: { patient: true },
});
```

> Note : ce calcul est correct pour UTC+1 (CET hiver). En été (CEST, UTC+2), le cron à 17h UTC = 19h Paris (pas 18h). Pour le MVP, c'est acceptable. Une évolution future pourrait utiliser `date-fns-tz` pour un calcul timezone-aware rigoureux.

### Algorithme fenêtre H-2

```typescript
const now = new Date();
const windowStart = new Date(now.getTime() + (2 * 60 - 10) * 60_000); // now + 1h50
const windowEnd   = new Date(now.getTime() + (2 * 60 + 10) * 60_000); // now + 2h10

const appointments = await prisma.appointment.findMany({
  where: {
    status: "CONFIRMED",
    startTime: { gte: windowStart, lte: windowEnd },
    reminderH2SentAt: null,
    patient: { reminderOptOut: false },
  },
  include: { patient: true },
});
```

### Authentification de la route cron

```typescript
// app/api/cron/reminders/route.ts
export async function GET(request: Request) {
  const authHeader = request.headers.get("authorization");
  if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
    return Response.json({ error: "Unauthorized" }, { status: 401 });
  }
  // ... logique cron
}
```

Vercel Cron injecte automatiquement `Authorization: Bearer <CRON_SECRET>` à chaque appel planifié. En local, passer le header manuellement pour tester.

### Pattern fire-and-forget après l'envoi (idempotence)

```typescript
// Pour chaque appointment sélectionné :
void sendReminderD1({ appointment, patient }).catch((err) =>
  console.error("[cron:reminder-d1]", err)
);
// Mettre à jour APRÈS lancement (accepté — en cas d'erreur d'envoi,
// EmailLog tracera FAILED et le prochain cron retentera)
await prisma.appointment.update({
  where: { id: appointment.id },
  data: { reminderD1SentAt: new Date() },
});
```

> Choix MVP : on marque `sentAt` même si l'email échoue, pour éviter les boucles infernales. Le `EmailLog` (status FAILED) permet d'auditer les échecs.
