## Story
**As a** praticien,
**I want to** me connecter de manière sécurisée à mon espace de gestion,
**so that** seul moi puisse accéder aux données de mes patients et à mon agenda.

## Acceptance Criteria
1. Supabase Auth est configuré et intégré au projet avec le package `@supabase/ssr`
2. Une page de login (`/login`) est créée avec un formulaire email/mot de passe
3. Le formulaire de login valide les entrées avec Zod et react-hook-form
4. La connexion réussie redirige vers `/dashboard`
5. Un middleware Next.js protège toutes les routes `/dashboard/**`
6. Les utilisateurs non authentifiés sont redirigés vers `/login`
7. Un bouton de déconnexion est fonctionnel dans le layout dashboard

## Dev Notes

### Architecture de l'Information
- **Espace Privé (Dashboard)**: Toutes les routes sous `/dashboard/*` sont protégées
  - `/dashboard`: Vue d'ensemble (statistiques du jour, prochains RDV)
  - `/dashboard/calendar`: Agenda interactif (vue jour/semaine/mois)
  - `/dashboard/patients`: Gestion de la base patient
  - `/dashboard/settings`: Configuration du cabinet
- **Page Login**: `/login` (hors espace privé, accessible publiquement)
[Source: docs/front-end-spec/1-architecture-de-linformation-sitemap.md]

### Package @supabase/ssr (Important)
Le package `@supabase/auth-helpers-nextjs` est **déprécié** et remplacé par `@supabase/ssr` qui est framework-agnostic et conçu pour le SSR avec gestion des cookies.

Structure recommandée des clients Supabase:
```typescript
// lib/supabase/client.ts - Client côté navigateur
import { createBrowserClient } from '@supabase/ssr'

export function createClient() {
  return createBrowserClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  )
}

// lib/supabase/server.ts - Server Components & Server Actions
import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'
// Utilise les cookies Next.js pour la gestion de session
```

### Relation Supabase Auth <-> User Prisma
- **Supabase Auth** gère l'authentification (auth.users) avec UUID comme identifiant
- **User Prisma** stocke le profil praticien dans la table `users`
- La synchronisation se fait via l'UUID: `User.id` = `auth.users.id`
- Note: La création du profil User Prisma lors de l'inscription n'est PAS dans le scope de cette story (login seulement)
[Source: docs/stories/1.2.couche-donnees-prisma-supabase.story.md - Task 3]

### Stratégie de Validation (Zod)
- **Côté Client**: Feedback immédiat sous les champs de saisie via react-hook-form
- **Côté Serveur**: Double vérification dans les Server Actions avant appel Supabase Auth
[Source: docs/front-end-spec/4-stratgie-de-validation-zod.md]

### Design System
- Utiliser les composants Shadcn UI installés: Card, Input, Button (story 1.1)
- Installer le composant Label: `npx shadcn@latest add label`
- Couleur principale pour les boutons: Blue-600 (#2563eb)
- Couleur d'erreur: Rose-500
[Source: docs/front-end-spec/2-design-system-ui-components.md]

### Variables d'environnement requises
Variables à ajouter dans `.env.local` (en plus de DATABASE_URL et DIRECT_URL existants):
- `NEXT_PUBLIC_SUPABASE_URL`: URL du projet Supabase (ex: https://xxxxx.supabase.co)
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`: Clé anonyme Supabase (Dashboard > Settings > API)

### Sécurité Middleware
- Toujours utiliser `getUser()` au lieu de `getSession()` dans le middleware
- `getSession()` lit uniquement le cookie sans validation serveur
- `getUser()` valide le token auprès de Supabase Auth (plus sécurisé)
[Source: Documentation Supabase SSR]

### Testing
- **Tests manuels requis**:
  - Tester la connexion avec des identifiants valides → redirection `/dashboard`
  - Tester le rejet avec des identifiants invalides → message d'erreur affiché
  - Tester la validation Zod → erreurs sous les champs (email format, password length)
  - Vérifier la protection des routes `/dashboard/*` → redirection `/login` si non connecté
  - Vérifier la déconnexion → redirection `/login` et session supprimée
  - Tester l'accès direct à `/dashboard` sans session → redirection `/login`
- **Note**: Les tests automatisés seront définis dans le Sprint 5 (story 5.2)
