# Story 8.1: Reprogrammation patient self-service

## Status
Done

## Story
**As a** patient (invité) ayant un rendez-vous à venir réservé via le tunnel public,
**I want** déplacer moi-même mon rendez-vous vers un autre créneau depuis le lien reçu (sans téléphoner au cabinet),
**so that** je peux ajuster mon rendez-vous en autonomie et le cabinet garde un agenda à jour sans charge téléphonique.

## Acceptance Criteria

### A. Route publique de reprogrammation (tokenée)
1. Une route `app/(public)/[cabinet-slug]/book/reschedule/page.tsx` existe et lit le `?token=` (réutilise le `cancellationToken` opaque existant comme **jeton de gestion** du RDV). Comportement strictement aligné sur `book/cancel/page.tsx` : token absent / inconnu / RDV `CANCELLED` ou `COMPLETED` ⇒ **écran d'erreur neutre** (pas d'énumération de l'existence d'un RDV) ; GET **throttlé** (30/min/IP, pattern SEC-003 de 5.3) ; `metadata.robots = { index: false, follow: false }` (`noindex`).
2. Token valide et RDV reprogrammable ⇒ écran affichant **le récapitulatif du RDV actuel** (date formatée en heure de **Paris** via `formatSlotParis`, type/intitulé) **+ un sélecteur de nouveau créneau** (rail de dates + créneaux) alimenté par la logique de créneaux disponibles existante.

### B. Politique de délai minimum (fenêtre de modification)
3. La reprogrammation n'est possible que si le RDV débute **à plus de `RESCHEDULE_MIN_NOTICE_HOURS` heures** (constante = **24**). En-deçà du délai ⇒ message dédié neutre « Ce rendez-vous est trop proche pour être modifié en ligne. Merci de contacter le cabinet. ». La règle est appliquée **côté route (affichage)** ET **côté Server Action (défense en profondeur)**. La comparaison se fait sur l'**instant absolu** (`startTime` vs `now`), donc insensible au fuseau (pas de calcul d'heure murale).

### C. Server Action `rescheduleByToken`
4. Une Server Action `rescheduleByToken(token, newSlotISO)` **publique** (non authentifiée : protection = token opaque + rate-limiter, **pas** de `requireUser()`). Elle valide `newSlotISO` (Zod : ISO 8601 avec offset, strictement futur, ≤ `MAX_BOOKING_HORIZON_DAYS` = 90 j — réutiliser les bornes du tunnel).
5. La logique serveur, dans l'ordre :
   a. rate-limit par IP (clé `reschedule:${ip}`, **10 / 10 min / IP**) **avant toute requête DB** ;
   b. `findUnique({ where: { cancellationToken } })` → token inconnu ⇒ `INVALID` (message neutre) ;
   c. RDV `CANCELLED` ou `COMPLETED` ⇒ `INVALID` ;
   d. délai minimum (AC 3) non respecté sur le RDV **courant** ⇒ `TOO_LATE` ;
   e. **anti-collision** sur le nouveau créneau en **excluant l'`id` du RDV courant** (mêmes règles que `createGuestBooking` : chevauchement d'un RDV `status != CANCELLED`) ⇒ `SLOT_TAKEN` ;
   f. **défense `TimeOff`** (story 7.2) : nouveau créneau dans un jour/plage bloqué ⇒ `SLOT_TAKEN` ;
   g. la **durée est conservée** : `endTime = newStart + (endTime − startTime)` du RDV existant (ne pas recalculer depuis `WorkingHours`/service — on déplace, on ne re-paramètre pas).
6. Sur succès : `prisma.appointment.update` de `startTime`/`endTime` uniquement (conserve `patientId`, `type`, `serviceTypeId`, `status`, `cancellationToken`). Le résultat est typé : `{ success: true }` ou `{ error: "INVALID" | "TOO_LATE" | "SLOT_TAKEN" | "RATE_LIMITED" | "SERVER" }`. **Aucune fuite de PII** dans les erreurs.

### D. Notifications & synchronisation agenda
7. Un **email de reprogrammation** est envoyé au patient en *fire-and-forget* (si `patient.email` présent), avec la **nouvelle** date/heure (Paris) et les liens de gestion (annuler / reprogrammer à nouveau, réutilisant le même token). Nouveau type de message `RESCHEDULED` (enum `MessageType`), template `RescheduleEmail.tsx` et helper `send-reschedule.ts` (calqués sur `CancellationEmail`/`send-cancellation`).
8. `revalidatePath("/dashboard/calendar")` est appelé après l'update pour que l'agenda praticien reflète le déplacement.

### E. Points d'entrée (enrichir `book/cancel` + messages transactionnels)
9. L'écran `components/public/CancelConfirmation.tsx` (route `book/cancel`) propose **en plus** une action « Reprogrammer plutôt » qui redirige vers `/{cabinetSlug}/book/reschedule?token=…` (le bouton n'est proposé que si le RDV est encore reprogrammable — sinon non affiché).
10. L'email de confirmation (`ConfirmationEmail.tsx`, 6.1) et le SMS de confirmation (`lib/sms/send-confirmation-sms.ts`, 6.3) ajoutent un lien **« Modifier / reprogrammer »** vers `…/book/reschedule?token=…` **en plus** du lien d'annulation existant (le SMS reste ≤ 1 segment dans la mesure du possible ; si la place manque, privilégier un seul lien « gérer mon RDV » — voir Dev Notes).

### F. Migration, qualité & tests
11. Migration Prisma **idempotente** pour ajouter la valeur d'enum : `ALTER TYPE "MessageType" ADD VALUE IF NOT EXISTS 'RESCHEDULED';` (même précaution transactionnelle que OPS-003 de 5.3 — valeur non consommée dans la migration ; cible Supabase = PG15, OK). **Aucune autre modification de schéma** (on réutilise `cancellationToken`, pas de nouveau champ).
12. Couverture de tests :
    - **Unit** : politique de délai (`> 24h` OK, `< 24h` `TOO_LATE`, bord exact) ; validation `newSlotISO`.
    - **Intégration** `tests/integration/public-booking/reschedule.test.ts` : token vide/inconnu ⇒ `INVALID` ; RDV `CANCELLED`/`COMPLETED` ⇒ `INVALID` ; trop proche ⇒ `TOO_LATE` ; collision ⇒ `SLOT_TAKEN` (et **non**-collision avec soi-même) ; `TimeOff` ⇒ `SLOT_TAKEN` ; succès ⇒ `update` (durée conservée) + email *fire-and-forget* + `revalidatePath` ; rate-limit ⇒ `RATE_LIMITED` ; résolution des liens email/SMS (URL générée == route existante).
    - `npx vitest run` revient à une **baseline verte** (0 échec), conformément à TEST-BASELINE-001 (5.3).

## Dev Notes

### Modèle de données concerné — `Appointment`
[Source: prisma/schema.prisma#Appointment]
- Champs clés : `id`, `patientId`, `startTime`, `endTime` (`DateTime` UTC), `status` (`AppointmentStatus`: `PENDING|CONFIRMED|CANCELLED|COMPLETED`), `type` (libellé-instantané), `serviceTypeId?`, `cancellationToken?` (`@unique`, UUID opaque — `null` pour les RDV dashboard), relations `patient`, `serviceType`, `messageLogs`.
- **Réutiliser `cancellationToken` comme jeton de gestion** : ne PAS créer de nouveau champ. Les RDV éligibles à la reprogrammation patient sont exactement ceux qui ont un token (créés via le tunnel public, qui ont reçu un lien). Les RDV dashboard (`cancellationToken = null`) sont gérés par le praticien (story 3.3) et n'ont pas de lien public.
- **Durée conservée** : `endTime − startTime` (en ms) du RDV existant. Lors du déplacement, `endTime = newStart + cette durée`. Ne PAS recalculer depuis `WorkingHours`/`ServiceType`.
- `MessageType` enum : ajouter `RESCHEDULED` (existants : `CONFIRMATION|CANCELLATION|PRACTITIONER_NOTIFY|REMINDER_D1|REMINDER_H2`). [Source: prisma/schema.prisma#MessageType]

### Pattern de référence n°1 — annulation publique (story 5.3) — À CLONER
[Source: app/(public)/[cabinet-slug]/book/cancel/actions.ts]
```
export type CancelByTokenResult = { success: true } | { error: "INVALID" | "RATE_LIMITED" | "SERVER" };
const ip = await getClientIp();
if (!checkRateLimit(`cancel:${ip}`, 10, 10*60_000).ok) return { error: "RATE_LIMITED" };
const appt = await prisma.appointment.findUnique({ where: { cancellationToken: token }, select: {...} });
if (!appt) return { error: "INVALID" };           // token inconnu = neutre, pas d'énumération
if (appt.status === "CANCELLED") return { success: true };  // idempotent
await prisma.appointment.update(...); void sendCancellationEmail(...).catch(...); revalidatePath("/dashboard/calendar");
```
- `getClientIp()` / `checkRateLimit()` : `@/lib/server/rate-limit` (story 5.3, SEC-001/SEC-002). `getClientIp` est anti-spoof (priorité `x-real-ip`).
- Le `try/catch` retourne `{ error: "SERVER" }` sans exposer le détail (`console.error` seulement).

[Source: app/(public)/[cabinet-slug]/book/cancel/page.tsx]
- Pattern de page : `params`/`searchParams` sont des `Promise` (Next 16) → `await`. `token` peut être `string | string[]` → normaliser `Array.isArray(t) ? t[0] : t`.
- Throttle GET : `checkRateLimit(\`reschedule-page:${ip}\`, 30, 60_000)` avant le `findUnique` non authentifié (défense anti-énumération SEC-003).
- Écrans neutres : « Trop de tentatives », « Lien invalide ou expiré », (nouveau) « Trop proche pour modifier en ligne ».

### Pattern de référence n°2 — anti-collision & TimeOff & TZ (createGuestBooking)
[Source: app/(public)/[cabinet-slug]/book/actions.ts]
- Helpers TZ Paris : `@/lib/cabinet/timezone` → `zonedDayBoundsUtc(date)` (bornes UTC du jour de Paris), `zonedDayOfWeek`, `zonedMinutes`. Indispensables pour requêter « les RDV du jour » indépendamment du fuseau serveur (REL-001 clos en 5.3 ; le runner CI/Vercel est en **UTC**).
- Anti-collision : relire les RDV du jour (`startTime ∈ [startUtc, endUtc)`, `status != CANCELLED`) puis `isOverlapping(slot, durationMin, apt)` de `@/lib/cabinet/slots`. **Pour la reprogrammation, exclure le RDV courant** (filtrer par `id` ou ignorer l'apt dont l'id == courant).
- Défense `TimeOff` (story 7.2) : `isDayFullyBlocked(slot, timeOffs)` / `slotInPartialTimeOff(slot, durationMin, timeOffs)` de `@/lib/cabinet/time-off` ; requête `timeOff` active intersectant `[startUtc, endUtc)`.
- Génération des créneaux dispo : **réutiliser `getAvailableSlots`** (déjà rate-limité, TZ-correct, `WorkingHours` + `TimeOff` + anti-collision). Le sélecteur de reprogrammation appelle cette action existante — ne pas réécrire la génération. (Note : `getAvailableSlots` exclut déjà les créneaux pris ; le créneau actuel du patient apparaîtra comme « pris » par lui-même, ce qui est acceptable — il choisit un *autre* créneau. Mentionner dans les tests, pas un blocant MVP.)
