# Story 13.2: Déploiement de production Vercel (configuration reproductible)

## Status
Done

## Story
**As a** praticien-développeur (mainteneur solo du dépôt),
**I want** une configuration de déploiement Vercel reproductible, documentée et sûre — build qui régénère le client Prisma, région de calcul figée en zone UE (RGPD), variables d'environnement de production inventoriées et validables, runbook de mise en ligne —,
**so that** je puisse déployer l'application en production (et en preview) de façon déterministe, sans build cassé ni dérive de configuration, tout en gardant les actions irréversibles (application des migrations en base prod, exécution réelle du `vercel deploy`) sous contrôle humain explicite.

> Deuxième story de l'épopée 13 « DevOps / Sécu infra » (post-MVP). Elle prolonge la 13.1
> (CI) côté **CD** : la 13.1 livrait la validation (tests + build en CI) et excluait
> explicitement « le volet CD (déploiement Vercel) … story 13.2 ».
> [Source: docs/stories/13.1.cicd-github-actions.story.md — « Hors périmètre (CD / OPS) »]
>
> **Garde-fou de boucle.** Cette story ne **déclenche aucun déploiement réel** ni aucune
> action contre une base de production. Elle livre uniquement la **configuration, le code
> de validation testable et le runbook**. L'exécution effective (`vercel link`/`deploy`,
> pose des variables dans le dashboard Vercel, `prisma migrate deploy` en prod) reste une
> **action ops/humaine** (garde-fou OPS-002), documentée mais non exécutée par la boucle.

## Contexte & constat bloquant (pourquoi cette story)
Le dépôt est déjà largement « Vercel-ready » : `vercel.json` (cron rappels 6.2),
`next.config.ts` (headers de sécurité), `.env.example` exhaustif, route cron protégée par
`CRON_SECRET`, et la CI 13.1. **Mais** un build Vercel « propre » échouerait aujourd'hui :

- Le client Prisma est généré dans `lib/generated/prisma` (cf. `prisma/schema.prisma`,
  `generator client { output = "../lib/generated/prisma" }`), répertoire **gitignoré**
  (`.gitignore:50`) et **non suivi** (`git ls-files lib/generated/` → 0 fichier).
- Le script `build` vaut `next build` **seul**, et il n'existe **aucun** script `postinstall`.
- Sur Vercel, l'enchaînement est `npm install` → `next build`. Sans étape `prisma generate`,
  l'import `@/lib/generated/prisma/client` (utilisé par `lib/prisma.ts`) est **introuvable**
  → **build cassé**. La CI 13.1 ne révèle pas le problème car elle lance `npx prisma generate`
  comme **étape explicite** avant le build — ce que Vercel ne fait pas par défaut.

[Source: prisma/schema.prisma (generator), .gitignore:50, package.json#scripts, lib/prisma.ts, .github/workflows/ci.yml]

## Acceptance Criteria
1. **Build Vercel reproductible (Prisma).** Un script `postinstall` (`"postinstall": "prisma generate"`) est ajouté à `package.json`, garantissant la régénération du client Prisma à chaque `npm install`/`npm ci` (Vercel, CI, et toute installation fraîche locale). Le build par défaut Vercel (`next build`) dispose alors du client `lib/generated/prisma`. Aucun changement du `output` du generator, du schéma, ni des migrations.
2. **Région de calcul UE (RGPD).** `vercel.json` fige la région sur `cdg1` (Paris) via `"regions": ["cdg1"]`, cohérent avec la localisation des données du socle RGPD (épopée 11). Le bloc `crons` existant (rappels 6.2, `/api/cron/reminders`) est **préservé à l'identique**. Un champ `"framework": "nextjs"` rend le préréglage explicite.
3. **Inventaire validable des variables de production.** Un module **pur et testable** `lib/config/required-env.ts` déclare la liste des variables d'environnement **requises en production** (DB, Supabase, JWT, chiffrement, app URL, email, cron) et expose une fonction pure `validateProductionEnv(env)` retournant les variables manquantes (et celles présentes mais vides). La distinction prod-obligatoire vs optionnel (ex. SMS Twilio `SMS_ENABLED=false`, `SUPABASE_MEDICAL_DOCS_BUCKET` avec repli) est encodée et **documentée par variable**.
4. **Script de vérification pré-déploiement.** Un script `scripts/check-prod-env.ts` (exposé via `npm run check:prod-env`) appelle `validateProductionEnv(process.env)` et **sort en code 1** avec un rapport lisible si des variables prod manquent, `0` sinon. Le script **ne lit ni ne modifie aucune base** ; il n'exécute aucun appel réseau. Il est **non bloquant pour le build** (outil d'aide ops, pas une porte de build qui casserait les previews).
5. **Runbook de déploiement.** `docs/ops/13.2-deploiement-vercel.md` documente la procédure complète et reproductible : (a) `vercel link` du projet ; (b) configuration des variables d'environnement de production (référence `.env.example` + sortie de `npm run check:prod-env`) ; (c) génération des secrets (`openssl rand -base64 32` pour `JWT_SECRET`/`DATA_ENCRYPTION_KEY`/`CRON_SECRET`) ; (d) **application des migrations** `prisma migrate deploy` en prod présentée comme **étape ops/humaine** (garde-fou OPS-002, jamais automatisée) ; (e) pré-requis Storage/RLS (renvoi `docs/ops/9.2-…`) et clé de chiffrement (renvoi `docs/ops/11.4-…`) ; (f) checklist de smoke-test post-déploiement (page publique `/book`, login dashboard, cron). 
6. **Aucune action irréversible, aucun secret committé.** La story n'exécute **pas** `vercel deploy`, ne pose **pas** de variable dans le dashboard Vercel, n'applique **aucune** migration en base, et n'ajoute **aucun secret** au dépôt. Les seuls fichiers modifiés/créés sont du code de config/validation, des tests et de la doc. `.env*` reste gitignoré.
7. **Portes locales (preuve).** Le module `validateProductionEnv` est couvert par des tests unitaires `tests/unit/config/required-env.test.ts` (cas : toutes présentes → aucune manquante ; manquantes/vides détectées ; optionnelles absentes tolérées). La suite Vitest complète reste **verte** (baseline 800 tests + nouveaux tests de cette story), et `npm run build` est **validé localement** après l'ajout du `postinstall` (preuve réelle de reproductibilité de la porte dure de build).

## Dev Notes

### Nature de la story (config/CD, pas de logique métier)
- Pas de changement de `prisma/schema.prisma`, pas de migration, pas de nouvelle dépendance runtime. On ajoute : un `postinstall`, deux clés dans `vercel.json`, un module de validation **pur**, un script CLI Node (tsx, déjà présent en devDep), des tests, et un runbook.
[Source: package.json#scripts (tsx déjà présent), prisma/schema.prisma]

### Pourquoi `postinstall: prisma generate` (et pas la `buildCommand`)
- Vercel exécute `installCommand` puis `buildCommand`. Mettre la génération dans `postinstall` la rend **portable** (Vercel, CI, install local frais) sans coupler la commande de build à Prisma. Le client étant gitignoré (`lib/generated/prisma`), c'est le **seul** point qui garantit sa présence avant `next build`. La CI 13.1 continue de fonctionner (son `npx prisma generate` explicite devient redondant mais inoffensif).
[Source: .gitignore:50, lib/prisma.ts, .github/workflows/ci.yml]

### Région `cdg1` (RGPD)
- L'épopée 11 (RGPD) chiffre les données cliniques et formalise audit/effacement. Fixer la région de calcul Vercel sur **Paris (cdg1)** maintient le traitement applicatif en UE, cohérent avec l'hébergement Supabase EU. La donnée at-rest reste chez Supabase ; ce réglage concerne le **compute** des Functions/SSR.
[Source: docs/architecture/6-chiffrement-applicatif-decision.md, ROADMAP épopée 11]

### Validation d'environnement : pure et non bloquante
- `validateProductionEnv` est une **fonction pure** (`env: Record<string,string|undefined>` → rapport), donc unit-testable sans mock ni base, dans la lignée des modules purs `lib/analytics/stats.ts` / `stats-csv.ts`. Le script `check:prod-env` est un **aide-ops** : on ne l'inscrit **pas** comme porte de build (`next build`) pour ne pas casser les previews Vercel sans secrets ; il s'exécute à la demande / dans le runbook. Variables prod-obligatoires retenues : `DATABASE_URL`, `DIRECT_URL`, `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `JWT_SECRET`, `DATA_ENCRYPTION_KEY`, `NEXT_PUBLIC_APP_URL`, `RESEND_API_KEY`, `EMAIL_FROM`, `CRON_SECRET`. Optionnelles (non bloquantes) : `SUPABASE_MEDICAL_DOCS_BUCKET` (repli code), `NEXT_PUBLIC_CABINET_SLUG` (repli), `PRACTITIONER_NOTIFICATION_EMAIL`, et le bloc SMS Twilio (gouverné par `SMS_ENABLED`, `false` par défaut).
[Source: .env.example, lib/security/crypto.ts (DATA_ENCRYPTION_KEY), lib/booking/session-cookie.ts (JWT_SECRET), app/api/cron/reminders/route.ts (CRON_SECRET)]

### Hors périmètre (actions ops/humaines — OPS-002)
- **Pas** d'exécution de `vercel deploy`/`vercel link`, **pas** de pose de variables dans le dashboard Vercel, **pas** de `prisma migrate deploy` en base prod, **pas** de configuration de domaine. Ces étapes sont **décrites** dans le runbook mais **exécutées par un humain**. Monitoring Sentry (13.3), backups (13.4) et rate-limiting durci (13.5) restent hors périmètre.
[Source: docs/stories/5.3.consolidation-pre-prod.story.md — OPS-002 ; ROADMAP épopée 13]

Le `postinstall` et `vercel.json` ne sont pas unit-testables : leur validité se prouve par (a) `npm run build` local vert après ajout du `postinstall` (reproductibilité de la porte dure de build) et (b) inspection du `vercel.json` (artefact déclaratif). L'exécution réelle sur Vercel reste une action ops (hors boucle).
