# Story 4.2: Flux de Réservation "Invité"

## Status
Ready for Review

## Story
**As a** patient invité,
**I want** finaliser ma réservation en fournissant mes coordonnées minimales,
**so that** je puisse obtenir un rendez-vous rapidement sans créer de compte.

## Acceptance Criteria
1. Le formulaire invité, accessible à `/[cabinet-slug]/book/guest` (étape 2 du tunnel, atteignable depuis la story 4.1), contient les champs: Prénom, Nom, Téléphone, Email. La page est inaccessible (redirection vers `/[cabinet-slug]/book`) si aucun créneau n'est sélectionné dans `useBookingStore`.
2. La validation Zod (`guestBookingSchema`) vérifie tous les champs côté client (feedback inline sous chaque champ via `react-hook-form` + `zodResolver`) ET côté serveur (Server Action) avant toute écriture en base.
3. Si l'email fourni correspond à un patient existant en base (`prisma.patient.findFirst({ where: { email } })`), le rendez-vous est lié à ce patient (`patientId` réutilisé). **Note schéma** : `Patient.email` n'est pas `@unique` côté Prisma — utiliser `findFirst` (pas `findUnique`). En cas d'homonymes (plusieurs patients avec le même email), le plus ancien (`createdAt asc`) est utilisé pour le MVP single-praticien.
4. Si aucun patient n'existe avec cet email (ou si l'email correspond mais nom/prénom diffèrent significativement — voir Dev Notes), un nouveau patient est créé via `prisma.patient.create`.
5. Un écran de succès à `/[cabinet-slug]/book/success` affiche le récapitulatif: date et heure du RDV (formatées en `Europe/Paris`), nom du cabinet, instructions (arriver 10 min avant), et un bouton "Retour à l'accueil" (vers `/[cabinet-slug]`).
6. Un toast `sonner` de confirmation s'affiche après la création réussie du RDV (variante `success`), et un toast d'erreur (variante `error`) en cas de conflit de créneau (statut `409`) ou d'erreur serveur.
7. Un cookie HTTP-only `booking_token` (JWT signé HS256) contenant l'`appointmentId` est posé pour 24h. La page `/[cabinet-slug]/book/success` lit ce cookie côté serveur, vérifie sa signature, et affiche les détails du RDV. Aucun cookie ⇒ redirection vers `/[cabinet-slug]/book`.
8. Le rendez-vous est créé avec `status = PENDING` et apparaît dans l'agenda du praticien (déjà câblé via stories 3.2 / 3.3 qui n'excluent que les `CANCELLED`).
9. Une dernière vérification anti-collision est effectuée côté serveur juste avant la création : si le créneau a été pris entre l'étape 1 et la soumission, l'action renvoie `{ error: "SLOT_TAKEN" }` et le client invite l'utilisateur à choisir un autre créneau.
10. La Server Action est protégée par les mêmes mitigations que `getAvailableSlots` (story 4.1) : validation Zod stricte, TODO `rate limiter` (story 13.5), aucune fuite d'information PII dans les erreurs renvoyées au client.

### Architecture - Single-Tenant
L'application est conçue pour un déploiement unique. Toutes les données appartiennent au même cabinet — pas de `tenant_id` en base. La résolution du `[cabinet-slug]` est non-vérifiée pour le MVP (cf. 4.1).
[Source: docs/architecture/1-structure-globale.md, docs/architecture/2-logique-de-donnes.md]

### Modèle de Données Prisma — Patient & Appointment
```prisma
model Patient {
  id          String   @id @default(uuid())
  firstName   String   @map("first_name")
  lastName    String   @map("last_name")
  phone       String
  email       String?  // ⚠️ optionnel, NON unique (cf. schéma)
  dateOfBirth DateTime? @map("date_of_birth")
  notes       String?
  createdAt   DateTime @default(now())
  appointments Appointment[]
}

model Appointment {
  id        String            @id @default(uuid())
  patientId String            @map("patient_id")
  startTime DateTime          @map("start_time")
  endTime   DateTime          @map("end_time")
  status    AppointmentStatus @default(PENDING)  // PENDING|CONFIRMED|CANCELLED|COMPLETED
  type      String   // ⚠️ obligatoire — défaut "Première consultation" pour les invités
  @@index([startTime])
}
```
**Conséquence directe** : `findUnique({ where: { email } })` est IMPOSSIBLE (pas de contrainte `@unique` sur `email`). Utiliser `findFirst` avec `orderBy: { createdAt: "asc" }`.
[Source: prisma/schema.prisma]

async function findOrCreatePatient(data: GuestBookingValues): Promise<{ id: string }> {
  const existing = await prisma.patient.findFirst({

### Sécurité du cookie (illustratif)
```typescript
// lib/booking/session-cookie.ts
import { cookies } from "next/headers";
import { SignJWT, jwtVerify } from "jose";

const secret = new TextEncoder().encode(process.env.JWT_SECRET);

export async function setBookingCookie(appointmentId: string) {
  const token = await new SignJWT({ appointmentId })
    .setProtectedHeader({ alg: "HS256" })
    .setIssuedAt()
    .setExpirationTime("24h")
    .sign(secret);
  cookies().set("booking_token", token, {
    httpOnly: true,
    secure: process.env.NODE_ENV === "production",
    sameSite: "lax",
    maxAge: 60 * 60 * 24,
    path: "/",
  });
}

export async function readBookingCookie(): Promise<string | null> {
  const token = cookies().get("booking_token")?.value;
  if (!token) return null;
  try {
    const { payload } = await jwtVerify(token, secret);
    return typeof payload.appointmentId === "string" ? payload.appointmentId : null;
  } catch {
    return null;
  }
}
```
