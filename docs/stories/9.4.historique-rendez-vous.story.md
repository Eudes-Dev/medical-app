## Story
**As a** praticien,
**I want** consulter l'historique complet des rendez-vous d'un patient depuis sa fiche — sous forme de tableau filtrable (Tous / À venir / Terminés) avec lignes dépliables révélant le motif, la durée, la modalité, le lieu et la note clinique de chaque consultation,
**so that** je dispose d'une vue chronologique lisible et premium du parcours du patient, directement intégrée au dossier patient, sans quitter la fiche.

> Quatrième story de l'épopée 9 « Dossier patient » (post-MVP), après 9.1 (notes de
> consultation, **Done**), 9.2 (documents médicaux, **Done**) et 9.3 (antécédents
> médicaux, **Done**). Elle répond directement à **US.3** du PRD :
> « En tant que praticien, je veux voir ma liste de patients **et l'historique de
> leurs rendez-vous**. »
> [Source: docs/prd/3-user-stories-prioritaires.md#US.3]
>
> Cette story livre **deux choses** :
> 1. La section « Historique des rendez-vous » de la fiche patient (composant
>    `AppointmentHistory`), alimentée par les RDV déjà chargés côté serveur.
> 2. L'enrichissement du modèle `Appointment` avec trois champs descriptifs
>    optionnels (`motif`, `modalite`, `lieu`) — le champ « Note » réutilise la
>    colonne existante `Appointment.notes`.
>
> Elle s'accompagne d'une **refonte UI** des trois sections livrées en 9.1/9.2/9.3
> (notes, documents, antécédents) autour d'un kit visuel partagé
> (`components/patients/patient-record-ui.tsx`) — handoffs « Claude Design ».
> Cette refonte **ne change pas le contrat fonctionnel** (props, types, Server
> Actions, validation, sécurité) déjà couvert par les tests.

## Acceptance Criteria
1. Le modèle Prisma `Appointment` est étendu de trois colonnes **optionnelles** : `motif String?`, `modalite String?`, `lieu String?`. Le champ « Note » **réutilise la colonne existante** `Appointment.notes` (aucun nouveau champ de note). Une migration SQL **non destructive** (uniquement des `ADD COLUMN`, aucun DROP/ALTER de colonne existante) accompagne le changement : tout RDV existant reste inchangé (nouvelles colonnes `NULL`).
2. Le type `PatientAppointment` (exporté par `app/dashboard/patients/actions.ts`) expose les champs `motif: string | null`, `modalite: string | null`, `lieu: string | null`, `note: string | null` (ce dernier dérivé de `Appointment.notes`). Les deux Server Actions qui chargent la fiche — `getPatientById` et `updatePatient` — mappent ces quatre champs (valeur `?? null`) pour chaque RDV retourné. Les RDV restent triés `startTime desc` (plus récent → plus ancien), comme depuis la story 2.2.
3. Un composant client `components/patients/appointment-history.tsx` (`"use client"`) exporte `AppointmentHistory` + `AppointmentHistoryProps`. Il reçoit la liste des RDV **déjà chargée et déjà triée** par le serveur (`appointments: PatientAppointment[]`) — **aucune logique réseau métier dans le composant**. Props optionnelles : `practitionerName`, `density` (`"Confort" | "Compact"`), `showAvatars`.
4. La section affiche des **onglets de filtre** « Tous » / « À venir » / « Terminés » avec **compteurs** calculés sur l'ensemble des RDV (indépendamment du filtre courant) : « À venir » = statut `PENDING` ou `CONFIRMED` ; « Terminés » = statut `COMPLETED`. Changer de filtre réinitialise la pagination et replie les lignes ouvertes.
5. Chaque ligne est **dépliable** (clic sur la ligne ou sur le bouton « Voir les détails », avec `aria-expanded`) et révèle un panneau de détails : **Motif**, **Durée** (calculée depuis `startTime`/`endTime`, format « 30 min » / « 1 h 15 »), **Modalité**, **Lieu**, et la **Note** clinique (si présente). Les valeurs absentes affichent un tiret `—`.
6. Présentation premium conforme au langage visuel du dossier patient : badge de statut coloré par statut (avec pastille animée « pulse » pour les RDV actifs `PENDING`/`CONFIRMED`, désactivée si `prefers-reduced-motion`), icône + couleur dérivées du **libellé de type** de consultation, avatar praticien (initiales), thème sky/cyan via le kit partagé `patient-record-ui` (`ACCENT`, `ACCENT_GRADIENT`, `EASE`, `SECTION_SHADOW`). Animations via `motion/react` avec respect de `useReducedMotion`.
7. Pagination « **Charger plus** » : `PAGE_SIZE` (5) lignes initiales, `PAGE_STEP` (3) lignes supplémentaires par clic, avec lignes **squelettes** animées pendant le délai de chargement simulé. Un **bouton « Nouveau rendez-vous »** redirige vers `/dashboard/calendar`. Un **état vide** clair est affiché quand le filtre courant ne renvoie aucun RDV.
8. La section est montée dans la fiche patient via `components/patients/patient-detail.tsx` (`<AppointmentHistory appointments={currentPatient.appointments} />`), en remplacement de l'ancienne liste inline de RDV. La fiche patient (`/dashboard/patients/[id]`) reste rendue dynamiquement.
9. **Refonte UI (sans régression de contrat)** : un kit visuel partagé `components/patients/patient-record-ui.tsx` est introduit et les composants `consultation-notes.tsx`, `medical-documents.tsx`, `medical-history.tsx` sont refondus visuellement **sans modifier** leurs props, types, Server Actions, schémas de validation ni garanties de sécurité/RGPD. La suite de tests existante (notes 9.1, documents 9.2, antécédents 9.3) reste **intégralement verte**.
10. Le mapping serveur des nouveaux champs (`motif`/`modalite`/`lieu`/`note`) est couvert par des tests d'intégration (`9.4-INT-00x`) sur `getPatientById` et `updatePatient`, dans le harnais existant (`tests/integration/patients/actions.test.ts`, Prisma + Supabase + `next/cache` mockés). Aucune base requise.

## Dev Notes

### Distinction avec les autres surfaces du dossier patient
- `AppointmentHistory` (cette story) : **vue chronologique** des RDV du patient (lecture seule depuis la fiche), filtrable et dépliable. Ne crée/modifie pas de RDV (le bouton « Nouveau rendez-vous » renvoie vers l'agenda `/dashboard/calendar`).
- `ConsultationNote` (9.1) : historique daté d'entrées de **consultation** (texte clinique libre).
- `MedicalDocument` (9.2) : **pièces jointes** (fichiers).
- `MedicalHistoryEntry` (9.3) : **fond clinique structuré et catégorisé** (allergies, traitements, ATCD).
- Les quatre coexistent. Les nouveaux champs `motif`/`modalite`/`lieu` enrichissent le modèle **transactionnel** `Appointment`, pas le dossier clinique.
[Source: prisma/schema.prisma — model Appointment ; docs/stories/9.1/9.2/9.3]

### Logique de Données
- Les RDV sont chargés via les Server Actions existantes `getPatientById` / `updatePatient` (`include: { appointments: { orderBy: { startTime: "desc" } } }`), tri **plus récent → plus ancien**. Le composant ne refait aucun tri métier.
- Les trois nouveaux champs sont **optionnels** (`String?`) : rétro-compatibles avec tous les RDV historiques (valeur `null`). Single-tenant : pas de `userId`/`tenantId`.
- Pas de nouveau schéma Zod : ces champs ne sont pas saisis via formulaire dans cette story (alimentés par seed / futurs écrans agenda). La « Note » réutilise `Appointment.notes`.
[Source: app/dashboard/patients/actions.ts, prisma/schema.prisma]

### Sécurité (héritée 5.2 / 5.3, identique à 9.1 / 9.2 / 9.3)
- L'accès aux RDV passe par `getPatientById` / `updatePatient`, **déjà** protégées par `requireUser()` (story 2.2). Cette story n'ajoute **aucune nouvelle surface réseau** : `AppointmentHistory` est purement présentationnel et ne contient ni Server Action ni fetch.
- Aucune URL/fichier public ; aucune interpolation SQL brute (Prisma paramétré).
[Source: app/dashboard/patients/actions.ts]

### Migration (format attendu)
Même style que les migrations 9.1/9.2/9.3 : en-tête de commentaire (story + « migration non destructive »), uniquement des `ADD COLUMN … TEXT`.
