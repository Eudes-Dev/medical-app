# Story 6.1: Emails transactionnels de confirmation

## Status

Ready for Done

## Story

**As a** patient,
**I want** recevoir un email de confirmation immédiat après ma réservation,
**so that** j'ai la preuve de mon rendez-vous et les informations pratiques du cabinet.

## Acceptance Criteria

1. Un email de confirmation est envoyé au patient dès la création d'un rendez-vous via le tunnel invité (`createGuestBooking`), quel que soit le statut initial (PENDING).
2. L'email contient : nom du cabinet, adresse, téléphone, date/heure du RDV formatée en français, type de consultation, et un lien d'annulation sécurisé (URL signée contenant le `cancellationToken` UUID du RDV).
3. Le praticien reçoit une notification par email (destinataire : `PRACTITIONER_NOTIFICATION_EMAIL`) pour chaque nouveau RDV créé via le tunnel public.
4. Un email d'annulation est envoyé au patient lorsqu'un RDV passe en statut `CANCELLED` (via `updateAppointmentStatus` dans `app/dashboard/calendar/actions.ts`).
5. Les échecs d'envoi sont loggés (`console.error`) sans bloquer la création/mise à jour du RDV. Le pattern est `void sendEmail(...).catch(err => console.error("[email]", err))` — fire-and-forget depuis la Server Action. Aucune erreur d'envoi ne remonte vers le client.
6. Les templates React Email sont prévisualisables localement via `npx react-email dev` (port 3001). Ils utilisent des composants responsives (`@react-email/components`).

### cancellationToken — pattern

UUID v4 généré côté serveur via `crypto.randomUUID()` (natif Node.js ≥ 14.17, disponible dans App Router). Stocké dans `Appointment.cancellationToken` (champ unique nullable — `null` pour les RDV créés depuis le dashboard).

```typescript
const cancellationToken = crypto.randomUUID();
await prisma.appointment.create({
  data: { ..., cancellationToken },
});
```

**Pas de JWT** — le token est opaque, non décodable côté client. La route d'annulation publique (hors périmètre de cette story, voir story 8.1) vérifiera ce token en DB.

### Pattern fire-and-forget (AC 5)

Dans une Server Action Next.js, ne pas `await` les emails pour ne pas bloquer la réponse :

```typescript
// Après prisma.appointment.create(...)
void sendConfirmationEmail(params).catch((err) =>
  console.error('[email:confirmation] envoi échoué:', err)
);
```

`void` supprime le warning TypeScript "promise non awaited". L'erreur est absorbée — le RDV est créé quelle que soit la réussite de l'email.

### Modèle Prisma — EmailLog (à créer)

```prisma
enum EmailType {
  CONFIRMATION
  CANCELLATION
  PRACTITIONER_NOTIFY
}
enum EmailStatus {
  SENT
  FAILED
}
model EmailLog {
  id                String      @id @default(uuid())
  appointmentId     String?     @map("appointment_id")
  type              EmailType
  status            EmailStatus
  providerMessageId String?     @map("provider_message_id")
  error             String?
  createdAt         DateTime    @default(now()) @map("created_at")
  @@map("email_logs")
}
```
