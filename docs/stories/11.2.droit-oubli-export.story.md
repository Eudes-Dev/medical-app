# Story 11.2: Droit à l'oubli & export des données patient

## Status

Done

## Story

**As a** praticien (responsable de traitement),
**I want** pouvoir, depuis la fiche d'un patient, **exporter l'intégralité de ses données** dans un fichier structuré lisible par machine **et** **effacer définitivement** ce patient et toutes ses données (dossier, RDV, documents, consentements) en une action confirmée,
**so that** je peux répondre aux demandes RGPD de **portabilité (art. 20)** et de **droit à l'effacement / droit à l'oubli (art. 17)**, deuxième brique de conformité qui — avec 11.1 (consentement) — fait progresser la levée de la réserve de mise en production du dossier patient (épopées 9.x).

> Deuxième story de l'épopée 11 « RGPD / Sécurité » (post-MVP). Elle complète 11.1
> (consentement) par les **droits du patient sur ses données**. Le patron de
> référence reste les stories 9.x / 11.1 : Server Actions protégées
> (`requireUser` + `assertValidUuid`), logique **pure et testable** isolée dans un
> module sans Prisma/Supabase, validation/messages centralisés, tests unit +
> intégration avec Prisma/Supabase **mockés** (aucune base requise).] 
>
> **Aucun changement de schéma Prisma** : l'export agrège des modèles existants,
> l'effacement s'appuie sur les cascades FK déjà en place. Cette story comble en
> revanche une **lacune connue** de l'actuel `deletePatient` (actions.ts) : la
> cascade SQL supprime les lignes `MedicalDocument` mais **laisse orphelins les
> objets binaires** dans le bucket Supabase Storage (cf. schema.prisma L.617-619 :
> « ⚠️ La cascade SQL ne supprime PAS les objets Storage »). L'effacement RGPD de
> cette story **purge d'abord le Storage** puis supprime le patient.
>
> Le **journal d'audit immuable** des accès/effacements (qui a exporté/supprimé,
> quand) relève de l'épopée **11.3** (non livrée). Cette story livre les deux
> actions et leur preuve fonctionnelle, pas la piste d'audit append-only.

## Acceptance Criteria

1. Un module **pur** `lib/rgpd/patient-export.ts` (aucun import Prisma/Supabase, testable en unit) expose : `PATIENT_EXPORT_SCHEMA_VERSION` (ex. `"1.0"`), le type `PatientExportInput` (patient + ses relations brutes), le type `PatientExport` (structure normalisée sérialisable JSON), `buildPatientExport(input, options?)` qui assemble un objet d'export normalisé, et `buildPatientExportFileName(patient, now)` qui produit un nom de fichier sûr (ex. `donnees-patient-martin-jean-2026-06-26.json`, slug ASCII en minuscules, sans séparateur de chemin). `buildPatientExport` inclut un en-tête de métadonnées (`schemaVersion`, `exportedAt` ISO, `policyVersion` = `CONSENT_POLICY_VERSION` réutilisée de 11.1) et les sections : `patient` (identité + champs administratifs), `appointments`, `consultationNotes`, `medicalHistoryEntries`, `consentRecords`, et `medicalDocuments` **métadonnées uniquement** (`fileName`, `mimeType`, `sizeBytes`, `category`, dates — **jamais** `storagePath`, conformément à 9.2). Les `Date` sont sérialisées en ISO 8601. La fonction est **déterministe** (tri stable des collections, ex. par `createdAt`).
2. Une Server Action `exportPatientData(patientId)` dans un nouveau module `app/dashboard/patients/data-rights-actions.ts` (`"use server"`, **pas** une route), protégée par `requireUser()` et validant `assertValidUuid(patientId)` **avant** tout appel Prisma. Elle charge le patient et ses relations (`include` : `appointments`, `consultationNotes`, `medicalHistoryEntries`, `consentRecords`, `medicalDocuments`), appelle `buildPatientExport`, et retourne `{ success: true; fileName: string; json: string }` (JSON indenté via `JSON.stringify(export, null, 2)`) ou `{ success: false; error: string }`. Patient introuvable → `{ success: false, error }` (message FR). UUID invalide → `{ success: false, error }` (jamais de fuite d'`id`). **Aucune** donnée de Storage n'est téléchargée (export = métadonnées + données structurées).
3. Une Server Action `erasePatientData(patientId)` dans le même module, protégée par `requireUser()` + `assertValidUuid(patientId)` **avant** tout appel Prisma/Storage, qui réalise le **droit à l'oubli** : (a) récupère les `storagePath` des `MedicalDocument` du patient ; (b) **purge les objets Storage** correspondants (best-effort, via `removeObjects` — un échec Storage est journalisé mais **ne bloque pas** la suppression, pour éviter un patient « non supprimable » ; même politique que `deleteMedicalDocument`) ; (c) supprime le patient via `prisma.patient.delete` (les lignes liées — RDV, notes, documents, antécédents, consentements, entrées de liste d'attente — partent par **cascade FK** existante) ; (d) `revalidatePath('/dashboard/patients')`. Retour `{ success: true; erasedDocuments: number } | { success: false; error: string }`. Patient introuvable → erreur FR ; UUID invalide → erreur FR.
4. Le module Storage `lib/storage/medical-documents.ts` expose une fonction `removeObjects(paths: string[])` qui supprime un **lot** d'objets du bucket (`supabase.storage.from(bucket).remove(paths)`), no-op si la liste est vide. Elle reste **fine et mockable** (même style que `removeObject`). `removeObject` existant est conservé (réutilisé par `deleteMedicalDocument`).
5. La fiche patient (`/dashboard/patients/[id]`) affiche une section « Données personnelles & RGPD » (composant client `components/patients/data-rights-section.tsx`), montée **après** la section consentement, réutilisant le kit `patient-record-ui` (`SectionShell`, `DeleteConfirm`, etc.) pour rester cohérente avec 9.x/11.1. Elle contient : un texte d'introduction clair (finalité : portabilité art. 20 + effacement art. 17), un bouton **« Exporter les données (JSON) »** et un bouton **« Supprimer définitivement le patient »** (zone visuellement distincte / « danger »).
6. **Export côté UI** : le clic sur « Exporter les données » appelle `exportPatientData`, et en cas de succès déclenche le **téléchargement** du fichier JSON côté navigateur (Blob `application/json` + `URL.createObjectURL` + ancre `download` au nom `fileName`, révoquée après usage). Toast de succès. En cas d'échec : toast d'erreur (message renvoyé par l'action). Aucune dépendance serveur supplémentaire (génération du Blob côté client à partir de la chaîne `json`).
7. **Effacement côté UI** : le bouton « Supprimer définitivement » ouvre une **confirmation explicite** (réutilise `DeleteConfirm` du kit) rappelant le caractère **irréversible** et **total** (toutes les données du patient). À la confirmation, appel de `erasePatientData` ; en cas de succès : toast de succès puis **redirection** vers `/dashboard/patients` (`router.push` + `router.refresh()`). En cas d'échec : toast d'erreur, pas de redirection. Le bouton existant de suppression dans `PatientDetailClient` (`deletePatient`) **n'est pas modifié** par cette story (coexistence documentée en Dev Notes : l'effacement RGPD purge en plus le Storage).
8. Les messages utilisateur passent par les catalogues centralisés : une clé `dataRights` est ajoutée à `lib/ui/toast-messages.ts` (`exported` « Données exportées. », `erased` « Patient et données supprimés définitivement. », `exportFailed` « L'export des données a échoué. Veuillez réessayer. »). La **réserve RGPD de mise en production** (9.x) est mise à jour dans la story et le `ROADMAP.md` : 11.2 livre la portabilité + l'effacement, mais ne lève pas seule la réserve (journal d'audit 11.3 et chiffrement applicatif 11.4 restent non livrés).

## Dev Notes

### Périmètre & distinction avec les surfaces existantes

- **Export (art. 20)** : agrège les modèles existants en un JSON structuré. Lecture seule, aucune écriture, aucun téléchargement de binaire Storage (les documents sont exportés en **métadonnées** seules — patron 9.2 où `storagePath` n'est jamais exposé).
- **Effacement (art. 17)** : suppression **hard** complète. S'appuie sur les cascades FK déjà déclarées (`Appointment`, `ConsultationNote`, `MedicalDocument`, `MedicalHistoryEntry`, `ConsentRecord`, `WaitlistEntry` ont tous `onDelete: Cascade` côté patient). **Ajout clé** : purge des objets Supabase Storage des documents (lacune de `deletePatient`).
- **Coexistence avec `deletePatient` (actions.ts)** : non modifié par cette story. `deletePatient` reste la suppression « administrative » (cascade DB seule) déclenchée depuis `PatientDetailClient`. `erasePatientData` est la suppression **RGPD** (purge Storage + cascade DB). Un éventuel remplacement/fusion des deux chemins relève d'une story ultérieure (éviter d'élargir le périmètre / casser l'UI existante ici).
[Source: app/dashboard/patients/actions.ts (deletePatient), prisma/schema.prisma (cascades), app/dashboard/patients/medical-document-actions.ts (deleteMedicalDocument)]

### Sécurité (héritée 5.2 / 5.3, identique à 9.x / 11.1)

- **Toutes** les Server Actions commencent par `await requireUser()` (`@/lib/server/auth`).
- Tout `id` est validé via `assertValidUuid()` (`@/lib/validations/uuid`) **avant** tout appel Prisma/Storage.
- Aucune surface publique (section réservée au dashboard authentifié). Prisma paramétré ; aucune interpolation SQL brute. Le nom de fichier d'export est un **slug** dérivé (anti-traversal : pas de `/`, `\`, `..`).
[Source: app/dashboard/patients/medical-history-actions.ts, lib/validations/uuid.ts, lib/server/auth.ts]

### Effacement — ordre des opérations (best-effort Storage)

```
1. requireUser() + assertValidUuid(patientId)
2. docs = medicalDocument.findMany({ where:{patientId}, select:{storagePath:true} })
3. try { removeObjects(docs.map(d => d.storagePath)) } catch (e) { console.error(...) /* on continue */ }
4. prisma.patient.delete({ where:{id: patientId} })   // cascade FK → toutes les lignes liées
5. revalidatePath('/dashboard/patients')
```

> Le best-effort Storage évite un patient « non supprimable » si le bucket est
> indisponible (même politique que `deleteMedicalDocument`). La trace de l'échec
> partiel relève du journal d'audit 11.3.
> [Source: app/dashboard/patients/medical-document-actions.ts (deleteMedicalDocument, removeObject best-effort)]

### Forme de l'export (déterministe, sérialisable)

```jsonc
{
  "schemaVersion": "1.0",
  "exportedAt": "2026-06-26T10:00:00.000Z",
  "policyVersion": "2026-06",
  "patient": { "id": "...", "firstName": "...", /* ... */ },
  "appointments": [ /* triés par createdAt */ ],
  "consultationNotes": [ /* ... */ ],
  "medicalHistoryEntries": [ /* ... */ ],
  "consentRecords": [ /* ... */ ],
  "medicalDocuments": [ { "fileName": "...", "mimeType": "...", "sizeBytes": 0, "category": "...", "createdAt": "..." } ]
  // ⚠️ jamais de storagePath
}
```

Module **pur** (pas de Prisma/Supabase) ⇒ testable en unit sans base. La Server
Action se contente de charger les données (Prisma) et de sérialiser la sortie.
[Source: docs/architecture/5-stockage-fichiers-decision.md (métadonnées vs binaire), lib/validations/consent.ts (CONSENT_POLICY_VERSION)]
