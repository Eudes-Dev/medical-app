# Story 11.1: Consentement RGPD

## Status
Done

## Story
**As a** praticien,
**I want** consigner, depuis la fiche d'un patient, le consentement RGPD recueilli pour chaque finalité de traitement (données personnelles, données de santé, communications) — avec son statut (accordé / retiré / non renseigné), sa date et la version de politique de confidentialité concernée,
**so that** je dispose d'une **preuve traçable du consentement** par patient et par finalité, première brique de conformité RGPD qui conditionne la mise en production du dossier patient (épopées 9.x).

> Première story de l'épopée 11 « RGPD / Sécurité » (post-MVP). Elle livre le
> **socle de traçabilité du consentement** côté praticien. Le patron de référence
> est la story 9.3 (antécédents médicaux) : données **structurées et catégorisées**
> attachées au patient, **sans stockage de fichier** ni dépendance Supabase Storage,
> Server Actions protégées + validation Zod isomorphe + migration non destructive.
>
> Différence de modèle avec 9.x : ici on ne tient pas un **historique d'entrées
> libres**, mais un **état courant unique par finalité** (`@@unique([patientId, type])`),
> mis à jour par bascule accordé ↔ retiré horodatée. Le **journal d'audit immuable**
> des changements de consentement relève de l'épopée **11.3** (non livrée) ; cette
> story livre l'état courant + ses horodatages, pas la piste d'audit append-only.

## Acceptance Criteria
1. Un nouvel enum Prisma `ConsentType { DATA_PROCESSING, HEALTH_DATA, COMMUNICATION }` et un nouveau modèle `ConsentRecord` sont ajoutés : `id`, `patientId` (FK `Patient`, `onDelete: Cascade`), `type` (`ConsentType`), `granted` (`Boolean @default(false)`), `grantedAt` (`DateTime?`), `revokedAt` (`DateTime?`), `policyVersion` (`String`), `note` (`String?`), `createdAt`, `updatedAt`. Contrainte `@@unique([patientId, type])` (un état courant unique par finalité) + `@@index([patientId])` + relation inverse `consentRecords ConsentRecord[]` sur `Patient`. Conventions snake_case (`@map`/`@@map("consent_records")`) identiques aux modèles existants. Une migration SQL **non destructive** (aucun DROP/ALTER de table ou colonne existante) accompagne le changement de schéma.
2. La saisie est validée par un schéma Zod isomorphe (`consentInputSchema`) dans `lib/validations/consent.ts` : `type` ∈ valeurs de l'enum `ConsentType` ; `granted` booléen ; `note` optionnelle, trimée, ≤ 500 caractères (chaîne vide normalisée en `undefined`/absence de note). La validation échoue proprement (type hors enum / note > 500) côté client **et** serveur, **sans appel Prisma** côté serveur, avec un message FR lisible. Le module est **pur** (aucun import Prisma/Supabase) afin d'être testable en unit. Il expose aussi : `CONSENT_TYPES` (ordre d'affichage), `CONSENT_TYPE_LABELS` (libellés FR courts), `CONSENT_TYPE_DESCRIPTIONS` (finalité détaillée FR), `CONSENT_NOTE_MAX_LENGTH` (500) et `CONSENT_POLICY_VERSION` (version courante de la politique, ex. `"2026-06"`).
3. Trois Server Actions sont disponibles dans `app/dashboard/patients/consent-actions.ts` (`"use server"`, **pas** une route), **toutes** protégées par `requireUser()`, et tout `id` reçu est validé via `assertValidUuid()` **avant** tout appel Prisma :
   - `getConsentRecords(patientId)` → `ConsentRecordData[]` (états enregistrés du patient ; UUID invalide → `[]`, même politique que `getMedicalHistoryEntries`) ;
   - `setConsentStatus(patientId, input)` → `{ success: true; record } | { success: false; error }` : **upsert** sur `@@unique([patientId, type])`. Quand `granted === true` : `granted=true`, `grantedAt=now`, `revokedAt=null`, `policyVersion=CONSENT_POLICY_VERSION`. Quand `granted === false` : `granted=false`, `revokedAt=now` (le `grantedAt` du dernier accord est conservé). `note` mise à jour dans les deux cas ;
   - `deleteConsentRecord(recordId)` → `{ success: true } | { success: false; error }` : remet la finalité à l'état « non renseigné » (suppression de la ligne).
   Chaque mutation appelle `revalidatePath('/dashboard/patients/{patientId}')` (le `patientId` est dérivé de la ligne retournée par Prisma pour `setConsentStatus`/`deleteConsentRecord`).
4. La fiche patient (`/dashboard/patients/[id]`) affiche une section « Consentement RGPD » listant **les trois finalités** dans l'ordre `CONSENT_TYPES`, chacune avec : son libellé (`CONSENT_TYPE_LABELS`), sa description de finalité (`CONSENT_TYPE_DESCRIPTIONS`), et un **badge de statut** explicite : « Accordé le {date} » (vert), « Retiré le {date} » (rouge/neutre), ou « Non renseigné » (gris) pour une finalité sans `ConsentRecord`. Les dates sont formatées en FR. Un état d'introduction clair indique la finalité de la section (preuve de consentement, non un acte juridique complet).
5. Pour chaque finalité, le praticien peut **enregistrer le consentement** (« Marquer comme accordé ») et **le retirer** (« Marquer comme retiré »), avec une **note** optionnelle (contexte : ex. « consentement papier signé le 12/06 »). À la réussite : toast de succès, statut et date mis à jour dans la liste (état local optimiste + `router.refresh()`). Le praticien peut aussi **réinitialiser** une finalité à « non renseigné » (action de remise à zéro avec confirmation inline, réutilisant `DeleteConfirm` du kit UI).
6. Les libellés et messages utilisateur passent par les catalogues centralisés : une clé `consent` est ajoutée à `lib/ui/toast-messages.ts` (`granted` « Consentement enregistré. », `revoked` « Consentement retiré. », `reset` « Consentement réinitialisé. »). Les libellés/finalités FR vivent dans `lib/validations/consent.ts` (réutilisables côté UI et tests).
7. La suppression d'un patient purge ses consentements (cascade FK). La version de politique consentie est **horodatée** (`policyVersion` figée au moment de l'accord) afin de tracer **à quoi** le patient a consenti. Aucun champ existant de `Patient` (notamment `notes`, `reminderOptOut`, `optOutToken`) n'est modifié, réutilisé ni supprimé : le consentement « COMMUNICATION » coexiste avec le mécanisme d'opt-out des rappels (6.2) sans le remplacer (documenté en Dev Notes).
8. La **réserve RGPD de mise en production** des épopées 9.x est mise à jour : 11.1 livre la **brique consentement** mais ne lève pas seule la réserve (le chiffrement applicatif 11.4, le journal d'audit 11.3 et le droit à l'oubli/export 11.2 restent non livrés). La story et le `ROADMAP.md` reflètent cet avancement partiel.

## Dev Notes

### Distinction avec les autres surfaces du dossier patient
- `Patient.notes` (`String?`) : note administrative libre — laissée intacte.
- `Patient.reminderOptOut` / `optOutToken` (6.2) : mécanisme d'**opt-out** des rappels pour les patients (lien de désinscription). Le consentement `COMMUNICATION` de 11.1 le **complète** comme preuve de recueil, **sans** le remplacer ni s'y substituer côté logique d'envoi des rappels (aucune modification de la logique 6.2 dans cette story).
- `ConsentRecord` (cette story) : **état courant du consentement par finalité** (RGPD), horodaté et versionné. Pas un historique libre (cf. 9.1/9.3), pas un fichier (cf. 9.2).
[Source: prisma/schema.prisma — model Patient ; docs/stories/6.2.rappels-automatiques.story.md]

### Logique de Données
- Relations : Patient (1) <---> (N) ConsentRecord (`onDelete: Cascade` côté ConsentRecord), avec `@@unique([patientId, type])` ⇒ au plus un état courant par finalité.
- Accès : Server Actions uniquement (pas d'API route). Single-tenant : pas de `userId`/`tenantId` (cohérent avec tous les modèles existants).
- Pas de FK vers `Appointment` : un consentement n'est pas rattaché à un RDV.
[Source: docs/architecture/2-logique-de-donnes.md, prisma/schema.prisma]

### Sécurité (héritée 5.2 / 5.3, identique à 9.1 / 9.2 / 9.3)
- **Toutes** les Server Actions commencent par `await requireUser()` (`@/lib/server/auth`).
- Tout `id` est validé via `assertValidUuid()` (`@/lib/validations/uuid`) **avant** tout appel Prisma.
- Inputs validés par Zod isomorphe ; Prisma paramétré (aucune interpolation SQL brute). Aucune surface publique (section réservée au dashboard authentifié).
[Source: app/dashboard/patients/medical-history-actions.ts, lib/validations/uuid.ts, lib/server/auth.ts]

### Convention Server Actions (forme de retour)
Mêmes signatures de retour que 9.3 :
```typescript
| { success: true; record: ConsentRecordData }
| { success: false; error: string }
```
`getConsentRecords` renvoie directement `ConsentRecordData[]` (ou lève `UnauthorizedError`), UUID invalide → `[]`.
[Source: app/dashboard/patients/medical-history-actions.ts]

### Upsert sur clé composite
`setConsentStatus` utilise `prisma.consentRecord.upsert` avec la clé composite générée par Prisma `where: { patientId_type: { patientId, type } }` (nom dérivé de `@@unique([patientId, type])`). Logique d'horodatage :
```
granted === true  → { granted: true,  grantedAt: now, revokedAt: null, policyVersion: CONSENT_POLICY_VERSION, note }
granted === false → { granted: false, revokedAt: now, note }   // grantedAt conservé
```
[Source: prisma/schema.prisma (contraintes @@unique existantes), patron 9.3]

### Note RGPD (réserve de production — mise à jour)
- Le consentement et les données de santé associées relèvent de l'**art. 9 RGPD**. Cette story livre la **brique consentement** (traçabilité de l'état par finalité + version de politique + horodatage), mais **ne lève pas seule** la réserve de mise en production du dossier patient : le **chiffrement applicatif (11.4)**, le **journal d'audit des accès (11.3)** et le **droit à l'oubli / export (11.2)** restent non livrés. La mise en production de l'ensemble du dossier patient reste donc conditionnée à un arbitrage PO/RGPD couvrant l'épopée 11.
