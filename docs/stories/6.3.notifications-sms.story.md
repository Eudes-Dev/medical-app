# Story 6.3: Notifications SMS (optionnel)

## Status

Ready for Review

## Story 

**As a** patient sans email,
**I want** recevoir un SMS de confirmation et de rappel,
**so that** je n'oublie pas mon rendez-vous même sans adresse mail.

## Acceptance Criteria

1. Si le patient n'a pas d'email mais un téléphone mobile FR valide, un SMS de confirmation est envoyé à la création d'un RDV via le tunnel invité (`createGuestBooking`), quel que soit le statut initial (PENDING).
2. Un SMS de rappel J-1 (cron 17h UTC) et H-2 (fenêtre glissante) est envoyé en complément du rappel email pour les patients ayant un téléphone mobile valide et n'ayant pas activé l'opt-out (`patient.reminderOptOut = false`). La logique est ajoutée à la route cron existante `app/api/cron/reminders/route.ts` (story 6.2).
3. Le coût de chaque SMS envoyé est journalisé en base (`MessageLog.cost`, `MessageLog.costUnit`) — mis à jour par le webhook Twilio Status (`POST /api/webhooks/twilio/status`). Permet un suivi budgétaire via une simple requête `SUM(cost)`.
4. Les numéros sont validés en format E.164 via `libphonenumber-js` avec la France (`FR`) comme région par défaut. Seuls les numéros mobiles (préfixes `+336`, `+337`) sont éligibles aux SMS. Les fixes (`+331`–`+335`) sont ignorés silencieusement.
5. En cas d'échec d'envoi SMS (réponse Twilio en erreur ou exception réseau), le système tente automatiquement un fallback email si `patient.email` est défini ; sinon, l'échec est loggé dans `MessageLog` (status `FAILED`) sans bloquer la création/mise à jour du RDV.
6. Un toggle global d'activation du module SMS est exposé via la variable d'environnement `SMS_ENABLED` (`true|false`). Quand `SMS_ENABLED=false`, toutes les fonctions `sendSms*` retournent immédiatement sans appel Twilio (no-op). Le toggle DB par cabinet est repoussé à la story 7.1.

## Dev Notes

### Contexte et prérequis

- **Prérequis obligatoires** :
  - Story **6.1** (Done) : `sendEmail`, `EmailLog` (à renommer en `MessageLog`), fire-and-forget pattern.
  - Story **6.2** (Ready for Review) : route cron `/api/cron/reminders`, modèles `optOutToken` / `reminderOptOut`, helpers `sendReminderD1` / `sendReminderH2`.
- **Pattern** : strictement identique à 6.1/6.2 — fire-and-forget, `sendSms` ne throw jamais, journalisation systématique dans `MessageLog`.
- **Priorité backlog** : P2 (consolidation 3-6 mois après MVP). Cette story peut être livrée OFF par défaut (`SMS_ENABLED=false`) sans impact sur les autres modules.

### Décisions d'architecture

| Décision | Choix retenu | Raison |
|---|---|---|
| Provider SMS | **Twilio** | API mature, SDK Node officiel, France-friendly (Sender ID alphanumérique ou numéro long FR), sandbox de test, tarification transparente. Alternatives évaluées : OVH SMS (FR-only, API moins ergonomique), Brevo (limites de débit MVP-incompatibles). |
| Validation numéros | `libphonenumber-js` | Bibliothèque légère (vs `google-libphonenumber`), API ergonomique, tree-shakeable, supporte E.164 + détection type (mobile/fixe). |
| Unification logs EMAIL + SMS | **Renommer `EmailLog` → `MessageLog`** avec colonne `channel: MessageChannel` | Architecture propre (un canal de communication = un seul journal). Permet des requêtes unifiées de suivi/coût. Le refactor des références code est limité (~5 fichiers : `lib/email/client.ts` + tests). |
| Toggle SMS | Env var `SMS_ENABLED=true\|false` | MVP single-tenant — pas besoin de toggle DB. La story 7.1 (config cabinet) introduira un toggle par cabinet en complément. |
| Trigger SMS confirmation | **Uniquement si `patient.email` est absent** | Lecture stricte de l'AC 1 ("Si le patient n'a pas d'email mais un téléphone mobile valide"). Évite le double notif (email + SMS) à la création — réduit le coût SMS et la surface "spam". Le rappel J-1/H-2 reste en double notification (AC 2 dit "en complément"). |
| Fallback SMS → email | Helper exécuté seulement si l'envoi SMS échoue et que `fallbackEmail` est fourni | Garantit qu'un patient avec email + téléphone reçoit toujours au moins une confirmation, même si Twilio est down. |
| Stockage du coût | Décimal `Decimal(10, 5)` + devise ISO, alimenté par le webhook Twilio Status | L'API `messages.create` retourne souvent `price: null` au moment de l'envoi — le prix réel est connu après la livraison via webhook. Coût indispensable pour AC 3 (suivi budgétaire). |
| Numéros éligibles | Mobiles FR uniquement (`+336`, `+337`) | MVP single-tenant FR. Les fixes ne reçoivent pas de SMS. Évolution future : étendre via config cabinet. |
| Signature webhook Twilio | Header X-Twilio-Signature OU secret en query string | Twilio supporte la signature HMAC native ; pour simplicité MVP, accepter aussi un `?secret=<TWILIO_STATUS_WEBHOOK_SECRET>` partagé. |

### Modifications Prisma — schema complet

```prisma
enum MessageChannel {
  EMAIL
  SMS
}

enum MessageType {  // renommé depuis EmailType
  CONFIRMATION
  CANCELLATION
  PRACTITIONER_NOTIFY
  REMINDER_D1
  REMINDER_H2
}

enum MessageStatus {  // renommé depuis EmailStatus
  SENT
  FAILED
}

model MessageLog {  // renommé depuis EmailLog
  id                String          @id @default(uuid())
  appointmentId     String?         @map("appointment_id")
  appointment       Appointment?    @relation(fields: [appointmentId], references: [id], onDelete: SetNull)
  channel           MessageChannel  @default(EMAIL)
  to                String          // email ou téléphone E.164
  type              MessageType
  status            MessageStatus
  providerMessageId String?         @map("provider_message_id")
  cost              Decimal?        @db.Decimal(10, 5)
  costUnit          String?         @map("cost_unit")
  error             String?
  createdAt         DateTime        @default(now()) @map("created_at")

  @@map("message_logs")
}
```

### Validation E.164 — exemple

```typescript
import { parsePhoneNumberFromString } from "libphonenumber-js";

export function toE164(raw: string, region: "FR" = "FR"): string | null {
  const parsed = parsePhoneNumberFromString(raw, region);
  return parsed?.isValid() ? parsed.number : null;  // ex: "+33612345678"
}

export function isMobileFR(e164: string): boolean {
  return /^\+33[67]\d{8}$/.test(e164);
}

export function getPatientSmsTarget(patient: { phone: string }): string | null {
  const e164 = toE164(patient.phone);
  return e164 && isMobileFR(e164) ? e164 : null;
}
```

### Pattern client SMS avec fallback

```typescript
// lib/sms/client.ts
export async function sendSms({
  to, body, appointmentId, type,
  fallbackEmail,
  fallbackEmailFactory,
}: SendSmsOptions): Promise<void> {
  if (process.env.SMS_ENABLED !== "true") return;  // no-op global

  try {
    const message = await twilio.messages.create({
      to,
      from: process.env.TWILIO_PHONE_NUMBER!,
      body,
      statusCallback: `${process.env.NEXT_PUBLIC_APP_URL}/api/webhooks/twilio/status?secret=${process.env.TWILIO_STATUS_WEBHOOK_SECRET}`,
    });
    await prisma.messageLog.create({
      data: { appointmentId, channel: "SMS", to, type, status: "SENT", providerMessageId: message.sid },
    });
  } catch (err) {
    const errMsg = err instanceof Error ? err.message : String(err);
    await prisma.messageLog.create({
      data: { appointmentId, channel: "SMS", to, type, status: "FAILED", error: errMsg },
    }).catch(() => {});
    console.error("[sms] send failed:", errMsg);
    if (fallbackEmail && fallbackEmailFactory) {
      try {
        await fallbackEmailFactory();
      } catch (fallbackErr) {
        console.error("[sms:fallback-email]", fallbackErr);
      }
    }
  }
}
```

### Coûts SMS — flux de données

1. `sendSms` appelle `twilio.messages.create(...)` avec `statusCallback` → retourne `MessageSid` (ex: `SM1234...`).
2. `MessageLog` est créé avec `status: SENT`, `providerMessageId: <sid>`, `cost: null` (encore inconnu).
3. Twilio livre le SMS → POST `/api/webhooks/twilio/status` avec `MessageSid`, `MessageStatus`, `Price` (négatif chez Twilio — prendre `Math.abs`), `PriceUnit`.
4. Le webhook met à jour `MessageLog` (`updateMany` car le MessageSid est unique mais sans index unique pour MVP) avec `cost` + `costUnit`.
5. Suivi budgétaire : `SELECT SUM(cost) FROM message_logs WHERE channel = 'SMS' AND created_at >= NOW() - INTERVAL '30 days'`.

### Notes de sécurité

- `TWILIO_AUTH_TOKEN` et `TWILIO_STATUS_WEBHOOK_SECRET` sont des secrets — ne jamais committer, présents uniquement dans `.env.local` et Vercel env.
- Le webhook `/api/webhooks/twilio/status` doit valider la signature OU le secret en query — sinon attaquable pour injecter de faux statuts/coûts.
- Le `body` du SMS de confirmation contient un lien d'annulation tokenisé (UUID) — pas de PII médicale dans le SMS (le SMS étant non-chiffré chez l'opérateur).
