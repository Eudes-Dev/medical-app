# Story 8.5: Liste d'attente (file priorisée + proposition d'un créneau libéré)

## Status
Done

## Story
**As a** praticien dont l'agenda est souvent complet et qui reçoit des demandes alors qu'aucun créneau n'est libre,
**I want** inscrire un patient sur une **liste d'attente** (avec un motif et un niveau d'urgence, et éventuellement le type de soin souhaité), puis, **quand un créneau se libère** (annulation/suppression d'un RDV), voir immédiatement **quels patients en attente sont compatibles** et **convertir** l'un d'eux en rendez-vous sur ce créneau **en un clic** (la modal de création pré-remplie),
**so that** je ne perde plus les demandes faute de place, je remplisse les créneaux annulés au lieu de les laisser vides, et je priorise les patients les plus urgents — sans ressaisir leurs informations.

- **Création unitaire de RDV** : `createAppointment(data)` dans `app/dashboard/calendar/actions.ts` (story 3.3/7.3) — valide via `appointmentSchema`, résout le type de soin (`resolveServiceSnapshot`), vérifie l'anti-collision (`checkConflict`), insère (`status: PENDING`), `revalidatePath`. C'est la **brique de conversion** : convertir une entrée de liste d'attente = créer un RDV pour ce patient sur le créneau choisi, puis marquer l'entrée comme traitée.
- **Modal de création** : `components/calendar/CreateAppointmentModal.tsx` (3.3/7.3/8.4) — `PatientSelect`, date/créneau, type de soin, durée, notes (+ section récurrence 8.4). Elle accepte déjà des **valeurs initiales d'édition** (`editAppointment`) ; 8.5 réutilise ce point d'entrée pour **pré-remplir patient + type de soin** lors d'une conversion.
- **Annulation / suppression de RDV** (le moment où un créneau **se libère**) : `updateAppointmentStatus(id, "CANCELLED")` et `deleteAppointment(id)` (3.3), plus les actions de série 8.4 (`cancelAppointmentSeries`/`deleteAppointmentSeries`). Ce sont les **déclencheurs** d'un créneau libéré.
- **Type de soin** : `ServiceType` (7.3) — `label`, `durationMin`, `active`, `color`. Une entrée de liste d'attente peut **viser** un type de soin (optionnel) pour le matching et la durée à la conversion.
- **Patient** : `Patient` (`firstName, lastName, phone, email?, …`). Une entrée de liste d'attente **référence un patient existant** (pas de patient « anonyme » : on inscrit un patient déjà au fichier, créé via 2.1/2.2).
- **Notifications** : infra email (`lib/email/*` — `send-confirmation`, `send-cancellation`, `send-reschedule`, …) et SMS (`lib/sms/*`) existent (6.1/6.2/6.3). 8.5 **ne crée aucun nouveau canal** ; l'éventuelle information du patient « un créneau s'est libéré » est traitée en **§Hors périmètre** (le flux d'**acceptation patient self-service par lien** est un gros lift type tunnel public — explicitement reporté).
- **Migrations** : flux établi (dernière = `20260625000009_add_appointment_series_id`, story 8.4). 8.5 ajoute **une** migration additive (nouvelle table + enums + index + relations inverses).

**Le trou que cette story comble** : aujourd'hui il n'existe **aucune persistance** d'une demande non satisfaite. Quand l'agenda est plein, la demande est perdue ; quand un RDV est annulé, le créneau reste vide alors que des patients attendaient. La 8.5 introduit (1) un **modèle `WaitlistEntry`** (patient + urgence + motif + type de soin souhaité optionnel + fenêtre de dates optionnelle + statut), (2) des **Server Actions CRUD** + une action de **matching** (entrées compatibles avec un créneau donné), (3) la **vraie page `/dashboard/waitlist`** (liste triée par priorité, ajout, retrait), (4) un **pont vers la conversion** (pré-remplir la modal de création depuis une entrée) et un **rappel des patients en attente compatibles** au moment où un RDV est annulé/supprimé.

## Acceptance Criteria

### A. Modèle de données — entrée de liste d'attente
1. Un nouveau modèle Prisma **`WaitlistEntry`** est ajouté (`prisma/schema.prisma`, `@@map("waitlist_entries")`) avec : `id String @id @default(uuid())`, `patientId String @map("patient_id")` (FK → `Patient`, `onDelete: Cascade`), `serviceTypeId String? @map("service_type_id")` (FK → `ServiceType`, `onDelete: SetNull`, **optionnel** = « n'importe quel soin »), `priority WaitlistPriority @default(NORMAL)`, `reason String?` (motif, ≤ 200 car.), `notes String?` (≤ 500 car.), `status WaitlistStatus @default(WAITING)`, `preferredFrom DateTime? @db.Date @map("preferred_from")` et `preferredTo DateTime? @db.Date @map("preferred_to")` (fenêtre de dates souhaitée **optionnelle**, calendaire comme `TimeOff`), `resolvedAppointmentId String? @map("resolved_appointment_id")` (RDV créé lors de la conversion, simple traçabilité — pas de relation/contrainte FK, opaque), `createdAt`/`updatedAt`. Index : `@@index([status, priority])` et `@@index([patientId])`.
2. Deux enums Prisma sont ajoutés : **`WaitlistPriority { NORMAL, HIGH, URGENT }`** (sémantique « niveau d'urgence ») et **`WaitlistStatus { WAITING, SCHEDULED, CANCELLED }`** (`WAITING` = en attente ; `SCHEDULED` = converti en RDV ; `CANCELLED` = retiré sans RDV). Les **relations inverses** `waitlistEntries WaitlistEntry[]` sont ajoutées sur `Patient` **et** `ServiceType` (additif, ne casse rien). Une **migration Prisma additive** (`prisma/migrations/<timestamp>_add_waitlist_entries`) crée la table, les enums et les index, **sans** modifier les tables/colonnes existantes ; client Prisma régénéré (`lib/generated/prisma`).
3. Un type applicatif **`WaitlistEntryWithPatient`** (`@/types`) expose l'entrée + `patient` (au moins `id, firstName, lastName, phone, email?`) + `serviceType?` (`id, label, durationMin, color`) pour l'affichage. Un mapper (côté `actions`) projette le résultat Prisma vers ce DTO (pas de fuite de champs non nécessaires).

### B. Validation (schéma séparé)
4. Un module `lib/validations/waitlist.ts` exporte **`waitlistEntrySchema` Zod** : `{ patientId: uuid requis ; serviceTypeId?: uuid ; priority: enum("NORMAL"|"HIGH"|"URGENT") (défaut "NORMAL") ; reason?: string trim ≤ 200 ; notes?: string trim ≤ 500 ; preferredFrom?: date ; preferredTo?: date }`, avec un **`refine`** : si `preferredFrom` **et** `preferredTo` sont fournis, alors `preferredTo >= preferredFrom` (sinon message « La date de fin doit être postérieure ou égale à la date de début »). Exporter `WaitlistEntryFormValues`. **Ne réutilise pas** `appointmentSchema` (sémantique différente : pas d'heure ni de durée ici).

### C. Server Actions — CRUD liste d'attente
5. Nouveau module `app/dashboard/waitlist/actions.ts` (`"use server"`), constante `WAITLIST_PATH = "/dashboard/waitlist"`. Toutes les actions appellent `requireUser()` en tête et `revalidatePath(WAITLIST_PATH)` après mutation (mêmes conventions que `calendar/actions.ts`). Les actions sur un id valident l'UUID via `assertValidUuid` ; erreurs `UnauthorizedError` relancées, `BadRequestError` → message « identifiant invalide », autre → message générique + `console.error` (gabarit identique aux actions calendrier).
6. `addToWaitlist(data: WaitlistEntryFormValues): Promise<WaitlistActionResult>` : valide via `waitlistEntrySchema`, vérifie que le `patientId` existe (`prisma.patient.findUnique`, sinon `success:false`), si `serviceTypeId` fourni vérifie qu'il existe **et n'est pas archivé** (`active`, réutiliser la même garde SEC-002 que `resolveServiceSnapshot`), crée l'entrée (`status: WAITING`). Retourne `{ success:true, entry }` ou `{ success:false, error }`.
7. `getWaitlist(): Promise<WaitlistEntryWithPatient[]>` : retourne les entrées **`status = WAITING`** (les `SCHEDULED`/`CANCELLED` n'apparaissent pas dans la file active), incluant `patient` + `serviceType`, **triées** par priorité **décroissante** (`URGENT` > `HIGH` > `NORMAL`) puis par `createdAt` **croissant** (FIFO à priorité égale). Le tri par énum n'étant pas ordonnable directement par Prisma, trier en mémoire avec un poids explicite `{ URGENT:3, HIGH:2, NORMAL:1 }`.
8. `removeFromWaitlist(id: string): Promise<{ success; error? }>` : passe l'entrée à `status: CANCELLED` (retrait sans RDV — on **conserve** la ligne pour un futur historique plutôt qu'un hard delete). `updateWaitlistEntry(id, data)` (optionnel mais recommandé) permet d'éditer priorité/motif/fenêtre d'une entrée `WAITING`.

### D. Matching « créneau libéré » + conversion en RDV
9. `getMatchingWaitlistEntries(slot: { startTime: Date; durationMin?: number }): Promise<WaitlistEntryWithPatient[]>` : retourne les entrées `WAITING` **compatibles** avec un créneau, **triées comme `getWaitlist`** :
   - **type de soin** : une entrée sans `serviceTypeId` matche tout ; une entrée avec `serviceTypeId` ne matche que si elle vise ce même soin (le créneau libéré porte le `serviceTypeId`/`type` du RDV annulé — passé par l'appelant) ;
   - **fenêtre de dates** : une entrée sans fenêtre matche toujours ; sinon le **jour calendaire** du `slot.startTime` (heure **locale**, cf. §Fuseau horaire) doit être dans `[preferredFrom, preferredTo]` inclus.
   - Le matching est **best-effort / non bloquant** : il **suggère**, il ne réserve rien.
10. **Conversion** : depuis une entrée de liste d'attente (page waitlist) **ou** depuis la suggestion affichée après une annulation, le praticien peut **« Programmer un rendez-vous »**. L'action ouvre `CreateAppointmentModal` **pré-remplie** avec le `patientId` et (si présent) le `serviceTypeId`/durée de l'entrée ; la création passe par le **`createAppointment` existant inchangé** (anti-collision incluse). **Après création réussie**, l'entrée de liste d'attente est passée à **`status: SCHEDULED`** avec `resolvedAppointmentId` renseigné, via une action `markWaitlistScheduled(entryId, appointmentId)` (`requireUser`, `assertValidUuid` sur les deux, `revalidatePath`). Aucune logique d'anti-collision n'est dupliquée : c'est `createAppointment` qui tranche (si le créneau est repris entre-temps → toast `slotTaken` standard, l'entrée **reste `WAITING`**).
11. **Rappel au moment de la libération d'un créneau** : quand un RDV est annulé/supprimé depuis le détail (`AppointmentDetailsModal` — `updateAppointmentStatus("CANCELLED")` / `deleteAppointment`), le composant appelle `getMatchingWaitlistEntries({ startTime, durationMin })` du créneau libéré et, **s'il existe ≥ 1 entrée compatible**, affiche une **notification non bloquante** (toast ou encart) « N patient(s) en attente pour ce créneau » avec un accès rapide à la liste d'attente (lien `/dashboard/waitlist`). C'est une **suggestion** : aucune action automatique, aucun email envoyé. *(Si l'intégration dans le détail s'avère trop intrusive, l'alternative acceptée est un simple lien/CTA — l'essentiel est que la libération d'un créneau soit reliée à la liste d'attente, AC testable côté action `getMatchingWaitlistEntries`.)*

### E. Page `/dashboard/waitlist` (remplace le placeholder)
12. `app/dashboard/waitlist/page.tsx` n'utilise **plus** `PlaceholderPage` : c'est une vraie page (Server Component) qui appelle `getWaitlist()` et rend la liste dans le **même shell** que les autres pages dashboard (`SidebarProvider` + `AppSidebar` + header/breadcrumb « Dashboard / Liste d'attente », cf. le markup du placeholder à reproduire). Elle affiche, pour chaque entrée : nom du patient, **badge de priorité** (couleur distincte par niveau — `URGENT` rouge, `HIGH` orange/ambre, `NORMAL` neutre, cohérent avec les conventions de badges existantes), type de soin souhaité (ou « Tout soin »), fenêtre de dates si présente, motif, ancienneté (`createdAt`), et des actions : **« Programmer un rendez-vous »** (conversion, AC 10) et **« Retirer »** (AC 8, avec `confirm()` natif comme l'existant). Un **état vide** explicite (« Aucun patient en liste d'attente ») est affiché si la file est vide.
13. Un bouton **« Ajouter à la liste d'attente »** ouvre une modale (`react-hook-form` + `zodResolver(waitlistEntrySchema)`) : `PatientSelect` (réutiliser le composant existant de `CreateAppointmentModal`), `select` de priorité (Normal/Haute/Urgente), `select` optionnel de type de soin (catalogue `ServiceType` actif, comme la modal de création), champ motif, fenêtre de dates optionnelle (deux date pickers ou inputs `date`), notes. Soumission → `addToWaitlist`, toast de succès, fermeture + refresh (`router.refresh()` ou re-fetch). Erreurs via toasts (`errors.validation`/`errors.server`).

### F. Messages utilisateur (FR centralisés)

Ajouter dans `lib/ui/toast-messages.ts` un bloc **`waitlist`** : `added` (« Patient ajouté à la liste d'attente. »), `removed` (« Patient retiré de la liste d'attente. »), `scheduled` (« Rendez-vous programmé depuis la liste d'attente. »). ### Previous Story Insights (8.1 → 8.4)
- **8.4** a posé le pattern « **modèle léger + migration additive** » : un seul ajout de schéma, additif, rétro-compatible, client régénéré. 8.5 fait de même mais avec une **nouvelle table** (plus lourd que la colonne `seriesId` de 8.4) → bien suivre le flux de migration (OPS-001) et **ne pas** toucher les tables existantes. 8.4 a aussi systématisé : `requireUser` une seule fois en tête d'action, `assertValidUuid` sur les ids, `revalidatePath` unique en fin, schéma Zod **séparé** par préoccupation, helpers purs testables sans DB, messages FR centralisés. **Reproduire tout ce gabarit.**
- **8.4 (DRY service)** : `resolveServiceSnapshot(serviceTypeId, fallbackType, fallbackDuration)` (`calendar/actions.ts`) centralise la résolution + refus du service archivé (SEC-002). `addToWaitlist` n'a pas besoin de snapshot (pas de RDV créé), mais **réutilise la même garde** « service introuvable/archivé » ; à la **conversion**, c'est `createAppointment` qui refait la résolution complète — **ne pas la dupliquer**.
- **8.1/8.2/8.3/8.4** : toutes hors-MVP, draftées sur acceptation explicite du risque, **validées par le PO (`validate-next-story`) avant dev**. 8.5 ajoutant une **nouvelle table** (plus de surface que 8.4), la validation PO est **d'autant plus recommandée** avant implémentation.
- **Convention TZ locale du dashboard** (8.3/8.4) : tout le dashboard raisonne en **heure locale du navigateur** ; **ne pas** importer les helpers Paris du tunnel public (REL-001). Le matching par fenêtre de dates compare des **jours calendaires locaux** (cf. §Fuseau horaire).

### Modèle `WaitlistEntry` — nouvelle table additive
[Source: prisma/schema.prisma — models Patient, ServiceType, Appointment ; enums existants]
- Style des modèles existants : `id String @id @default(uuid())`, colonnes snake_case via `@map`, `createdAt @default(now()) @map("created_at")`, `updatedAt @updatedAt @map("updated_at")`, enums en MAJUSCULES, `@@map("table_name")`, `@@index([...])`. **Suivre ce style.**
- Relations : `patient Patient @relation(fields:[patientId], references:[id], onDelete: Cascade)` (un patient supprimé purge ses entrées, cohérent avec `Appointment`), `serviceType ServiceType? @relation(fields:[serviceTypeId], references:[id], onDelete: SetNull)` (détache sans casser, cohérent avec `Appointment.serviceType`). **Ajouter les inverses** `waitlistEntries WaitlistEntry[]` sur `Patient` et `ServiceType`.
- `preferredFrom`/`preferredTo` en `@db.Date` (calendaire, **pas** de composante horaire) — **même choix que `TimeOff.startDate/endDate`** (story 7.2) pour éviter toute ambiguïté de fuseau au stockage.
- `resolvedAppointmentId String?` : **simple traçabilité opaque** (pas de relation FK ni de contrainte) — volontairement léger, à l'image de `cancellationToken`/`seriesId` (champs opaques nullables ajoutés sans sur-ingénierie). Justification anti-sur-ingénierie : 8.5 n'a pas besoin de jointure forte vers le RDV résolu ; un id suffit à un futur « historique des conversions ».
- ⚠️ Anti-sur-ingénierie : **pas** de moteur de proposition automatique par email/lien d'acceptation dans ce socle (cf. §Hors périmètre). Le statut `SCHEDULED`/`CANCELLED` + `resolvedAppointmentId` suffit à un futur historique sans refonte.

### Validation — schéma dédié, indépendant d'`appointmentSchema`

### Server Actions — gabarit calendrier à recopier
[Source: app/dashboard/calendar/actions.ts — requireUser, assertValidUuid, revalidatePath, resolveServiceSnapshot, toAppointmentWithPatient, CreateAppointmentResult, try/catch]
