# Story 4.1: Page de Réservation Publique

## Status
Done

## Story
**As a** patient,
**I want** accéder à une page publique pour voir les créneaux disponibles du cabinet,
**so that** je puisse choisir un rendez-vous qui me convient sans avoir besoin de créer un compte.

## Acceptance Criteria
1. La route dynamique `/[cabinet-slug]` affiche la page d'accueil du cabinet (single-tenant: slug accepté mais non vérifié pour le MVP — voir Dev Notes "Cabinet Slug")
2. La page d'accueil affiche les informations de base du cabinet (nom, adresse, horaires d'ouverture, téléphone) à partir d'une source unique partagée avec la génération de créneaux
3. Un bouton "Prendre Rendez-vous" mène vers `/[cabinet-slug]/book` (CTA accessible: hauteur min 44px, label ARIA explicite)
4. La page `/[cabinet-slug]/book` affiche un calendrier (sélecteur de date) et les créneaux disponibles pour la date choisie
5. L'interface de sélection de créneau est optimisée pour mobile: zones tactiles min 44×44px, grille 2 colonnes (mobile) / 4 colonnes (desktop), feedback visuel `:active` et `:focus-visible`
6. Seuls les créneaux non-occupés sont affichables et sélectionnables. Un créneau est considéré occupé s'il chevauche un `Appointment` existant dont le `status != CANCELLED`
7. La sélection d'un créneau l'enregistre dans le store Zustand `useBookingStore` et navigue vers `/[cabinet-slug]/book/guest` (étape 2 — formulaire invité, implémentée en story 4.2). Le bouton "Continuer" est désactivé tant qu'aucun créneau n'est sélectionné
8. La Server Action `getAvailableSlots` valide ses entrées via Zod et est protégée par un rate limiter (cf. story 13.5 si déjà implémentée — sinon TODO documenté)
9. La page expose les métadonnées SEO de base (`title`, `description`, `og:title`) et respecte les exigences a11y WCAG AA (navigation clavier, contrastes, landmarks `<main>`, `<nav>`)

### Cabinet Slug (MVP single-tenant)
Pour le MVP, le segment `[cabinet-slug]` est conservé dans l'URL pour préparer le multi-tenant (story 18.1) **mais n'est pas validé**. Toute valeur est acceptée et résout vers l'unique cabinet configuré dans `CABINET_INFO`. ### Algorithme de génération des créneaux
```typescript
import { startOfDay, endOfDay, addMinutes, startOfToday } from 'date-fns'
import { CABINET_INFO } from '@/lib/cabinet/config'

const generateSlots = (date: Date): Date[] => {
  const { start, end, slotMinutes } = CABINET_INFO.openingHours
  const slots: Date[] = []
  for (let hour = start; hour < end; hour++) {
    for (let min = 0; min < 60; min += slotMinutes) {
      const slotTime = new Date(date)
      slotTime.setHours(hour, min, 0, 0)
      slots.push(slotTime)
    }
  }
  return slots
}

const isOverlapping = (slot: Date, slotMinutes: number, apt: { startTime: Date; endTime: Date }) => {
  const slotEnd = addMinutes(slot, slotMinutes)
  return slot < apt.endTime && slotEnd > apt.startTime
}

export const getAvailableSlots = async (date: Date) => {
  const allSlots = generateSlots(date)
  const { slotMinutes } = CABINET_INFO.openingHours
  const appointments = await prisma.appointment.findMany({
    where: {
      startTime: { gte: startOfDay(date), lt: endOfDay(date) },
      status: { not: 'CANCELLED' },
    },
    select: { startTime: true, endTime: true },
  })
  return allSlots.filter(slot => !appointments.some(apt => isOverlapping(slot, slotMinutes, apt)))
}
```

### Sécurité - Server Action publique
La Server Action est **non authentifiée** (réservation par patient invité). Risques: scraping, DoS, énumération. Mitigations requises:
- Validation Zod stricte (date dans le futur, max 90 jours)
- Rate limiting par IP (cf. story 13.5 — si non livrée, marquer comme TODO et ouvrir un ticket de blocage)
- Pas d'exposition d'information personnelle (la requête Prisma utilise `select` pour ne renvoyer QUE `startTime`/`endTime`)
