# Story 5.2: Sécurisation & Tests E2E

## Status
Ready for Review

## Story
**As a** praticien,
**I want** que toutes les actions sensibles soient sécurisées et que les flux critiques (réservation invité, authentification) soient couverts par des tests de bout en bout,
**so that** les données de mes patients restent protégées, qu'aucune régression silencieuse ne passe en production, et que la sortie MVP repose sur des garanties vérifiables.

## Contexte métier
Sprint 5 — qualité & déploiement, dernière étape avant la livraison MVP. Les briques sont déjà en place :
- Auth Supabase via `@supabase/ssr` (story 1.3) — middleware `/dashboard/*` opérationnel, vérification serveur via `supabase.auth.getUser()`.
- Chaque Server Action du dashboard appelle déjà `supabase.auth.getUser()` (cf. `app/dashboard/patients/actions.ts`, `app/dashboard/calendar/actions.ts`).
- Validation Zod systématique (`lib/validations/*`) et helpers UX centralisés (`lib/ui/toast.ts`, `lib/ui/error-mapper.ts` — story 5.1).
- Playwright **déjà installé** (`@playwright/test ^1.60.0`) avec `playwright.config.ts`, `tests/e2e/global-setup.ts` (reset DB + login + storageState), helpers (`tests/e2e/helpers/auth.ts`) et 3 suites calendrier livrées en story 3.6.

Cette story consolide trois axes :
1. **Centraliser** la garde d'authentification dans un helper unique (`lib/server/auth.ts`) pour supprimer la duplication actuelle (8+ blocs `createClient() → getUser() → throw/return` répétés).
2. **Renforcer** la validation des inputs sensibles (UUIDs, sanitization).
3. **Combler** les manques E2E : flux invité (story 4.2) et flux d'authentification praticien (story 1.3), incluant la protection des routes.

## Acceptance Criteria

### Sécurité — Garde d'auth centralisée
1. Un helper `lib/server/auth.ts` expose :
   - `requireUser(): Promise<{ id: string; email?: string }>` — récupère l'utilisateur via `supabase.auth.getUser()` ; **throw** `UnauthorizedError` (depuis `lib/errors.ts`) si non authentifié.
   - `getCurrentUser(): Promise<User | null>` — variante non-bloquante (pour les pages publiques qui adaptent leur affichage).
2. Toutes les Server Actions du dashboard (`app/dashboard/patients/actions.ts`, `app/dashboard/calendar/actions.ts`, `app/dashboard/actions.ts`) utilisent `requireUser()` au lieu de réimplémenter la séquence `createClient() → getUser() → throw/return`. Les fonctions concernées au minimum : `getPatients`, `createPatient`, `updatePatient`, `getPatientById`, `deletePatient`, `getAppointmentsByDateRange`, `createAppointment`, `updateAppointment`, `deleteAppointment`, `updateAppointmentStatus`, `checkConflict`, plus toute action exportée dans `app/dashboard/actions.ts`.
3. Les actions publiques (`app/(public)/[cabinet-slug]/book/actions.ts`) restent **explicitement non authentifiées** (réservation invité). Un commentaire `// PUBLIC: pas d'auth requise` est ajouté en tête de chaque export pour expliciter la décision.
4. Le middleware (`middleware.ts`) reste l'unique point de redirection des routes `/dashboard/*` — aucun changement de matcher ni de logique de redirection (la couverture existante est déjà validée par 1.3).

### Sécurité — Validation et sanitization
5. Un helper `lib/validations/uuid.ts` expose `uuidSchema` (Zod) et `assertValidUuid(id: string): void` (throw une erreur typée `BadRequestError` en cas d'invalide). Toutes les Server Actions qui acceptent un `id` (`getPatientById`, `updatePatient`, `deletePatient`, `updateAppointment`, `deleteAppointment`, `updateAppointmentStatus`) valident l'UUID **avant** la requête Prisma.
6. Une nouvelle classe d'erreur `BadRequestError` est ajoutée à `lib/errors.ts` (mêmes conventions que `UnauthorizedError`). Mapping côté UI dans `lib/ui/error-mapper.ts` : code `BAD_REQUEST` → message FR neutre « La demande est invalide. ».
7. Les schémas Zod existants (`patientSchema`, `appointmentSchema`, `guestBookingSchema`) appliquent déjà `.trim()` et bornes de longueur. Cette story **vérifie** la couverture et complète les schémas qui n'ont pas de `.trim()` sur les champs texte (nom, prénom, email, téléphone, notes, motif). Aucune migration de DB.
8. Les requêtes Prisma utilisent toujours des paramètres typés (pas de `$queryRaw` sur entrées utilisateur — Prisma protège des injections SQL par défaut). Aucune introduction de SQL raw n'est tolérée dans cette story.

### Tests E2E — Réutilisation de l'infra existante
9. Les nouveaux tests E2E **réutilisent** l'infra Playwright en place (`playwright.config.ts`, `global-setup.ts`, `tests/e2e/helpers/auth.ts`, `prisma/seed-e2e.ts`). Aucune installation, ni nouvelle config Playwright, ni nouveau `globalSetup`.
10. Une suite `tests/e2e/public-booking/4.2-flux-invite.spec.ts` couvre le flux réservation invité (story 4.2) :
    - Accès à `/(public)/[cabinet-slug]` puis navigation vers `/(public)/[cabinet-slug]/book`.
    - Sélection d'une date dans l'horizon autorisé (≤ 90 jours).
    - Sélection d'un créneau disponible.
    - Remplissage du `GuestForm` (firstName, lastName, phone, email valides).
    - Soumission et apparition de l'écran de succès (`/[cabinet-slug]/book/success`).
    - Cas d'erreur : validation Zod en client (téléphone trop court) → toast `error` avec message FR.
    - Cas d'erreur : conflit de créneau (`SLOT_TAKEN`) simulé en injectant un RDV via le seed avant le test → toast erreur générique « Ce créneau vient d'être réservé. Merci d'en choisir un autre. ».
    - Note d'implémentation : ce projet utilise le projet Playwright `desktop-chromium` *sans* `storageState` partagé pour ces tests (override `use: { storageState: undefined }` dans la suite), car le tunnel invité est strictement non authentifié.
11. Une suite `tests/e2e/auth/1.3-auth-praticien.spec.ts` couvre l'authentification praticien (story 1.3) :
    - Connexion via `/login` avec identifiants valides (issus de `E2E_PRACTITIONER_EMAIL` / `E2E_PRACTITIONER_PASSWORD`, alimentés par le seed).
    - Redirection vers `/dashboard` après succès.
    - Rejet avec identifiants invalides → message d'erreur visible dans le formulaire.
    - Déconnexion (bouton dans le layout dashboard) → retour à `/login`.
    - Cette suite **désactive** le storageState : `test.use({ storageState: { cookies: [], origins: [] } })` pour repartir d'un état non connecté.

    - `GET /dashboard/patients` sans cookie → redirection vers `/login`.

    - Une fois connecté (via le helper `loginAsPractitioner` de `tests/e2e/helpers/auth.ts`), les trois accès aboutissent à la page attendue (status 200, sélecteur clé visible).

### Pattern d'authentification serveur réel (source : code existant)
Le projet utilise `@supabase/ssr` (et non `@supabase/auth-helpers-nextjs` qui est déprécié, cf. story 1.3). Le pattern actuel répété dans les Server Actions du dashboard :

```typescript
// app/dashboard/patients/actions.ts (extrait existant)
import { createClient } from "@/lib/supabase/server";
import { UnauthorizedError } from "@/lib/errors";

const supabase = await createClient();
const { data: { user } } = await supabase.auth.getUser();
if (!user) {
  throw new UnauthorizedError("User must be authenticated to access patient data");
}
```

Le helper cible (Task 1) à créer :

```typescript
// lib/server/auth.ts (à créer)
import { createClient } from "@/lib/supabase/server";
import { UnauthorizedError } from "@/lib/errors";

/**
 * Récupère l'utilisateur authentifié ou throw UnauthorizedError.
 * À utiliser en tête de chaque Server Action du dashboard.
 *
 * Sécurité : utilise getUser() (validation serveur Supabase Auth)
 * et non getSession() (lecture cookie locale uniquement).
 */
export async function requireUser(): Promise<{ id: string; email?: string }> {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) {
    throw new UnauthorizedError("User must be authenticated");
  }
  return { id: user.id, email: user.email };
}

/** Variante non bloquante pour les pages qui adaptent leur affichage. */
export async function getCurrentUser() {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();
  return user;
}
```
