# Story 8.4: Rendez-vous récurrents (création d'une série + gestion de la série à venir)

## Status
Done

## Story
**As a** praticien qui reçoit un patient à intervalle régulier (suivi hebdomadaire, séance toutes les 2 semaines, contrôle mensuel),
**I want** créer **en une fois** une **série de rendez-vous récurrents** (même patient, même créneau horaire, même type de soin) selon une fréquence et un nombre d'occurrences, puis pouvoir **annuler/supprimer toute la série à venir** depuis le détail d'un RDV,
**so that** je n'ai plus à ressaisir manuellement chaque rendez-vous d'un suivi régulier, tout en gardant une vue claire de quels créneaux appartiennent à une même série.

État du domaine **déjà livré** (à NE PAS refaire) :

C'est la **brique unitaire** qu'une série appelle N fois.
- **Anti-collision** : `checkConflict(startTime, endTime, excludeId?)` détecte le chevauchement d'un créneau avec un RDV non-`CANCELLED`. **Réutilisée telle quelle** par occurrence.
- **Formulaire de création** : `components/calendar/CreateAppointmentModal.tsx` (story 3.3/7.3) — sections Patient / Date & créneau / Type de soin / Durée / Notes, `react-hook-form` + `zodResolver(appointmentSchema)`. La **récurrence est une section supplémentaire** de ce formulaire, **uniquement en mode création** (`!isEdit`).
- **Détail d'un RDV** : `components/calendar/AppointmentDetailsModal.tsx` (story 3.3) — actions Modifier / Confirmer / Annuler / Terminer / Supprimer (avec `confirm()` natif). La **gestion de série** s'ajoute ici (annuler / supprimer **la série à venir**).
- **Modèle `Appointment`** : `prisma/schema.prisma` — `id, patientId, startTime, endTime, status, type, serviceTypeId?, notes?, cancellationToken?, …`. **Aucun champ de série n'existe encore** → c'est le seul changement de schéma de cette story.
- **Migrations** : flux établi (dernière = `20260604000008_add_message_type_rescheduled`, story 8.1). 8.4 ajoute **une** migration additive (colonne nullable + index).

**Le trou que cette story comble** : aujourd'hui chaque RDV est **isolé** ; pour un suivi régulier, le praticien doit rouvrir la modal et ressaisir N fois. Il n'existe **aucun lien** entre des RDV appartenant à un même suivi. La 8.4 introduit (1) un **identifiant de série** (`seriesId`) groupant des occurrences, (2) une **action serveur d'expansion** qui génère les occurrences avec anti-collision par créneau, (3) une **section récurrence** dans la modal de création avec aperçu des dates, (4) un **marqueur de série** + **actions de série** (annuler/supprimer à venir) dans le détail.

## Acceptance Criteria

### A. Modèle de données — identifiant de série
1. Le modèle `Appointment` gagne un champ **`seriesId String?`** (UUID opaque, `@map("series_id")`, **nullable** — `null` pour tout RDV non récurrent, y compris l'historique) + **`@@index([series_id])`**. Une **migration Prisma additive** est créée (`prisma/migrations/<timestamp>_add_appointment_series_id`), nommée et appliquée selon le flux existant (cf. 8.1). Aucune colonne existante n'est modifiée ni supprimée ; **rétro-compatibilité totale** (les RDV existants restent `seriesId = null`). Le type applicatif `AppointmentWithPatient` (et le mapper `toAppointmentWithPatient`) exposent `seriesId: string | null`.

### B. Saisie de la récurrence (formulaire de création)
2. En **mode création uniquement** (`!isEdit`), la modal `CreateAppointmentModal` propose une section **« Récurrence »** : un interrupteur/checkbox **« Rendez-vous récurrent »** (désactivé par défaut → comportement 3.3 unitaire inchangé). Quand il est **activé**, deux contrôles apparaissent :
   - **Fréquence** : `Hebdomadaire` (toutes les semaines), `Toutes les 2 semaines`, `Mensuelle` (même quantième de mois).
   - **Nombre d'occurrences** (RDV total de la série, **1ʳᵉ incluse**) : entier borné, **min 2, max 26** (valeur par défaut raisonnable, ex. 4).
3. Quand la récurrence est active, le formulaire affiche un **aperçu lisible des dates générées** (ex. liste « mar. 9 juin 09:00 · mar. 16 juin 09:00 · … »), recalculé à la volée depuis la date/heure de début, la fréquence et le nombre d'occurrences. L'aperçu reflète **exactement** les créneaux qui seront tentés (mêmes heure de début et durée que la 1ʳᵉ occurrence).
4. La validation de la récurrence est **séparée** de `appointmentSchema` (qui reste inchangé) : un **`recurrenceSchema` Zod** valide `{ frequency: "weekly" | "biweekly" | "monthly"; occurrences: number ∈ [2,26] }`. Le formulaire ne soumet la récurrence **que** si l'interrupteur est actif ; sinon il appelle `createAppointment` comme aujourd'hui (zéro régression du flux unitaire).

### C. Génération de la série (action serveur + anti-collision)
5. Une **nouvelle Server Action** `createRecurringAppointments(data, recurrence)` (dans `app/dashboard/calendar/actions.ts`) : `requireUser()`, valide `data` via `appointmentSchema` et `recurrence` via `recurrenceSchema`, **résout le type de soin** (snapshot `type` + durée — **même logique que `createAppointment`**, factorisée pour éviter la duplication), génère **un `seriesId` UUID unique** puis, pour **chaque** date d'occurrence calculée :
   - calcule `startTime`/`endTime` (durée du soin),
   - appelle `checkConflict(startTime, endTime)`,
   - **si libre** : crée le RDV (`status: PENDING`, `seriesId` renseigné) ; **si occupé** : **ignore** cette occurrence (ne casse pas la série) et la compte comme « ignorée ».
6. L'action retourne un **résumé structuré** : `{ success: true, seriesId, created: number, skipped: number, skippedDates: Date[] }` (ou `{ success: false, error }`). Règles :
   - Si **aucune** occurrence n'a pu être créée (toutes en conflit), retourner `success: false` avec un message explicite (aucun `seriesId` orphelin laissé en base — rien n'a été inséré).
   - La 1ʳᵉ occurrence doit respecter la règle « futur » d'`appointmentSchema` (réutilisée). Les occurrences générées héritent de la même heure locale (cf. §Fuseau horaire).
   - `revalidatePath("/dashboard/calendar")` est appelé **une seule fois** en fin d'action (pas par occurrence).
7. **Génération des dates** (helper pur, testable sans DB) `buildRecurrenceDates(start: Date, frequency, occurrences): Date[]` :
   - `weekly` → `addWeeks(start, i)` ; `biweekly` → `addWeeks(start, 2*i)` ; `monthly` → `addMonths(start, i)` pour `i ∈ [0, occurrences-1]` (date-fns, **heure locale préservée**, gestion DST/fins de mois par date-fns).
   - Renvoie exactement `occurrences` dates, la 1ʳᵉ = `start`. **Aucune** date n'est filtrée ici (le filtrage anti-collision se fait dans l'action, pas dans le helper pur).

### D. Restitution & gestion de la série (détail d'un RDV)
8. Dans `AppointmentDetailsModal`, quand le RDV affiché a un `seriesId` non nul, un **marqueur visuel « Série »** (badge/libellé) est affiché. Les **actions de série** suivantes sont proposées **en plus** des actions unitaires existantes (qui restent inchangées pour « cet épisode ») :
   - **« Annuler toute la série à venir »** : passe au statut `CANCELLED` tous les RDV de la même série dont `startTime >= ` celui du RDV courant et **non déjà** `CANCELLED`/`COMPLETED` (les passés/terminés ne sont pas touchés).
   - **« Supprimer toute la série à venir »** : supprime (hard delete) les RDV de la même série dont `startTime >= ` celui du RDV courant.
   - Les deux actions sont protégées par une **confirmation** (`confirm()` natif, comme l'existant) mentionnant le **nombre** d'occurrences impactées.
9. Deux **Server Actions de série** servent le point 8 : `cancelAppointmentSeries(seriesId, fromStartTime)` et `deleteAppointmentSeries(seriesId, fromStartTime)` — `requireUser()`, `assertValidUuid(seriesId)`, opèrent **uniquement** sur `seriesId = X AND startTime >= fromStartTime` (jamais sur d'autres séries), retournent `{ success, affected }`, `revalidatePath`. **Modifier l'horaire d'une seule occurrence** via `updateAppointment` (3.3) reste possible et **ne touche pas** les autres (elle conserve son `seriesId` — c'est une occurrence simplement déplacée). *(Hors périmètre : « modifier toute la série », cf. §Hors périmètre.)*

### E. Messages utilisateur (FR centralisés)
10. Les libellés réutilisés passent par `lib/ui/toast-messages.ts` (convention 5.1) — ajouter sous `appointment` : `seriesCreated` (base, ex. « Série de rendez-vous créée. »), `seriesCancelled` (« Série annulée. »), `seriesDeleted` (« Série supprimée. »). Le **détail chiffré** (« 4 créés, 2 ignorés ») peut être composé par le composant à partir du résumé de l'action (comptes dynamiques), de la même façon que les `aria-label` chiffrés tolérés dans le code existant — **sans** créer une clé toast par valeur. En cas d'occurrences ignorées (`skipped > 0`), informer l'utilisateur de manière **non bloquante** (la série partielle a bien été créée).

### F. Qualité & tests
11. Couverture de tests :
    - **Unit** (`tests/unit/calendar/recurrence-utils.test.ts`) : `buildRecurrenceDates` — longueur = `occurrences`, 1ʳᵉ date = `start`, espacements corrects pour `weekly`/`biweekly`/`monthly` (dont un cas « piège » : début 31 janvier → `monthly` géré par date-fns ; conservation de l'heure locale). `recurrenceSchema` — accepte des bornes valides, rejette `occurrences < 2`, `> 26`, fréquence inconnue.
    - **Integration** (`tests/integration/calendar/recurring-actions.test.ts`, prisma mocké comme `appointment-actions.test.ts`) : `createRecurringAppointments` crée N RDV avec le **même** `seriesId` quand tout est libre ; **ignore** les occurrences en conflit et renvoie `{ created, skipped, skippedDates }` cohérents ; renvoie `success: false` si **toutes** en conflit (et n'insère rien) ; `cancelAppointmentSeries`/`deleteAppointmentSeries` ne ciblent que `seriesId` + `startTime >= from` (vérifier le `where` Prisma) ; `requireUser` appliqué (rejette non authentifié).

### Modèle `Appointment` — ajouter UN champ nullable, rien d'autre
[Source: prisma/schema.prisma — model Appointment]
- Champs actuels : `id, patientId, startTime, endTime, status (PENDING par défaut), type (snapshot 7.3), serviceTypeId?, notes?, cancellationToken?, reminderD1SentAt?, reminderH2SentAt?, createdAt, updatedAt`. Relations : `patient` (Cascade), `serviceType` (SetNull), `messageLogs`. Index : `patientId`, `startTime`, `serviceTypeId`.

Pas de modèle `RecurrenceSeries` séparé, pas de stockage de la « règle » (fréquence) : grouper + filtrer par `startTime` suffit à toutes les opérations de cette story (annuler/supprimer à venir). ### Création unitaire — la brique à appeler N fois, et à factoriser
[Source: app/dashboard/calendar/actions.ts — createAppointment, checkConflict, toAppointmentWithPatient]
- `createAppointment(data: AppointmentFormValues)` : `requireUser()` → `appointmentSchema.safeParse` → **résolution service** (si `serviceTypeId` : `prisma.serviceType.findUnique`, refuse `!active` (SEC-002), prend `label`→`snapshotType` et `durationMin`→durée) → `endTime = addMinutes(startTime, durationMin)` → `checkConflict` → `prisma.appointment.create({ status: "PENDING", … })` → `revalidatePath(CALENDAR_PATH)`. **Le bloc de résolution service est à extraire** en helper privé (ex. `resolveServiceSnapshot(serviceTypeId, fallbackType, fallbackDuration)`) consommé par `createAppointment` **et** `createRecurringAppointments`.
- `checkConflict(startTime, endTime, excludeId?)` : `findFirst` des RDV non-`CANCELLED` qui chevauchent (`startTime < end && endTime > start`). **Réutiliser tel quel** par occurrence (sans `excludeId` à la création). Pas d'optimisation prématurée : N ≤ 26 appels séquentiels sont acceptables ; un éventuel batch (un seul `findMany` sur toute la plage) est une amélioration optionnelle à noter, **pas** un prérequis.
- `toAppointmentWithPatient(a)` mappe vers le DTO : **y ajouter** `seriesId: a.seriesId ?? null` (et sélectionner `seriesId` dans les requêtes). `CreateAppointmentResult` existe déjà comme union `{success}` ; créer `CreateRecurringResult` sur le même modèle avec le résumé chiffré.
- `CALENDAR_PATH = "/dashboard/calendar"` (constante du module) — réutiliser pour les `revalidatePath`.

### Validation — schéma séparé, ne pas toucher `appointmentSchema`
[Source: lib/validations/appointment.ts]
- `appointmentSchema` = `{ patientId, startTime (refine: futur), duration ∈ {15,30,45,60,90}, serviceTypeId?, type?, notes? }`. **Inchangé** : la série réutilise ce schéma pour `data` (la 1ʳᵉ occurrence). La règle « futur » (`d.getTime() > Date.now()`) s'applique donc à la date de départ.
- Ajouter **à côté** `recurrenceSchema = z.object({ frequency: z.enum(["weekly","biweekly","monthly"]), occurrences: z.number().int().min(2).max(26) })` + `export type RecurrenceFormValues`. Garder `APPOINTMENT_DURATIONS` tel quel.
